# 32 面向对象编程之多态（C 实现）

这一课，Miro Samek 继续讲解 Object-Oriented Programming 的第三个核心概念：polymorphism，不过这一次重点不再是 C++，而是如何在 portable、standard-compliant 的 C 语言中手工实现多态。上一课已经借助 C++ 的 virtual function 解释了 late binding 的原理，这一课则进一步把这些机制“拆开”，一步一步在 C 里重新搭起来。

这节课的意义很明确：
1. 让你真正理解 virtual function 在底层到底是怎么工作的。
2. 说明多态在 C 中不是“做不到”，而是需要显式实现 VPTR 和 VTABLE。
3. 给出一个很重要的工程判断标准：什么时候值得用 polymorphism，什么时候不该用。

Samek 也明确提醒了一点：和 encapsulation、single inheritance 不同，polymorphism 在 C 里不是“几乎白送”的能力，而是会引入额外代码复杂度和运行时开销。所以如果系统里大量依赖多态，直接使用 C++ 通常是更合理的选择。C 版本更多适用于框架或库内部封装实现细节，例如 QP/C 这类实时框架。

## 本课的目标：在 C 里手工复现 C++ 的虚函数机制

上一课已经 reverse-engineer 过 C++ 中 polymorphism 的典型实现：

1. 对象中增加一个隐藏的 VPTR。
2. 每个类有一张放在 ROM 中的 VTABLE。
3. virtual call 在运行时通过 VPTR 找到 VTABLE，再间接调用对应函数。

这一课不再依赖编译器自动生成这些机制，而是要求你在 C 中自己把它们写出来，并且设计方式要和 C++ 的 VPTR-VTABLE 方案保持一致。

## 第一步：在 Shape 中显式加入 VPTR

要在 C 里支持多态，首先要修改基类 `Shape` 的数据结构。做法是在 `Shape` 结构体中显式加入一个 VPTR，而且这个指针必须放在第一个成员位置。

这样设计的原因有两个：

1. 这和上一课分析的 C++ 对象布局一致。
2. 把 VPTR 放在对象起始位置，可以让 late binding 的访问路径最直接。

这个 VPTR 的类型是指向 `const ShapeVtable` 的指针。这里的 `const` 非常关键，因为它意味着 VTABLE 本身只读，可以放进 ROM，而不是占用宝贵的 RAM。

课程顺带提醒了一个 C 语言细节：即使此时还没有完整定义 `ShapeVtable` 结构体，也可以先在 `Shape` 中声明一个“指向它的指针”。因为编译器此时只需要知道“这是个指针”，并不需要知道整个 VTABLE 结构的大小。

## 第二步：用结构体表示 VTABLE

虽然名字叫 VTABLE，但它通常不是数组，而是一个结构体。结构体中的每个成员，都是一个 virtual function 的函数指针。

对 `Shape` 来说，这一课用到的 virtual operation 有两个：

1. `draw()`
2. `area()`

因此 `ShapeVtable` 就是“由两个函数指针组成的结构体”。这一步本质上是在 C 中把“类的方法表”具体化。

## 第三步：理解函数指针是整个方案的基础

课程在这里专门停下来讲了函数指针，因为如果这个概念不牢，后面的 VTABLE 和 virtual call 都会看起来很绕。

关键点有三层：

1. 函数也有地址，因此可以像变量一样被指针引用。
2. 函数指针不仅记录地址，还包含完整的函数签名信息。
3. 声明函数指针时，必须用括号把 `*` 和名字括起来，否则语法会被解释错。

也就是说，VTABLE 之所以能工作，本质上就是把一组“具有统一签名的函数入口地址”收集到一个结构体里，再在运行时按需取出并调用。

## 第四步：为客户端提供 virtual call 接口

有了 VPTR 和 VTABLE，客户端还需要一种统一方式来触发 late binding。课程给出三种实现方式。

### 方式一：普通成员函数式的 vcall 封装

最直接的做法，是在 `Shape` 层提供两个函数：

1. `Shape_draw_vcall()`
2. `Shape_area_vcall()`

这些函数内部做的事情完全一致：

1. 通过对象 `me` 找到内部的 VPTR。
2. 通过 VPTR 找到 VTABLE。
3. 从 VTABLE 中取出对应函数指针。
4. 把 `me` 作为第一个参数传进去，完成调用。

这正是 late binding 的核心路径。

其中最值得注意的一点是：`me` 指针会被使用两次。

1. 第一次是为了从对象里取出 VPTR，使调用和“对象实例”绑定。
2. 第二次是作为成员函数的第一个参数，传给真正的实现函数。

这说明 virtual call 的目标之所以取决于对象真实类型，关键就在于“先从对象中找到自己的 VTABLE”。

### 方式二：用 C99 的 inline 函数消除额外调用开销

上一种写法虽然清晰，但多了一层 `vcall()` 函数本身的调用开销。Samek 认为更好的方式是把这些函数写成 `static inline`，直接放到头文件中。

这样做的好处是：

1. 接口仍然有类型检查，安全性比宏高。
2. 编译器有机会直接展开调用，避免额外函数开销。
3. 代码可读性仍然不错。

不过课程也提醒，这要求编译器至少支持 C99，因为 `inline` 是 C99 才引入的关键字。在 Keil 工程里，需要显式开启 C99 mode，否则代码无法编译。

### 方式三：用宏实现低开销 virtual call

如果编译器还停留在 C89，那么没有 `inline`，这时唯一的低开销替代方案就是宏。

但 Samek 明确认为宏不是理想方案，原因包括：

1. 宏只是文本替换，没有真正的类型系统保护。
2. 为了避免参数表达式产生副作用，往往要写很多括号。
3. 可读性和可调试性都比 inline 差。

所以这只是“旧编译器环境下的无奈替代”，不是推荐设计。

## 第五步：在构造函数中建立 VTABLE 和 VPTR

前面只是搭好了调用机制，但真正让对象具备多态能力，还必须完成两件事：

1. 定义类自己的 VTABLE。
2. 在对象构造时，把 VPTR 指向正确的 VTABLE。

这与上一课分析的 C++ 行为完全一致。区别只在于：C++ 编译器会自动生成这些代码，而这里需要手工实现。

### Shape 的 VTABLE

在 `shape.c` 中，需要定义一个 `static const ShapeVtable` 实例。这张表应放在模块内，并在定义时立即完成初始化，因为 `const` 对象创建后不能再改。

为了初始化 VTABLE，需要先提供两个函数原型：

1. `Shape_draw()`
2. `Shape_area()`

它们通常声明为 `static`，因为只在 `shape.c` 内部使用。

随后把这两个函数的地址填进 VTABLE，再在 `Shape_ctor()` 里把对象的 VPTR 设置为这张表。

### Shape 作为抽象基类时的默认实现

课程特别指出，在 `Shape` 这一层，`draw()` 和 `area()` 其实很难有真正有意义的实现，因为 `Shape` 太抽象了。它只是一个“所有图形共有接口”的承载者。

因此这里通常只给出最小化实现，甚至只是空函数，并通过把未使用参数强转为 `void` 的方式消除编译器警告。

这也说明一个重要设计直觉：基类的 virtual function 不是一定要有具体业务逻辑，它更重要的职责是定义统一接口。

## 第六步：用 drawGraph() 验证多态接口已经成立

一旦 `Shape` 的 virtual call 机制建立起来，就可以马上复用上一课 C++ 的经典例子：写一个通用的 `drawGraph()`。

这个函数的思路和 C++ 版本几乎一样：

1. 接受一个 `Shape *` 指针数组。
2. 遍历数组。
3. 对每个元素调用 `Shape_draw_vcall()`。

这样，高层代码完全不需要知道数组里装的是 `Shape`、`Rectangle` 还是其他派生类对象。真正的实现分派会在运行时完成。

这正是 polymorphism 的工程价值所在：高层算法依赖抽象接口，而不依赖具体类型。

## 第七步：派生类 Rectangle 如何接入这套多态机制

有了基类 `Shape` 的 VPTR/VTABLE 机制之后，派生类 `Rectangle` 还需要做两件事：

1. 定义自己的 VTABLE。
2. 提供自己的 `draw()` 与 `area()` 实现，并让 VPTR 最终指向这张新表。

### Rectangle 可以直接复用 ShapeVtable 的结构

这一课假设 `Rectangle` 没有新增 virtual function，只是重写已有的 `draw()` 和 `area()`。在这种情况下，不需要重新定义一种新的 VTABLE 类型，直接复用 `ShapeVtable` 就够了。

只有当派生类要增加新的 virtual operation 时，才需要进一步处理“VTABLE 的继承”。本课刻意不扩展到那一步，以便聚焦核心机制。

### 为什么要对函数指针做强制类型转换

这里有一个很典型的 C 与 C++ 差异。

`Rectangle_draw()` 和 `Rectangle_area()` 的第一个参数类型是 `Rectangle *`，但 `ShapeVtable` 中对应槽位期待的是以 `Shape *` 为第一个参数的函数指针签名。

在 C++ 中，编译器理解继承关系，能自动完成这种语义上的匹配；但在 C 中，编译器并不知道 `Rectangle` 和 `Shape` 的继承关系，因此这里必须显式对函数指针做强制类型转换，才能把 `Rectangle` 的实现函数放进 `Shape` 的 VTABLE 里。

这再次说明：C 版本的 polymorphism 之所以麻烦，不是因为原理复杂，而是因为语言本身不理解类层次关系，很多本该自动完成的动作都需要你手工表达。

### 派生类构造函数中设置 VPTR 的顺序非常重要

在 `Rectangle_ctor()` 中，必须先调用 `Shape_ctor()`，然后再把 VPTR 改写为 `Rectangle` 自己的 VTABLE。

顺序不能反，因为：

1. `Shape_ctor()` 会先把 VPTR 设成 `Shape` 的 VTABLE。
2. `Rectangle_ctor()` 的职责，是在此基础上覆盖为 `Rectangle` 的 VTABLE。

这和上一课分析的 C++ 构造顺序完全一致，也就是说，C 版手工实现和 C++ 编译器自动生成的行为在概念上是一一对应的。

## 第八步：在主程序中验证 polymorphism 的运行效果

课程最后把上一课 C++ 中的测试代码移植到 C 版本主程序中：

1. 创建若干 `Shape *` 指针，实际分别指向 `Shape` 或 `Rectangle` 对象。
2. 构造这些对象。
3. 把它们放入 `graph[]` 数组。
4. 调用通用的 `drawGraph()`。

由于 C 不会自动 upcast，所以这里在数组初始化以及某些构造调用中还需要显式类型转换，才能让编译器接受。

但从运行结果看，整个行为与上一课 C++ 版本一致：同一个 `Shape_draw_vcall()`，会根据对象中保存的 VPTR 跳到不同的实现函数。

## 在调试器里观察：C 版 late binding 与 C++ 几乎完全一致

这节课最有价值的部分之一，是 Samek 再次带着你在调试器里看真实执行过程。

### 构造阶段

调试 `Rectangle_ctor()` 时可以清楚看到：

1. 先进入 `Shape_ctor()`，对象的 VPTR 被设为 `Shape` 的 VTABLE。
2. 返回到 `Rectangle_ctor()` 后，VPTR 被覆盖成 `Rectangle` 的 VTABLE。
3. VTABLE 地址位于 ROM 区域。

这说明手工写出的 C 版本构造流程，与 C++ 自动生成的 VPTR 初始化流程本质相同。

### 调用阶段

在 `drawGraph()` 中单步执行 `Shape_draw_vcall()` 时，反汇编会展示出和上一课 C++ 版本几乎一模一样的机器指令序列：

1. 先从对象中取出 VPTR。
2. 再从 VTABLE 中取出具体函数地址。
3. 把 `me` 指针准备为第一个参数。
4. 最后通过寄存器间接跳转。

第一轮循环调用到的是 `Shape_draw()`，第二轮则跳到 `Rectangle_draw()`。也就是说，late binding 的行为完全按预期工作，而且在机器码层面与 C++ 实现没有本质差别。

这个结论很重要：多态不是某种“只能靠高级语言黑魔法才能做到”的特性，它本质上就是一套可解释、可验证的数据结构与间接调用机制。

## 另一种常见实现：把整张 VTABLE 直接嵌进对象

课程还顺带提到，VPTR 指向外部 VTABLE 并不是实现多态的唯一方法。另一种常见做法，是把整张 VTABLE 直接嵌入对象本身。

这样做的优点是：

1. virtual call 逻辑更直接。
2. 语法上甚至可以更接近 C++ 的点运算风格。

但代价也很明显：

1. VTABLE 不再只有一份，而是每个对象都有一份。
2. 这会显著增加 RAM 占用。
3. 对于对象数量较多的系统，RAM 可能会翻倍甚至更多。

在嵌入式场景下，这通常是一个非常现实的成本，因此 Samek 更推荐上一课和本课采用的 VPTR 指向 ROM 中共享 VTABLE 的方案。

## 什么时候应该考虑 polymorphism

课程最后给出了一条非常实用的工程判断标准。

如果你在代码里不断看到这类模式：

1. 对象里保存一个 `kind` 或 `type` 字段。
2. 每次做行为分派时，都写一大段 `switch(kind)`。
3. 或者写很多 `if/else` 根据类型决定调用哪个函数。

那么这通常就是一个 code smell，说明你可能应该考虑引入 polymorphism。

原因是这种写法会导致：

1. 每次新增一种类型，都得回头修改很多已有代码。
2. 类型相关逻辑分散在整个系统中，难以维护。
3. 高层代码直接依赖具体类型，扩展性很差。

与之相比，多态把“具体行为的选择”收敛到对象自己的 VTABLE 中。高层代码只保留统一接口调用，因此新增或删除派生类时，不必到处修改分支逻辑。

## 什么时候不该用 polymorphism

Samek 对这一点讲得非常直接：polymorphism 唯一成立的前提，是“行为选择必须发生在运行时”。

如果某个差异其实在编译期、链接期就已经确定，那么用 polymorphism 往往是在滥用运行时机制。

课程举的典型反例，是产品线管理。

很多公司会为多个相近型号的设备维护同一套代码，然后在运行时用大量条件分支判断：

1. 产品型号
2. 版本号
3. 市场区域
4. 某些功能开关

这种做法会把代码逐步变成难以理解、难以测试的条件泥潭。即便用 polymorphism 去替换这些 IF-THEN-ELSE，也未必是最优方案，因为这些差异往往并不是“运行时才知道”，而是在构建某个具体产品时就已经确定。

更好的方式通常是：

1. 设计清晰的抽象接口，例如 BSP 接口。
2. 为不同产品提供不同实现模块。
3. 通过目录结构、文件组织和 link-time 组合来生成最终产品。

也就是说，这类问题更属于 physical design，而不是 runtime polymorphism。

## OOP 之外，嵌入式更需要重视 physical design

课程在最后顺势把话题拉回嵌入式工程实践。Samek 认为，相比逻辑设计层面的 OOP，很多嵌入式团队更缺的是 physical design 能力，也就是：

1. 如何划分目录。
2. 如何拆分模块。
3. 如何组织头文件与实现文件。
4. 如何通过链接组合不同产品变体。

这类设计往往比“是不是用了多态”更直接决定系统的可维护性。换句话说，polymorphism 是一个有力工具，但它只能解决“运行时按对象类型分派行为”的那一类问题，不能替代模块边界和构建结构上的良好设计。

## 总结

32 课真正完成了一个很关键的闭环：把上一课在 C++ 里观察到的 polymorphism 原理，完整地移植到了标准 C 中。

这一课的核心结论可以概括为：

1. C 语言完全可以实现 polymorphism，但需要手工维护 VPTR、VTABLE 和 virtual call。
2. VTABLE 本质上是“由函数指针构成的结构体”，而 VPTR 让每个对象在运行时找到自己所属类的方法表。
3. `static inline` 是在 C 中实现低开销 virtual call 的首选方式，宏只是对旧编译器的兼容方案。
4. 派生类必须提供自己的 VTABLE，并在构造函数里于调用基类构造之后覆盖 VPTR。
5. 从调试器和反汇编可以验证，C 版 late binding 与 C++ virtual function 在底层机器码上几乎一致。
6. polymorphism 适合解决“必须在运行时根据对象真实类型选择行为”的问题。
7. 如果差异其实可以在编译期或链接期确定，那么更应优先考虑 abstraction、BSP 和 physical design，而不是 runtime polymorphism。

如果说第 29 课解释了封装，第 30 课解释了继承，第 31 课解释了 C++ 多态的机制，那么这一课则把多态彻底还原成了一种可以在嵌入式系统里亲手实现、亲自验证的工程技术。这样一来，你在以后无论使用 C 还是 C++，都不会只停留在“会用”虚函数，而是能真正理解它的成本、价值和适用边界。
