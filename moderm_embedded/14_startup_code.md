# 14 从编译到链接：为什么你能替换掉默认 vector table

lesson 13 已经把你带到了 `main()` 之前，也第一次把 `.intvec` 和默认 startup code 暴露出来。到了 lesson 14，Miro Samek 不再满足于“知道向量表在那里”，而是进一步追问：既然现在看到的 vector table 来自 IAR 库，那它到底是怎么被链接进最终镜像的？如果理解了这个过程，我们是不是就能把它换成自己的版本？

所以这一课真正的主题并不只是 vector table 本身，而是更底层的 build process。Samek 很明确地告诉你：想接管 startup code，就必须先理解编译器、对象文件、链接器、库和链接脚本各自干什么。否则你只是机械地加一个 `startup_tm4c.c`，却不知道为什么它有时会生效、有时会被库里的默认实现悄悄盖过去。

## 从 lesson 13 的疑问继续追：默认 vector table 为什么不完整

lesson 13 最后你已经在地址 0 附近看到了向量表，但那个表其实来自 IAR 标准库里的通用实现，而不是专门针对 TM4C123 的完整设备级表。这带来两个问题：

1. 它只是“足够让程序跑起来”的默认版本。
2. 它并没有把芯片手册里的完整异常/中断入口都按你的工程方式暴露出来。

也正因为这个缺口，lesson 14 的目标才成立：你要开始自己提供 startup 文件，逐步取代工具链默认给你的那份通用 vector table。

## 先退一步：嵌入式开发为什么必须理解 cross-development

Samek 在这一课一开始先讲 host 和 target。看似基础，其实很有必要。因为 build process 的很多细节，如果默认你在“同一台机器上编译并运行”，就容易理解错。

嵌入式开发的典型场景是：

1. 编译器、链接器、IDE 运行在桌面主机上。
2. 生成的程序却是给完全不同架构的目标板运行。
3. 这叫 cross-development。

这个前提非常关键，因为它解释了为什么：

1. 你必须使用面向 ARM 的工具链，而不能拿 PC 上的 x86 linker 来拼接 ARM 对象文件。
2. “文件格式一样”不代表“机器码语义一样”。
3. 链接器必须理解目标处理器的指令编码，才能做重定位和修补。

也就是说，lesson 14 虽然讲的是 vector table，但本质上是在建立一种“镜像是如何在宿主机上被制造出来”的底层视角。

## 编译器先产出什么：object file 不是最终可执行镜像

课程接着把源文件 `main.c`、`delay.c` 放进编译器流程中，得到 `main.o`、`delay.o`。这是很多教材都会讲的一步，但 Samek 不满足于一句“object file contains relocatable code”。他偏要继续追问：所谓 relocatable，到底是什么意思？

于是他直接去看 `.o` 文件。

如果用文本编辑器硬开，会觉得几乎是乱码；但其中能看到 `ELF` 字样，说明对象文件采用的是 ELF 格式。然后再用 IAR 的 `ielfdumparm` 工具导出可读内容，就能发现 object file 里已经有：

1. `.text`
2. `.data`
3. `.bss`
4. 调试信息
5. 符号表
6. 供链接器后续修补的各种元数据

这一步非常重要，因为它打破了“object file 就是一小块已完成机器码”的误解。实际上，对象文件里很多内容都还是半成品。

## 为什么不能用 `.o` 文件大小判断代码体积

Samek 专门提醒一点：对象文件大小并不等于模块实际代码大小。因为 `.o` 里除了真正机器码，还带了大量调试信息、符号、section 元数据等。

这和 lesson 13 强调 map file 的理由是一致的。对于嵌入式开发者来说，真正可靠的代码/数据体积信息，仍然应该去看链接完成后的 map file，而不是随便看某个 `.o` 文件在文件系统里占了多少 KB。

## final image 与 object file 的根本差别：链接器已经“下了最终判决”

课程把 `main.o` 和最终输出 `c.out` 并排比较，目的是让你真正看到“链接器到底修了什么”。Samek 挑了一个很典型的例子：`main()` 调用 `delay()` 的 `BL` 指令。

在 `main.o` 里，这条 `BL` 的编码还是一种占位形式，因为编译阶段还不知道 `delay()` 最终会被放在什么地址。等到了链接完成的 `c.out` 里，这条指令的编码已经变成真实可执行偏移量。

这就是 relocation 的具体表现：

1. 编译器先生成“结构正确但地址未定”的代码。
2. 链接器决定最终布局。
3. 再把指令里依赖地址的部分修补成最终值。

Samek 不只是讲了原理，还通过同一条 `BL` 在 object file 和 final image 里编码不同，直观证明了这件事。

## 不只是函数地址，变量地址同样要靠 linker 修补

lesson 14 还特别强调：需要修补的不只是函数调用。像 lesson 13 里加进去的 `p1`、`w`、`t`、`p2` 等全局对象，在编译时同样不知道最终会放到 RAM 哪个具体位置。

因此，编译器在 object file 里往往只能先生成一组待修补的常量池或引用占位。到了链接阶段，等各 section 的地址都定下来后，链接器再把这些变量地址写回去。

也就是说，relocatable code 的意思不只是“代码能搬家”，更准确地说是：

1. 指令中的目标地址还未最终决定。
2. 常量池里的对象地址还未最终决定。
3. 跨模块引用还未最终绑定。
4. 这些都必须等 linker 完成全局布局后统一解决。

## 这也解释了为什么 linker 必须懂目标架构

Samek 借这个机会特别强调：linker 不是一个单纯做字符串拼接的工具。它必须理解目标架构的指令格式，知道某条 `BL` 的立即数怎么编码，知道常量池如何组织，也知道异常向量该放在哪个 section。

因此：

1. x86 的 linker 不能用来链接 ARM 程序。
2. 即使两边都使用 ELF，也不代表二进制语义通用。
3. 工具链必须按目标架构成套提供。

这类观点看似常识，但在真正做 bare-metal、移植 startup 文件或者更换工具链时非常关键。因为很多“明明都是 ELF，为什么不能混着来”的困惑，本质上都死在这一步。

## exported symbols / imported symbols：链接器到底在匹配什么

课程随后把视角从“指令修补”转到“符号解析”。Samek 给出的框架非常清楚：每个 object file 都有两类符号。

1. exported symbols：本对象定义并对外提供。
2. imported symbols：本对象需要但自己没有定义。

例如：

1. `main.o` 导出 `main` 以及多个全局对象。
2. `main.o` 也导入 `delay`，因为它调用了 `delay()` 但自己没定义。
3. `delay.o` 则导出 `delay`，从而把这个依赖补上。

通过这个过程，你才真正能理解“链接器解析跨模块引用”不是一句空话，而是把 undefined symbol 一个个在 exported list 里对上。

## 为什么 `__vector_table` 会把你引到库里去

到了这里，Samek 开始把 build process 和向量表问题接上。因为在你的工程对象文件处理完后，undefined list 里还会残留一个关键符号：`__vector_table`。

这个符号不会由 `main.o` 或 `delay.o` 提供，于是链接器就会开始去标准库里找。按照 map file 的结果，IAR 最终从库 `rt7M_tl.a` 中拉进了 `vector_table_M.o`，它正是默认向量表的来源。

这一步极其关键，因为它回答了 lesson 13 那个悬而未决的问题：默认 vector table 为什么会自动出现？

答案不是“IDE 魔法”，而是：

1. 链接器发现 `__vector_table` 还未定义。
2. 它在库对象里找到了一个导出该符号的模块。
3. 于是把那个模块拉进最终镜像。

## 库和普通对象文件的链接规则为什么不同

Samek 在这里顺带讲了一个非常重要、很多人第一次会忽略的规则：

1. 直接加入工程的对象文件，一定会参与最终链接。
2. 库里的对象文件，只有当它们能满足当前 undefined symbols 时，才会被拉进来。

这条规则解释了为什么标准库可以包含大量功能模块，但最终镜像不会把整个库都搬进去。

更细一点，Samek 还指出：库里对象文件通常粒度很细，常常一个对象只放一个函数或一个核心符号。这样只有真正需要的那一小块会被纳入镜像，避免体积膨胀。

对 startup code 来说，这条规则正好也意味着一件很重要的事：

如果你在自己的工程里直接提供了一个更合适的 `__vector_table`，那就有机会让链接器不再去库里拉那份通用实现。

## lesson 14 真正的转折点：开始自己提供 `startup_tm4c.c`

在理解完 build process 之后，课程终于回到最初目标：你要开始用自己的 startup 文件替换掉默认 vector table。

lesson 14 的 IAR 工程里，Samek 新增了一个最小版本的 `startup_tm4c.c`：

```c
int const __vector_table[] @ ".intvec" = {
    0x20004000,
    0x9
};
```

这个文件还明显标着“INCOMPLETE”，因为 Reset handler 甚至都没真正实现。但它的教学作用不是“马上能运行”，而是先验证两个更基础的问题：

1. 你自己的 `__vector_table` 能不能进最终镜像。
2. 它能不能被放到 `.intvec`，也就是地址 0 对应的位置。

这一步的重点不是功能完整，而是控制权转移。

## 为什么 `.intvec` 这个 section 名字这么关键

在 startup 文件里，Samek 不是随便定义一个数组，而是明确把它放进 `.intvec` section。这和 lesson 13 里 map file 看到的地址 0 区域对应起来了。

这意味着：

1. 你不是只提供了一个叫 `__vector_table` 的符号。
2. 你还要求链接器把它放进 reset 后 CPU 会去读的那个物理位置。

只有“符号正确 + section 正确”同时成立，你的自定义向量表才真正替代了默认版本。这也是为什么 lesson 14 要先讲链接器和链接脚本，而不是一上来只贴一个数组定义。

## 这一课虽然写的是 startup 文件，但真正学的是“替换机制”

很多人会把 lesson 14 简化成“自己写一个 vector table 数组”，其实这会错过 Samek 真正想传达的重点。这一课真正完成的是：

1. 你知道默认 startup 来自哪里。
2. 你知道链接器为什么会把它拉进来。
3. 你知道如何通过自己的对象文件提供同名关键符号。
4. 你知道这个符号还必须被放到正确 section。

也就是说，你学到的不是一个 startup 文件模板，而是“如何接管工具链默认行为”的方法。

这个能力非常重要，因为后面不只是 vector table，很多运行时入口、弱符号覆盖、BSP 默认实现替换，背后都依赖同一套 build/link 机制。

## lesson 14 为什么还不急着把 Reset_Handler 写完

工程里的 `startup_tm4c.c` 明明还是不完整的，Reset handler 甚至只是一个占位值。这不是 Samek 忽略了正确性，而是教学上故意分层：

1. 先确保你自己的表被接管并放对位置。
2. 再去逐步把里面每一项填正确。

如果一开始就把 stack pointer、Reset handler、所有异常和 IRQ 入口一次性全塞进去，初学者很容易只是在复制模板，而没有理解背后的链接逻辑。lesson 14 的节奏正好避免了这个问题。

## 总结

lesson 14 表面上是在继续讲 startup code，实际上它真正解决的是一个更底层的问题：为什么默认 vector table 会自动出现，以及你怎样才能可靠地把它替换成自己的实现。

为了回答这个问题，Samek 先把整个嵌入式 build process 拆开来看：编译器生成的 object file 仍然是 relocatable 的，函数调用和变量地址都还需要链接器修补；ELF 只是容器格式，真正的地址决策与符号解析发生在链接阶段；而标准库里的模块只有在满足 undefined symbol 时才会被拉进最终镜像。

正因为如此，当 `__vector_table` 仍未定义时，链接器才会去 IAR 库里取默认实现；而当你自己在工程里提供了 `startup_tm4c.c` 并把 `__vector_table` 放进 `.intvec` section 后，向量表的控制权才开始回到你手里。

从课程推进上说，lesson 14 完成的是“把工具链默认 startup 替换成自定义 startup 的前半程”。它先让你理解替换为什么可能、为什么生效，以及 section 和 symbol 在其中各扮演什么角色。接下来的 lesson 15，才会继续把这张你已经接管的空表真正填完整：正确的栈顶、正确的 Reset 入口、完整的异常与中断处理函数，以及更像工程代码而不是演示代码的错误处理策略。
