# 39 用状态处理函数实现状态机，并抽象出可复用的事件处理器

这一课，Miro Samek 继续讲 state machine implementation，不过这次他不再满足于“再展示一种写法”，而是明确提出要给出他认为在 C 里更接近最优的实现方式。前面几课已经看过两类主要实现：

1. nested-switch
2. state table

它们各有优点，但也各有明显缺陷。nested-switch 直观但容易膨胀成大函数；state table 结构规整，但会产生稀疏表和大量碎片化的小 handler。于是这一课的目标，就是把前面几种实现中值得保留的优点“混搭”起来，同时尽量去掉它们的缺点。

Samek 这节课的核心思路非常有意思：不要先从 C 代码结构出发，而是反过来先想象一种最理想的状态机描述语言，再倒推如何把它最小代价地落到 C 里。最终得到的结果，是一种基于 state-handler function 的实现方式，并进一步抽象成一个可复用的 `Fsm` 基类，集成进前面构建的 `Micro-C-AO` 框架。

## 先倒着想：如果先设计一种“状态机 DSL”，它应该长什么样

课程一开始提出一个很重要的方法论：在寻找“更优实现”时，不妨先暂时忘掉 C 语法，想象一种专门用来描述状态机的 Domain Specific Language，看看最清晰、最简洁的状态机文本表达形式会是什么样。

Samek 以 `TimeBomb` 为例，尝试把图形状态图改写成文本形式。这个假想 DSL 的核心特征是：

1. 以 state 为中心组织内容。
2. 在每个 state 内部列出 `Entry`、`Exit`、以及各个 transition。
3. transition 用事件名触发，再配合动作和 `TRAN(target)` 指定目标状态。
4. guard condition 直接写在相关 transition 内。
5. initial transition 则被看成来自一个特殊的初始伪状态。

这种写法很接近状态图本身，因此可读性非常高。Samek 还顺带提到，现实中确实存在这类状态机 DSL，例如 SMC（State Machine Compiler）就提供了一种专门描述状态机的语言，再由工具把它翻译成 C/C++ 等代码。

## Samek 的目标不是引入专门编译器，而是把最少的 DSL 语法直接塞进 C

尽管 SMC 这类工具很有代表性，但 Samek 这节课不打算走“专用编译器”路线。他的目标是更务实的：在尽量不偏离 C 的前提下，让状态机代码本身就看起来像状态机 DSL。

换句话说，理想状态不是：

1. 先写一门完全不同的语言，
2. 再靠编译器翻译成 C，

而是：

1. 直接写合法 C 代码，
2. 但这份 C 代码在抽象层次上尽量接近 state diagram。

这也是这一课最终实现方式最有价值的地方：它并没有引入重型工具链，却仍然显著提升了状态机代码的表达力。

## 核心思想：把每个状态实现成一个函数

从假想 DSL 出发，Samek 发现最自然的映射方式是：把每个 state 直接实现成一个 state-handler function。

这很合理，因为状态最重要的职责，本来就是：

1. 收到事件。
2. 根据事件执行动作。
3. 决定是否发生状态转移。

而“根据输入执行动作并返回结果”，本来就是函数最擅长表达的事情。

因此，相比于：

1. 用大 `switch` 去包住所有状态，
2. 或用表去间接索引大量细碎 action handler，

现在改为：

1. 每个状态本身就是一个函数。
2. 状态变量不再是枚举值，而是“当前状态处理函数”的指针。

这一步是整课最核心的设计转折。

## 状态变量从枚举变成“函数指针”

在前面的实现里，state variable 一般是一个枚举值，比如 `WAIT4BUTTON_STATE`、`BLINK_STATE` 等。转移时，本质上就是给这个枚举变量赋新值。

而这节课里，状态变量被重新定义成一个 `StateHandler`，也就是“指向状态处理函数的函数指针”。

这意味着：

1. 当前状态就是“当前该调用哪个状态处理函数”。
2. 转移状态就是“把状态指针改成另一个状态处理函数”。
3. 调度当前状态时，不需要再 `switch(state)`，而是直接调用该函数指针。

这比枚举状态再查分支更直接，也更贴近状态机 DSL 的抽象层次。

## 状态处理函数的签名：保持与事件驱动模型一致

每个 state-handler function 需要访问两类信息：

1. 对象自身数据，也就是 `me` 指针。
2. 当前事件，也就是 `e` 指针。

因此它们的参数形式与前面 state table 中的 action handler 非常相似。Samek 只是把返回类型重新设计成 `State`，这个返回值本质上承载的是处理状态信息，例如：

1. 事件被忽略。
2. 事件被处理但未转移。
3. 发生了状态转移。

这一点延续了上一课通过返回 status 来辅助 dispatch 的做法，但现在这个 status 体系被直接嵌进 state-handler 函数模型本身。

## `TRAN()` 宏：让状态转移看起来像 DSL，而不是普通赋值

如果状态变量现在是函数指针，那么状态转移本质上就是把 `me->state` 改成目标状态处理函数。

Samek 并没有让代码里直接写裸赋值，而是定义了一个 `TRAN(target_)` 宏。它做两件事：

1. 把状态变量赋值为目标 state-handler。
2. 返回一个表示“发生了转移”的状态码。

为了同时完成这两件事，宏使用了 C 的 comma operator。这样在代码里你就可以写出非常接近 DSL 的形式：

1. 执行动作。
2. `status = TRAN(TimeBomb_blink);`

这比直接写“改状态变量 + 手动设置状态码”更整洁，也更像状态图里写 `TRAN(blink)` 的表达方式。

## Entry/Exit 通过特殊事件统一处理

前一课已经引入了 `ENTRY_SIG` 和 `EXIT_SIG` 这两个保留信号，这一课继续沿用它们。

在 state-handler 模型中，这种设计更加自然：同一个状态处理函数内部，本来就可以用 `switch(e->sig)` 区分不同“事件”，因此：

1. 普通业务事件走普通分支。
2. `ENTRY_SIG` 用来执行进入动作。
3. `EXIT_SIG` 用来执行退出动作。

这让一个状态的所有相关逻辑，包括：

1. entry action
2. exit action
3. transitions
4. guard conditions

都能集中在同一个 state-handler function 内部完成。相比 state table 那种分散成多个小函数的写法，这种组织方式显然更符合“一个 state 是一个完整行为单元”的直觉。

## 这一实现方式站在 nested-switch 与 state table 之间的“甜蜜点”上

Samek 在课程中明确指出，这种 state-handler 风格正好落在两种旧方案之间的 sweet spot。

### 相比 nested-switch

它的优点是：

1. 不再把整个状态机塞进一个庞大的 `dispatch()`。
2. 每个状态的逻辑被局部化到单独函数中。
3. 状态机扩展时，不需要在一个大函数里不断加 case。

### 相比 state table

它的优点是：

1. 不需要稀疏的二维表。
2. 不需要为每个状态-事件组合拆成极细粒度 handler。
3. 代码不会被切得那么碎。
4. 可读性更接近状态图本身。

因此 Samek 认为，这种实现兼顾了结构化与可读性，是在 C 里非常有吸引力的折中方案。

## dispatch 也被简化了：当前状态就是一个可直接调用的函数

一旦状态变量变成函数指针，`dispatch()` 的运行逻辑也被进一步简化。

它的大致流程变成：

1. 保存 `prev_state`。
2. 直接调用 `me->state(me, e)`。
3. 得到返回状态码。
4. 如果返回值表示发生了转移，则：
   - 调用旧状态的 `EXIT_SIG`
   - 调用新状态的 `ENTRY_SIG`
5. 如果当前事件是初始化，则只做进入，不做退出。

这里还有一个实现细节：由于 state-handler 内部总会解引用事件指针，因此调用 `ENTRY_SIG` 和 `EXIT_SIG` 时，不能像上一课那样传 `NULL`，而是要传入静态常量事件对象。

这再次体现了这一课实现的一个特点：大量原来零散的机制，现在开始被整合成一套更统一、更通用的运行模型。

## 还可以进一步抽象：dispatch 其实已经完全通用化了

到这里，Samek 发现了更大的机会。此时的 `dispatch()` 几乎已经不包含任何 `TimeBomb` 专属逻辑，它实际上适用于任何采用同样 state-handler 模式的有限状态机。

这意味着：

1. 它不应该继续停留在具体应用类里。
2. 它应该被提升到框架层，成为一个 reusable event processor。

这正是面向对象中 inheritance 应该发挥作用的地方：把应用共享的通用行为抽到基类中，让具体状态机只保留自己的状态定义与动作逻辑。

## 引入 `Fsm` 基类：把通用状态机管理从应用中抽离出来

Samek 因此提出一个新的类层次：

1. `Fsm`：通用 finite state machine 基类。
2. `Active`：继承 `Fsm`，在其上加事件队列和线程。
3. `TimeBomb`、`Blinky` 等应用类：再继承 `Active`。

这个设计有两个好处。

### 1. 把通用状态机机制集中管理

例如：

1. 当前状态指针
2. 初始化
3. dispatch
4. entry/exit 调度

这些内容都可以由 `Fsm` 基类统一提供，而不是每个应用状态机重复实现。

### 2. `Fsm` 不依赖 active object，也可单独复用

Samek 特别强调，把 `Fsm` 和 `Active` 分开是有意义的。因为有些状态机可能只是 passive FSM，并不需要自己的线程和事件队列，例如：

1. 某些中断服务程序中的小状态机。
2. 某些被动调用的内部控制逻辑。

如果把状态机管理直接塞进 `Active`，这些场景就无法复用。独立出 `Fsm` 基类后，框架层级会更清晰。

## 这其实是在做一次 refactoring，而不是改变行为

Samek 顺带点出，这类“只优化设计结构、不改变对外行为”的改动，就是典型的 refactoring。

也就是说，本课后半段的重点并不是给 `TimeBomb` 添加新功能，而是：

1. 保持状态机行为不变。
2. 提升代码抽象层次。
3. 把可复用的机制抽到框架层。

这也是为什么这节课会和前面 OOP in C 的 inheritance 主题重新连接起来。状态机实现本身，终于开始成为一个可复用的软件组件，而不仅仅是一段应用级技巧代码。

## 为什么 Samek 认为这是“更优”的实现方式

虽然本课工具返回里后半段内容不完整，但根据 Samek 已经明确给出的标准，这种 state-handler + reusable event processor 的方案之所以被他认为更优，核心原因大致有以下几点。

### 1. 抽象层次更高

代码中的主要单元不再是低层的 case 分支或表格单元，而是与 state diagram 一一对应的 states。

### 2. 可读性更强

一个状态的 entry、exit、transition、guard 全部集中在一个函数里，人读代码时更容易建立设计对应关系。

### 3. 不需要稀疏表

避免了 state table 版本中“行列很多、空洞很多”的问题。

### 4. dispatch 完全可复用

一旦基于统一的 state-handler 协议，整个事件处理器就能被框架层统一提供。

### 5. 易于扩展

新增状态通常就是新增一个函数和少量声明，而不是去修改一个庞大 `switch` 或扩一整行/列状态表。

## 这一实现方式也更像“直接在 C 里写状态机 DSL”

Samek 在本课中反复强调一个感觉上的变化：他希望你把这些 state-handler 看成一种在 C 里直接实现出来的领域专用语言，而不是普通函数集合。

原因是这时你在代码里已经能直接看到状态机本体元素：

1. state
2. entry
3. exit
4. transition
5. guard
6. initial pseudostate

而不需要先把它们还原成“某个枚举值”“某个索引”“某个动作表项”“某个分支块”。

这正说明此时代码的抽象层次更接近设计模型本身。

## 总结

39 课的核心，是从“具体状态机怎么写”进一步上升到“状态机实现框架应该长什么样”。

这一课最重要的结论可以概括为：

1. nested-switch 和 state table 各有优缺点，但都还不是 Samek 心目中最理想的状态机实现方式。
2. 一个更优的思路，是先从状态机 DSL 的角度想清楚理想表达，再把最少量的 DSL 语法直接映射成合法 C。
3. 在这种映射中，每个 state 最自然的实现方式就是一个 state-handler function。
4. 状态变量从枚举变成函数指针，状态转移则通过 `TRAN()` 宏把目标状态 handler 赋给当前状态变量。
5. `ENTRY_SIG` 和 `EXIT_SIG` 让 entry/exit action 能与普通事件处理统一放进同一个 state-handler 内部。
6. 这种实现方式处在 nested-switch 与 state table 的 sweet spot：既避免巨型 dispatch，也避免稀疏表和过度碎片化。
7. 一旦 dispatch 逻辑变得完全通用，就应该通过 inheritance 把它抽象成可复用的 `Fsm` 基类，并进一步整合进 active object 框架。

如果说第 38 课在讨论 data-driven 的 state table，那么第 39 课则进一步说明：状态机最好的实现，不只是“换一种数据结构”，而是要让代码本身尽量直接表达状态机的语义元素，并把通用的事件处理逻辑沉到框架层。这样，状态机才真正从一种编程技巧，提升成可复用的软件基础设施。
