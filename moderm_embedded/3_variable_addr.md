# 3 变量、作用域与指针：从“值”走向“地址”

## 引言

第 2 课你已经能控制执行路径，但程序仍主要停留在“变量是一个名字”。第 3 课要完成一个关键跃迁：

- 变量不只是名字，它一定对应某个地址。
- 地址可以被取出来、保存到指针，再通过指针间接访问。
- 局部变量和全局/静态变量，通常位于不同存储区。

理解这一课，后面访问寄存器地址（第 4 课）就会非常自然。

## Lesson-03 源码锚点

lesson-03 的核心示例（简化）如下：

```c
static int counter = 0;

int main(void) {
    int volatile *p_int;
    p_int = &counter;
    while (*p_int < 21) {
        ++(*p_int);
    }

    p_int = (int *)0x20000002U;
    *p_int = (int)0xDEADBEEF;

    return 0;
}
```

它一次性展示了三件大事：

1. 取地址：`&counter`
2. 间接访问：`*p_int`
3. 强制把常量地址解释为指针并写入（演示危险行为）

## 局部变量与静态变量的存储位置

### 局部变量（automatic storage）

函数内部定义的普通局部变量通常在栈（stack）上，生命周期随函数进入/退出。

### 文件作用域静态变量（`static`）

`static int counter = 0;` 这种变量通常在 `.data` 或 `.bss`，整个程序生命周期都存在。

这也是为什么你在调试器里会看到：

- 局部变量常在 Locals 视图容易出现。
- 全局/静态变量更适合在 Watch 或 Memory 视图直接观察。

## `&` 与 `*` 的语义

- `&obj`：取对象地址，类型从 `T` 变成 `T*`
- `*ptr`：解引用指针，访问指针所指对象

例如：

```c
int counter = 0;
int *p = &counter;
*p = 10;
```

执行后 `counter` 变为 10，因为 `*p` 和 `counter` 指向同一存储单元。

## 指针并不神秘：它只是“地址值”

在 32 位 MCU 上，指针通常是 32 位数值，里面装的是地址。

所以你可以把指针看成“带类型信息的地址变量”：

- 地址值决定去哪里访问。
- 指针类型决定按多大单位解释和访问。

`int*` 通常按 4 字节单元访问，`char*` 按 1 字节访问。

## 通过指针改写循环变量

lesson-03 的循环：

```c
while (*p_int < 21) {
    ++(*p_int);
}
```

它与直接写 `counter` 的逻辑等价，但教学意义更强：

1. 让你明确“变量访问其实是地址访问”。
2. 让你看到解引用在每次循环里发生读写。
3. 为后续“寄存器地址当指针使用”铺路。

## 强制地址转换示例的风险

这一句是本课很关键的警示：

```c
p_int = (int *)0x20000002U;
*p_int = 0xDEADBEEF;
```

它可能触发多种问题：

1. 地址对齐问题：`int` 常要求 4 字节对齐，`0x20000002` 明显未对齐。
2. 地址合法性问题：该地址是否在有效 RAM 区间、是否可写。
3. 硬件异常：某些配置下会触发 BusFault/HardFault。

课程给这个例子不是鼓励你这么做，而是让你感受“指针给你强大能力，也给你直接踩硬件规则的机会”。

## 地址对齐为什么重要

在 Cortex-M 上，很多访问在对齐情况下更高效也更安全。

假设 `int` 为 4 字节：

- 合法对齐地址：`...0`, `...4`, `...8`, `...C`
- `...2` 这类地址是非对齐访问

某些核心/配置可以处理部分非对齐访问（可能分解成多次访问，性能下降），某些则会直接 fault。工程代码应该主动避免。

## 0xDEADBEEF 的意义

`0xDEADBEEF` 是经典调试填充值（magic pattern）：

- 可读性强，一眼能认出来。
- 用于测试内存写入路径或检查未初始化区。

它不是“特殊语义值”，只是一个便于识别的十六进制常量。

## volatile 指针在这里的作用

源码里指针声明为：

```c
int volatile *p_int;
```

含义是“指向 volatile int 的指针”：

- 通过 `p_int` 访问目标对象时，每次读写都应保留。
- 适用于目标对象可能被外部改变（寄存器、中断共享变量等）场景。

注意它不同于 `int * volatile p_int`（后者是“指针本身不可随意优化”）。

## 调试器观察建议

第 3 课建议你重点做两组观察：

1. 观察 `counter` 地址和值：
   - 在 Watch 输入 `counter` 和 `&counter`。
   - 在 Memory 跳到该地址，查看值变化。

2. 观察 `p_int` 的内容与 `*p_int` 的联动：
   - `p_int` 改变时，`*p_int` 指向对象也随之改变。

这种“值-地址-解引用”三角关系一旦建立，后续学习会顺很多。

## 与第 4 课的直接衔接

第 4 课将出现类似表达式：

```c
*((unsigned int *)0x400FE608U) = 0x20U;
```

如果你已经理解第 3 课，这句就很直白：

1. 常量地址强转成指针。
2. 解引用后写值。
3. 只不过这次地址不在普通 RAM，而在外设寄存器区。

也就是说，第 4 课“点灯”并不是全新知识，而是第 3 课指针模型的直接应用。

## 常见错误清单

### 1. 野指针

声明指针后未初始化就解引用：

```c
int *p;
*p = 1;  // 未定义行为
```

### 2. 类型不匹配访问

把一个地址当成错误类型解引用，可能造成宽度错误、对齐错误或语义错误。

### 3. 把指针当整数随意运算

虽然底层是地址数值，但任意运算很容易越界到非法区域。

## 小结

第 3 课完成了课程前期最关键的一次抽象升级：

1. 变量有地址，地址可被程序显式操作。
2. 指针让代码获得“间接访问能力”。
3. 这种能力是寄存器编程的基础，同时也是内存错误的主要来源。

下一课（第 4 课）你会把这套模型直接用于硬件：通过内存映射寄存器第一次点亮板载 LED，从“纯软件”正式迈入“软硬件交互”。
