# 10 栈为什么会“炸掉”：递归、越界写和异常处理到底暴露了什么

这一课，Miro Samek 表面上是在继续讲函数和栈，但实际上已经明显进入“程序为什么会以非常难看、非常真实的方式失败”的主题。前一课你刚刚通过递归 factorial 看到了栈帧如何一层层压上去，而 lesson 10 则把这个观察继续推到极端：既然你已经知道函数调用会消耗栈，那如果我故意把栈压到头，会发生什么？如果我不是真的用光栈，而只是悄悄把返回地址写坏，又会发生什么？

这就是这一课真正的价值。它不只是告诉你“栈会溢出”，而是用两个非常典型的灾难场景，让你看到 Cortex-M 程序失控时，调试器里究竟会呈现出什么样子。对于嵌入式开发来说，这比抽象地背诵“要小心 stack overflow”有用得多。

## 从 lesson 9 的 factorial 出发：递归本来就是最适合观察栈的实验样本

lesson 9 里，`fact()` 已经让你看到递归调用如何在栈上形成一层层嵌套。lesson 10 没有换题目，而是故意沿用这个函数，因为递归本身就最容易放大栈行为。

课程先在 `fact()` 里加入一个局部数组：

```c
unsigned foo[6];
foo[n] = n;
```

或者更早的实验里先把它放大到 `foo[10]`、`foo[100]`。这个改动看似很小，但它实际上做了两件事：

1. 每次递归调用都要额外消耗一大块自动存储。
2. `foo[n] = n` 还埋下了数组越界写的伏笔。

所以 lesson 10 的两个核心灾难，其实都从同一行局部数组代码长出来。

## IAR 的 Stack View / Call Stack View：第一次把“栈帧”看成真实对象

Samek 这一课很重视调试器视图，不再只用一般的 memory window，而是专门打开：

1. `Stack` 视图
2. `Call Stack` 视图

这样做的意义很大。因为你会第一次明显看到：每调用一次 `fact()`，栈上就多出一份独立的 `foo[]`。递归越深，这些局部数组就一层层叠上去。

这一步把一个很重要的认识从“概念”变成“画面”：

1. 栈不是抽象的“函数调用区”。
2. 它就是一块普通 RAM。
3. 每个局部变量、返回地址、保存寄存器，最终都只是这块 RAM 上的一段内容。

这也是为什么后面一旦栈出问题，本质上就是 RAM 中的关键布局被破坏。

## 自动变量为什么一开始是垃圾值

当 `foo[]` 出现在栈视图里时，Samek 特别强调了一个经常被忽略但非常重要的事实：自动变量不会自动初始化。你看到的 `foo[]` 里那些奇怪值，不是什么“旧的合法状态”，而只是 RAM 以前残留的内容。

他甚至给出一个形象类比：栈就像一摞盘子，但这些盘子全都是脏的。你不能假定拿起来就是干净的，必须自己洗，也就是自己初始化。

这一点在裸机程序里尤其关键，因为：

1. RAM 里可能残留上一次执行、flash loader、调试器或别的模块留下的数据。
2. 如果你依赖未初始化自动变量，错误常常具有偶然性。
3. 这类错误往往最难复现，也最容易被误判成“编译器有问题”。

所以 Samek 借 lesson 10 第一个实验，顺便把“automatic variable 必须显式初始化”这个习惯重新敲牢。

## 真正把栈压爆：从“增长得很快”到“SP 跌出 RAM”

当 `foo` 扩大到 `100` 个元素之后，递归深度稍微上来，栈就被迅速吃光。课程里最关键的观察点不是“程序挂了”，而是挂之前和挂之后调试器里到底能看到什么。

Samek 让你在 `fact()` 的递归调用附近下断点，继续执行，然后观察：

1. `Call Stack` 一层层增长。
2. `SP` 不断向更低地址移动。
3. 到一定深度后，`SP` 恰好掉到 RAM 起始边界附近。
4. 再往下，就进入了不存在的地址空间。

这时程序不会优雅地告诉你“stack overflow”，而是会掉进硬件异常处理。课程里看到的是 `BusFault_Handler` 附近的死循环。

## 为什么是 BusFault，而不是别的异常

Samek 在这里顺带引出一个之后会成为主线的问题：Cortex-M 遇到非法内存访问时，不是“随机崩溃”，而是通过异常机制报告。

当栈指针已经跌出合法 RAM 区域，CPU 继续用它进行压栈、取数据或写返回现场时，就等于访问不存在的内存空间。于是会触发 BusFault。

这里 lesson 10 其实第一次让你真正感受到“异常处理”不是库里摆设，而是 CPU 级硬件保护机制的一部分。虽然这一课还没有正式展开 startup code 和 vector table，但 BusFault 的出现已经提前埋好了线索：

1. 程序挂住时，不要只看 C 代码。
2. 要去看当前异常处理函数。
3. 更要看 `SP` 是否还在合理范围内。

Samek 甚至明确建议你养成习惯：一旦程序卡在异常里，先检查 stack pointer。

## 但真正可怕的不是“完全溢出”，而是“只破坏一点点”

Samek 紧接着说了一句非常关键的话：真正难诊断的常常不是完全跑出 RAM，而是栈只被部分破坏，刚好把某个数据、某个保存寄存器或某个返回地址写坏，但还没有立即触发异常。

这正是第二个实验要展示的内容。

## 第二个灾难场景：不是用光栈，而是用数组越界精确打坏返回地址

这次课程把 `foo` 改成：

```c
unsigned foo[6];
```

然后让 `main()` 调用：

```c
x = fact(7U);
```

由于 `foo` 只有索引 `0..5` 有效，而 `foo[n] = n` 中的 `n` 是 7，所以这次不再是“栈空间不够”，而是“在一个看起来仍然有足够空间的栈帧里越界写”。

Samek 通过单步汇编，把这次“犯罪现场”拆得非常细。你会看到：

1. 函数序言先保存寄存器，比如 `push R4, LR`。
2. 再通过 `sub sp, ...` 在栈上给 `foo[]` 腾空间。
3. 然后计算 `foo[n]` 的地址。
4. 最终 `STR` 把值写到了越界位置。

关键就在于：这个越界位置刚好落在保存的 `LR` 附近，于是返回地址被污染了。

## 这个 bug 为什么比 stack overflow 更危险

因为从外表看，一切都还“像是正常运行”：

1. 没有立刻掉进异常。
2. 栈总体空间也并没有耗尽。
3. 递归还能继续展开、继续回退。

但决定控制流的关键值已经被悄悄写坏了。这类错误正是嵌入式里最讨厌的一类：

1. 现场通常在很早之前就被污染。
2. 真正爆炸要等几百、几千个时钟周期之后。
3. 最终症状和根因往往隔着很远。

Samek 在这里特别强调：调试这类问题，绝不能靠机械单步把所有过程一格一格走完。你必须学会“战略性打断点”。

## 为什么要在函数 return 处布断点

既然你已经知道被破坏的是返回地址，那么最合理的观察点就是函数即将 return 的地方。Samek 正是这么做的。

当执行停在 `fact()` 返回前，你能看到：

1. 栈里还保留着一层层递归现场。
2. 栈大小本身仍然没有爆掉。
3. 但最外层返回路径已经埋好了雷。

接着继续运行，随着递归一层层 unwind，栈一点点回收，直到最后一层执行 `POP ... PC`。这条指令会把本该恢复到 `PC` 的返回地址取出来，但现在取出的已经是你先前越界写进去的 `7`。

也就是说，这次事故的根因不是“return 逻辑错了”，而是“return 读到的地址早就被污染了”。

## 为什么 Samek 故意把破坏值选成奇数 7

这里是整节课最妙的设计之一。Samek 明说，这个值是“精心而恶劣地”挑过的。

因为在 Cortex-M 上，函数/返回地址最低位必须为 1，用来表示 Thumb state。如果你把返回地址破坏成偶数，那么 CPU 很可能会更早、在更直接的位置就因非法状态进入异常，故事会太早结束。

而把它改成 `7` 这样一个奇数，就会发生更离谱的事情：

1. `POP ... PC` 竟然能成功。
2. CPU 会把 `PC` 设到低地址区域。
3. 这些地址原本并不是代码，而是向量表数据。
4. 但处理器仍会尝试把那里解释成 Thumb 指令流。

这就把 bug 从“立即崩溃”变成了“诡异漂移”。调试器里看到的现象会非常像悬疑故事，这也是 Samek 说“worthy of Sherlock Holmes”的原因。

## 向量表为什么会被当成代码执行

课程里最让人印象深刻的一幕，就是被破坏后的返回地址落到了低地址区，而那里恰好存放的是异常/中断向量表。向量表本质上是一串 32-bit 地址常量，但 CPU 并不知道“这里是数据，不该执行”，它只是按照 `PC` 所指位置去取指。

于是会出现一种很迷惑的画面：

1. 调试器反汇编窗口开始把向量表数据解释成 16-bit 指令。
2. 看上去程序还在“执行一些东西”。
3. 甚至由于真正的 `main()` 紧跟在向量表后面，控制流还会误入部分真实代码。

这正是 lesson 10 特别想让你体验的地方：控制流一旦被污染，CPU 不会“懂你的本意”，而是会严格、愚蠢但一致地执行它眼前的位模式。所谓“程序跑飞”，就是这么来的。

## 这一课还顺便回收了 lesson 8 的一个知识点

Samek 特别提醒，如果你已经忘了为什么 Cortex-M 上返回地址必须是奇数，就该回去复习 lesson 8。这个提示很重要，因为它说明课程到这里已经开始真正形成链条：

1. lesson 8 里的 Thumb state 不再只是架构 trivia。
2. lesson 9 里的调用栈不再只是函数教学。
3. lesson 10 把两者和真实 bug 连接起来了。

也就是说，这一课不是单独讲“如何制造 crash”，而是在把前面分散学过的细节汇总成真实的系统行为。

## stack size 和 heap size：不是 IDE 配置项，而是生存边界

在展示 stack overflow 之后，Samek 还顺势讲了一个很实际的工程动作：修改 linker 里的 stack size，并把 heap 设为 0。

这一步有两个含义。

第一，栈大小不是抽象资源，而是真正由链接配置决定的一块 RAM 预算。你必须根据应用真实调用深度和局部变量规模去估算，而不是默认值是多少就永远用多少。

第二，在实时嵌入式里，Samek 非常明确地反对随手使用 heap。因为动态分配常常带来碎片、不可预测延迟和更复杂的故障模式，所以在这门课当前阶段，把 heap 设为 0 是一种很合理的纪律。

虽然这一课并未展开讲链接脚本和 startup code，但这一步已经在为后面 lesson 13-15 铺路：栈和堆最终都是链接和启动阶段要明确安排的资源。

## 这节课真正训练的不是“避免犯错”，而是“看懂程序怎么死”

如果只从代码层面总结，lesson 10 的教训似乎很普通：

1. 局部数组会占栈。
2. 递归会迅速耗尽栈。
3. 数组越界会破坏返回地址。

但 Samek 真正训练你的，其实是另一项更关键的能力：

1. 在调试器里识别栈是否已经跌出 RAM。
2. 区分“栈完全溢出”和“返回地址被破坏”这两种不同灾难。
3. 学会根据症状推断控制流是怎么一步步跑飞的。
4. 用断点策略缩小时间窗口，而不是盲目单步。

对嵌入式工程来说，这种能力比抽象理解更重要。因为系统真正出事时，往往不会给你一句清晰错误提示，只会留下一点寄存器值、一个异常处理函数和一段已经面目全非的控制流。

## 总结

lesson 10 的主题，是通过故意制造两类典型错误，把“栈”从一个静态概念变成一个会真实伤人的系统资源。第一类错误是递归叠加大局部数组，最终把 `SP` 压出合法 RAM，触发 BusFault；第二类错误是数组越界写，精确破坏保存在栈上的 `LR`，让函数在看似正常返回时把控制流送进完全错误的位置。

这两类错误虽然都和栈有关，但本质上揭示的是两种不同的失败模式：一种是资源耗尽，另一种是现场污染。Samek 用它们非常具体地说明了，程序挂死、跑飞、掉进异常处理、甚至把向量表当代码执行，都不是“玄学崩溃”，而是栈布局被破坏之后非常机械的后果。

从课程推进上看，lesson 10 也是一个明显的分水岭。它开始让你认真面对异常、startup code、返回地址、链接配置这些原本像“底层细节”的东西，因为这些细节一旦出错，应用逻辑根本没有资格继续谈。下一课 lesson 11 会暂时从故障调试切回数据类型与表达式规则，但你会发现，Samek 关心的仍然是同一个问题：C 代码看上去简单，底层到底会不会按你以为的那样执行？
