# 16 中断入门：为什么轮询不够，以及 SysTick 如何打断程序

这一课，Miro Samek 终于正式引入中断（interrupt）。前面几课一直在做铺垫，比如寄存器访问、位操作、启动代码、向量表、异常处理入口，这些内容单看可能有些分散，但到了 lesson 16，前面的准备开始汇合成一个非常具体的问题：为什么嵌入式程序不能总靠轮询（polling）工作？以及，中断到底替我们做了什么？

Samek 这一课没有一上来就讲一堆抽象定义，而是先回到大家最熟悉的 blinky 程序。这个程序之前一直靠 `delay()` 忙等，让 LED 一亮一灭。它虽然简单，却正好暴露出最经典的低效写法：CPU 明明没有别的事情要做，却还在死命数循环，只为了“等半秒钟过去”。

## 从 busy-wait 开始：轮询为什么是“最笨”的等待方式

课程先把前几课中的实验性代码清理掉，回到最原始的 blinky 程序。主函数的大意很简单：

1. 初始化 GPIO。
2. 进入 `while(1)`。
3. 打开红灯，调用 `delay()`。
4. 关闭红灯，再调用一次 `delay()`。

这里的 `delay()` 本质上就是一个空转循环。程序不断检查循环计数器是不是减到零，这就是 polling，只不过是最原始、最浪费 CPU 的那种 polling。

Samek 用了一个非常形象的类比：这就像一个人为了第二天不睡过头，整夜都盯着时钟数秒，而不是设一个闹钟。闹钟的关键价值不在于“会响”，而在于你在它响之前可以去做别的事情。中断对处理器的意义也是一样的。

这一步非常重要，因为它不是在说 polling 绝对不能用，而是在强调：

1. 长时间 busy-wait 会把 CPU 完全绑死。
2. CPU 在等待期间无法处理其他任务。
3. 一旦系统要响应多个外部事件，纯 polling 结构会迅速失控。

所以，中断不是“另一种写法”，而是当系统开始面对异步世界时，必须引入的基础机制。

## 什么是中断：它本质上是处理器的“闹钟”和“打断机制”

Samek 对中断的解释很务实。所谓 interrupt，不只是“时间到了会提醒你”，更广义地说，它是在某种外部或内部条件满足时，强制处理器先停下当前工作，转去处理更紧急的事情。

这些条件可以包括：

1. 定时器到期。
2. 按键被按下。
3. 串口收到数据。
4. 模数转换完成。
5. 外设报告错误状态。

也就是说，中断的核心不是“定时”，而是“异步通知”。

这一点和前面单线程顺序执行的 C 程序形成了鲜明对比。普通 C 程序默认认为控制流由自己掌握，下一条执行什么完全由当前代码决定；而中断意味着，程序执行流可以在某个你没有显式写出函数调用的地方，被硬件突然接管。

## 中断不是纯软件机制，它需要硬件支持

Samek 特别强调了一点：中断不是靠软件“模拟”出来的功能，而是硬件和软件共同配合的结果。

用“闹钟”作类比时，光有“设置几点响”的软件逻辑还不够，还得有真正会响的时钟，以及能够听到闹钟的人。对 MCU 来说也一样：

1. 你需要能产生中断请求的外设。
2. 你需要 CPU 内部能够感知中断线状态并切换控制流的硬件。
3. 你还需要软件提供对应的处理函数。

所以 lesson 16 既是在讲一个编程技巧，也是在第一次认真建立“中断是硬件事件驱动的软件入口”这个概念。

## SysTick：这节课选用的第一个中断源

为了把概念落到实处，Samek 选择了 TivaC / Cortex-M 最容易上手的中断源之一：SysTick。

SysTick 是 Cortex-M 内核自带的系统定时器，不是芯片厂商随意定义的某个外设，而是 ARM 架构级别的一部分。这使它非常适合拿来做教学示例，因为：

1. 几乎所有 Cortex-M 都有它。
2. 寄存器数量少，结构简单。
3. 它和 CPU 内核关系很近，便于理解“异常/中断”机制。

课程里介绍的三个关键寄存器是：

1. `SysTick->VAL`：当前计数值。
2. `SysTick->LOAD`：重装载值。
3. `SysTick->CTRL`：控制寄存器。

它是一个 24-bit 的倒计数器。每个时钟周期减一，减到零时产生 SysTick 中断，然后从 `LOAD` 自动重装再继续计数。

所以，`LOAD` 里放什么值，就决定了“隔多久产生一次中断”。

## CPU 是如何“听见”中断的：抢占与异步性

这一课第一次明确引入两个以后会反复出现的关键词：

1. preemption
2. asynchronous

Samek 的解释很清楚：CPU 在正常执行程序时，会在每条指令边界检查中断条件。如果某个中断请求已经有效，CPU 就不会继续按原来的顺序取下一条指令，而是转去执行中断入口序列。这就叫 preemption，也就是“抢占”。

这里最关键的不是“检查中断”本身，而是中断请求线与程序执行流之间并不同步。代码在按时钟一条条执行，但中断条件可能在任意时刻出现，因此中断总被称为 asynchronous event。

这对程序设计的含义非常大：

1. 中断不是你在源码里显式调用出来的。
2. 它可能在任意两个普通指令之间发生。
3. 程序从此不再是单纯的线性因果世界。

也正因为这样，后面几课才会逐步引出 race condition、critical section、事件驱动和 RTOS 等更高级主题。

## 为什么中断入口成本更高

Samek 顺带提到，Cortex-M 的中断入口并不是“零代价跳转”。普通指令如 `MOV`、`ADD` 可能一个周期就能完成，而异常入口往往需要十几个周期起步。

这说明两个事实：

1. 中断非常有价值，但不是免费机制。
2. 设计系统时不应把高频、琐碎、可以批处理的工作都丢进中断里。

这节课先只点到为止，但这个“中断有开销”的认识，后面会在实时性分析中变得越来越重要。

## 先重构 polling 代码：让它更像未来的 ISR

Samek 的教学方式一向很讲层次。他并没有马上把逻辑搬进 `SysTick_Handler()`，而是先把原来 polling 版的 blinky 改成“每轮只 delay 一次”的结构。

核心思路是：

1. 以前是一轮循环里先亮灯再等，再灭灯再等。
2. 现在改成每次循环只做一次“切换状态”，然后等待一次。

这样做的好处是，循环体里除了等待之外的那部分逻辑，就和未来中断处理函数要做的工作几乎一样了。

这里课程顺势引入了一个非常实用的位操作习惯用法：

```c
GPIOF_AHB->DATA_Bits[LED_RED] ^= LED_RED;
```

也就是用 XOR 来翻转 LED 状态。这个写法的意义不只是“代码更短”，更重要的是它把“亮/灭两个分支”压缩成了一个明确的状态切换操作。后面当这段逻辑被放进中断处理函数时，结构就非常自然。

## 配置 SysTick：从硬件定时器取代软件延时

等到 polling 版本整理到位，课程才正式开始配置 SysTick。关键代码可以概括成这样：

```c
SysTick->LOAD = SYS_CLOCK_HZ/2U - 1U;
SysTick->VAL  = 0U;
SysTick->CTRL = (1U << 2) | (1U << 1) | 1U;
```

这里分别完成三件事：

1. 指定半秒计数周期。
2. 清零当前计数值。
3. 选择时钟源、打开中断、启动计数器。

课程还专门解释了为什么 `LOAD` 要减一。因为计数器会数到零，所以如果你想让它覆盖恰好 $N$ 个时钟周期，装入的应该是 $N-1$。

如果系统时钟是 $16\text{MHz}$，半秒对应的周期数就是：

$$
16{,}000{,}000 / 2 = 8{,}000{,}000
$$

因此装入值是：

$$
8{,}000{,}000 - 1 = 7{,}999{,}999
$$

Samek 还提醒你检查这个值是否能装进 24-bit 计数器。这个细节看似简单，但它体现的是很典型的 embedded 思维：配置硬件参数时，永远要确认数值范围是否匹配寄存器位宽。

## 中断处理函数第一次真正派上用场

到了这一步，lesson 15 留下来的向量表和空的 `SysTick_Handler()` 终于开始真正工作。和 polling 版本对比，你会发现 ISR 的任务极其简单：

1. 它不再负责“等待”。
2. 它只负责在中断发生时切换 LED 状态。

也就是说，等待已经从“CPU 空转”转成“硬件计时器在后台计数”。CPU 只有在真正需要动作的时刻，才被叫醒去执行少量工作。

对应的代码在课程工程里非常直白：

```c
void SysTick_Handler(void) {
    GPIOF_AHB->DATA_Bits[LED_RED] ^= LED_RED;
}
```

这段代码的重要性不在于它复杂，而恰恰在于它足够简单，能让你清楚看到中断驱动程序结构和 polling 程序结构的本质差异。

## 主循环为什么可以空下来

在 SysTick 配置完成后，`main()` 里只剩初始化、打开中断和一个空的 `while(1)`：

```c
__enable_interrupt();
while (1) {
}
```

这在初学者看来常常很不习惯，因为会觉得“主循环空着是不是程序没干活”。但 lesson 16 想建立的恰恰是这个新观念：

1. 程序的有效行为不一定发生在主循环里。
2. 在中断驱动设计中，真正的工作可以由异步事件触发。
3. 主循环甚至可以只是一个 idle 占位点。

从这里开始，程序结构已经不再是“main 主导一切”，而是在向“初始化后等待异步事件”转变。这是后面事件驱动架构最早的雏形。

## 向量表的意义第一次真正变得具体

这一课虽然没有把启动代码拿出来重新逐行分析，但你会第一次清楚感受到向量表不是个抽象概念，而是实际把“硬件中断源”映射到“C 函数入口”的桥梁。

在 lesson 16 的工程里，`startup_tm4c.c` 已经把 `SysTick_Handler` 放进了异常向量表，而且还通过 weak binding 提供默认处理函数。现在一旦 SysTick 到期，硬件就会顺着这张表跳到你在 `bsp.c` 里写的那个函数。

这一步把前面零散知识串起来了：

1. 启动文件定义向量表。
2. 向量表给每个异常编号关联入口地址。
3. 外设或内核定时器触发中断。
4. CPU 根据向量表进入对应的 handler。

真正理解到这里，中断对你来说就不再是“IDE 里突然冒出来的一个回调函数”，而是一条完整的硬件到软件的路径。

## 这一课真正完成的转变：从顺序等待到异步响应

如果只看代码量，lesson 16 其实变化不大，甚至比前面某些课还更短。但从程序模型上看，这一课是一个明显分水岭。

之前的 blinky 程序属于典型的顺序式控制流：

1. 做一件事。
2. 等一会儿。
3. 再做下一件事。

而引入 SysTick 之后，程序开始具备异步响应结构：

1. 硬件定时器独立计时。
2. CPU 正常执行当前代码。
3. 到期后由中断抢占。
4. 执行 ISR。
5. 再回到原先代码继续执行。

这看似只是把 delay 换成定时器，实际上却让你第一次进入“真实嵌入式系统”的语境。因为现实中的嵌入式程序几乎都要同时面对多个异步事件，而中断就是进入这个世界的门槛。

## 总结

lesson 16 的重点，不只是会写 `SysTick->LOAD`、`SysTick->CTRL` 和 `SysTick_Handler()`，而是建立了三个非常关键的认识。

第一，中断存在的根本原因，是 busy-wait/polling 在面对异步事件时效率太低、扩展性太差。硬件定时器和中断让 CPU 不必再傻等。

第二，中断是硬件与软件共同完成的机制。SysTick 负责在后台计时，CPU 负责在指令边界检测并进行抢占，向量表负责把异常源映射到你的 C 函数。

第三，引入中断后，程序结构开始从线性顺序执行，转向“初始化 + 等待异步事件 + 被抢占处理”的模型。后面所有关于 exception frame、race condition、critical section、RTOS 和 active object 的内容，都是在这个转变基础上继续展开的。

从课程节奏上说，lesson 16 更像是“中断世界的开门课”。它先让你看到中断为什么必要，以及最简单的周期性中断是怎么替代 delay 的。接下来的 lesson 17 和 18，Samek 才会继续追问更底层的问题：中断到底是怎么进来的？它又是怎么返回的？为什么 Cortex-M 的 ISR 能像普通 C 函数一样写？这些真正困难但也真正有价值的细节，正是从这一课开始铺开的。
