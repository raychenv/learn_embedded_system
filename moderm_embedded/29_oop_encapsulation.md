# 29 面向对象编程与封装

这一课，Miro Samek 开始正式引入 Object-Oriented Programming，重点不是教某一门“面向对象语言”，而是解释面向对象背后的三个核心概念：Encapsulation、Inheritance、Polymorphism。本课先聚焦第一个概念，也就是封装（encapsulation），并且特意从嵌入式开发者熟悉的 C 语言出发，说明 OOP 并不等于“必须使用 C++/Java/Python”，而是一种组织软件的方式。

课程随后做了两件很重要的事：
1. 先用标准 C 手工模拟出 class、object、constructor、operation 这些面向对象概念。
2. 再把同一套设计翻译成 C++，对比两者在语法上的差异，以及在底层机器码上的相似性。

## OOP 不等于某种语言

很多人一提到 OOP，就会立刻想到 C++、Java 或 Python。但 Samek 强调，面向对象首先是一种设计思想，而不是语法特性。它的基础是三个概念：

1. Encapsulation：把数据和操作打包在一起，并隐藏内部实现细节。
2. Inheritance：在已有类型基础上扩展和复用行为。
3. Polymorphism：通过统一接口表现出不同具体行为。

这一课只讲第一个，也就是 encapsulation。

## 从 BSP 到封装思想

课程先回顾前面已经做过的 BSP 设计。`bsp.h` 负责声明“做什么”，`bsp.c` 负责实现“怎么做”。这其实已经体现了两个思想：

1. Abstraction：把 LED 操作抽象为 `on/off/toggle` 这样的简单接口。
2. Information Hiding：板级寄存器细节被隐藏在 BSP 内部，调用者不需要知道。

但 BSP 这种模块化写法有一个明显局限：它适合处理“固定数量”的资源，比如板子上就 3 个 LED；如果面对的是图形界面中的 rectangle、circle、triangle 这类对象，数量往往是开放的、运行时决定的，就不适合继续用“为每个资源单独写一套函数”的方式硬编码。

这时就需要 class/object 这样的抽象。

## 在 C 里手工实现一个 Shape 类

课程用 `Shape` 作为例子演示如何在 C 里模拟 class。

### 数据部分

`Shape` 用一个结构体表示，其属性只有两个坐标：`x` 和 `y`。由于屏幕范围有限，使用 16 位有符号整数即可，允许负值则表示图形在屏幕外。

这一步本身并不新鲜，前面课程已经讲过 `struct`。关键变化在于：从现在开始，不再鼓励直接访问结构体成员，而是通过一组专门函数来操作它。

### 操作部分

课程为 `Shape` 提供了三类函数：

1. 构造函数 `Shape_ctor()`：初始化对象坐标。
2. `Shape_moveBy()`：按给定偏移移动图形。
3. `Shape_distanceFrom()`：计算两个图形之间的距离。

其中有两个非常重要的编码约定：

1. 所有关联函数名都以类名开头，比如 `Shape_moveBy`。
2. 所有关联函数的第一个参数都是指向对象本身的指针，课程里把它命名为 `me`。

这个 `me` 指针，本质上就是后面 C++ 中的 `this` 指针。

### const 的含义

这一课顺带复习了 `const` 在指针声明中的位置含义：

1. `Shape * const me` 表示指针本身不能改指向，但对象内容可以改。
2. `Shape const * me` 表示对象内容不能通过该指针修改。

这让接口表达更精确，也更符合“接口应该容易被正确使用、难以被错误使用”的设计原则。

## class 和 object 到底是什么

Samek 在这里给了一个很经典的解释：

1. class 是“模板”或“cookie-cutter”。
2. object 是按这个模板创建出来的具体实例。

从 OOP 术语看：

1. class 中的数据叫 attributes。
2. class 中的函数叫 operations。

所以即便在 C 里只是“结构体 + 一组以类名开头的函数”，本质上已经是在模拟一个 class 了。

## 对象的创建方式

这课还特别强调了 object 可以有多种分配方式：

1. 静态分配：例如全局或静态对象 `s1`。
2. 自动分配：例如函数栈上的局部对象 `s2`。
3. 动态分配：例如用 `malloc()` 在堆上创建对象 `ps3`。

课程也明确提醒：动态内存分配在实时嵌入式系统里通常是坏主意，这里只是为了完整说明“对象可以如何实例化”。如果真的使用堆，还必须显式 `free()`，并在启动文件里配置足够的 heap 空间。

## constructor 的意义

OOP 的另一个基本规则是：对象在被使用前必须先初始化。

在 C 里这件事要手工完成，所以创建对象后必须显式调用 `ctor`：

1. `Shape_ctor(&s1, ...)`
2. `Shape_ctor(&s2, ...)`
3. `Shape_ctor(ps3, ...)`

这和前面课程里 QP/C 框架对 `QXThread` 对象的初始化方式是同一种模式。也就是说，课程其实早就已经在用“对象 + 构造函数”的思想，只是这一课才把它提升到 OOP 视角来解释。

## 通过操作而不是直接访问数据

对象初始化之后，就应该通过定义好的 operation 来操作它，而不是直接读写属性。

课程用几个断言验证 `distanceFrom()` 的性质：

1. 一个对象到它自己的距离必须是 0。
2. `distance(s1, s2)` 必须等于 `distance(s2, s1)`。
3. 距离满足类似三角不等式的约束。

此外，课程还演示了 const 对象不能调用会修改状态的 `moveBy()`。编译器会直接报错。这再次体现了好的类接口应当把错误尽量提前到编译期暴露出来。

## 在底层，封装其实还是普通函数调用

这一部分很有嵌入式味道。Samek 不满足于讲概念，而是继续往汇编层面解释：在 Cortex-M 上，调用类操作时，传入的 `me` 指针会按 AAPCS 放进 `r0` 寄存器。

因此：

1. 每次进入 class operation，`r0` 里都有当前对象指针。
2. 类属性访问会被编译成“基址寄存器 + 偏移”的寻址形式。
3. 调试时只要看 `me`，就能直接看到当前对象内容。

这说明在 C 中模拟封装并不低效，反而非常直接、可控，而且很适合调试。

## 把同一设计翻译成 C++

接下来课程把整个 lesson 29 从 C 改写成 C++，目的不是展示 C++ 有多“高级”，而是让你直接看到：

1. C++ 能自然支持 class、constructor、private/public。
2. 但它在底层并没有凭空制造魔法，很多机制与 C 版本本质相同。

### 第一步：把 C 工程编译成 C++

课程先把 `.c` 文件改成 `.cpp`，然后重新加入工程。此时最先出现的是若干类型检查错误，比如：

1. `malloc()` 返回 `void *`，在 C 里可以隐式转换。
2. 在 C++ 里则必须显式转换成目标指针类型。

修掉这些小问题后，原先按 C 风格写的代码其实就可以被 C++ 编译器接受。这说明对 C 程序员来说，学习 C++ 的第一步完全可以是“把它当成更严格的 C”。

### 第二步：把 Shape 真正改成 C++ class

改写后的 `Shape` 类体现了几个典型语法变化：

1. `struct`/`typedef` 组合变成 `class Shape`。
2. 属性放到 `private:` 下，真正受到语言级访问限制。
3. 操作放到 `public:` 下，供外部调用。
4. 构造函数名称直接就是类名 `Shape(...)`。
5. 不再需要显式传 `me` 指针，因为编译器会隐式提供 `this`。
6. 如果成员函数不修改对象，需要把 `const` 写在函数签名末尾。

这相当于把原来依赖“编码约定”维持的封装，升级成由语言本身强制约束的封装。

### 第三步：实现文件中的变化

在 `shape.cpp` 中，成员函数定义使用 `Shape::moveBy` 这样的作用域解析运算符，而不是 C 里的 `Shape_moveBy`。原来函数体中所有通过 `me->x`、`me->y` 访问属性的地方，都可以替换为：

1. 显式 `this->x`、`this->y`
2. 或直接写 `x`、`y`，由编译器隐式补全 `this->`

构造函数还可以使用 C++ 风格的 initializer list 来初始化成员。

### 第四步：对象使用方式变化

在 `main.cpp` 里，对象使用方式也更“像对象”了：

1. 静态对象在定义时直接初始化。
2. 自动对象在栈上定义时直接调用构造函数。
3. 动态对象用 `new` 分配，用 `delete` 释放。
4. 调用成员函数时使用 `.` 或 `->`。

从语法上看，确实比 C 里显式传 `me` 更自然。

## C++ 幕后到底做了什么

这是本课最值得注意的部分。Samek 在调试器里验证了几个事实：

1. 静态对象的构造函数会在 `main()` 执行前，由额外的 C++ 初始化代码自动调用。
2. 自动对象会在定义点附近调用构造函数。
3. `new` 底层最终仍然会调用 `malloc()` 分配内存。
4. 动态对象分配完内存后，构造函数会自动执行。

更关键的是，继续查看汇编会发现：

1. `moveBy()` 的调用方式与 C 版本等价。
2. 成员函数内部依然存在 `this` 指针。
3. 访问对象属性时依然是“对象指针 + 成员偏移”。
4. 很多机器码与前面的 C 实现完全一致。

课程借此传达的核心观点是：C++ 的 OOP 语法主要是在提升表达能力和安全性，而不是引入神秘的运行时开销。对于这种简单封装场景，C 和 C++ 的底层实现几乎可以做到一模一样。

## 封装并不能自动解决并发问题

课程最后又把话题拉回 RTOS，这一点非常重要。

Samek 演示了一个例子：把某个 `Shape` 静态对象 `s1` 设为共享对象，然后：

1. 在 `blinky1` 线程里调用 `moveBy()` 修改它。
2. 在 `blinky2` 线程里调用 `distanceFrom()` 读取它。

虽然所有访问都通过“封装良好的类接口”进行，但断言依旧可能失败，因为两个线程仍然在并发访问同一个共享对象，于是发生 race condition。

这说明：

1. encapsulation 解决的是接口组织和信息隐藏问题。
2. encapsulation 本身并不提供并发安全。
3. class method 在底层仍然只是普通函数调用，不会天然防止线程竞争。

所以，如果想要“并发意义上的真正封装”，还需要更进一步的设计，比如后续课程会讲到的 active object。

## active object 的引出

课程没有在这一课展开 active object 细节，但已经给出方向：

1. 它把并发编程、面向对象编程、事件驱动编程结合在一起。
2. 它的目标是避免传统 shared-state concurrency 模型里大量共享对象带来的 race condition。
3. 这也是 Samek 后续课程的重要主线之一。

## 总结

29 课的重点，不是“学会 C++ 语法”，而是建立对 OOP 尤其是 encapsulation 的正确理解：

1. OOP 首先是设计方法，不依赖某一门语言。
2. 在 C 里完全可以通过“结构体 + 关联函数 + `me` 指针”实现 class 风格封装。
3. C++ 只是把这种模式变成了语言内建能力，并用 `private/public/constructor/this` 提供更自然、更安全的表达。
4. 对于简单类封装，C++ 生成的底层代码可以与 C 版本几乎完全一致。
5. 封装不等于并发安全；在 RTOS 中，共享对象依然会发生 race condition。
6. 如果要进一步解决并发问题，需要走向 active object 这样的更现代架构。

这一课相当于为后面讲 inheritance 和 polymorphism 打基础，也把“现代嵌入式软件为什么越来越强调对象、事件和架构”这个问题正式铺开了。
