# 18 ARM Cortex-M 如何处理中断：为什么 ISR 可以写成普通 C 函数

这一课，Miro Samek 终于把一个很多人一开始都会觉得“理所当然”、但其实很不简单的现象讲透了：为什么在 ARM Cortex-M 上，中断服务函数往往可以直接写成普通 C 函数的样子？

如果你只在 Cortex-M 上写过代码，可能会觉得 `void SysTick_Handler(void)` 这种写法再自然不过。但在上一课，Samek 刚刚通过 MSP430 告诉你：在很多处理器上，中断和普通函数之间有非常本质的差别：

1. 中断返回时要用不同的机器指令。
2. 中断通常还要保存比普通函数更多的寄存器。

也就是说，从传统嵌入式 CPU 的视角看，“ISR 像普通 C 函数一样写”其实并不是默认情况，而是一个需要被解释的现象。

第 18 课的任务，就是把这个现象的底层原因拆开：Cortex-M 在中断入口到底自动做了什么，中断返回为什么还能沿用普通函数形式，以及这一切是如何和 ARM 的过程调用标准（AAPCS）精确配合起来的。

## 从上一课的 MSP430 对照出发：问题到底是什么

Samek 这一课不是直接从 Cortex-M 讲起，而是先回顾 lesson 17 的 MSP430。这样做非常重要，因为它先把“中断与普通函数为什么不同”这个问题重新钉牢。

在 MSP430 上，你已经看到两件事：

1. 普通函数用普通返回指令，例如 `RET`。
2. 中断服务函数要用专门的中断返回指令，例如 `RETI`。

原因也很直接：

1. 普通函数返回时只需要恢复普通调用现场。
2. ISR 返回时要恢复中断相关状态，语义更复杂。

而且，MSP430 的 ISR 往往还要保存比普通函数更多的寄存器。所以 lesson 18 的核心问题就自然形成了：

如果 Cortex-M 上的 ISR 看起来像普通 C 函数，那它是怎么同时满足这两个要求的？

## lesson 18 的实验目标：精确控制中断何时发生

Samek 仍然沿用前面一贯的风格，不是先给结论，而是先做实验。但要分析中断入口和返回，必须能精确控制中断发生的时刻。

在 MSP430 那一课里，可以通过改定时器计数器的值，让中断在下一拍触发。但 Cortex-M 的 SysTick 不适合用这个办法，因为 `STCURRENT` 是 write-clear 寄存器，往里写值只会清零，并不会像普通计数器那样“逼近触发点”。

所以这节课改用 Cortex-M 提供的另一个更直接的方法：手动把 SysTick 的 pending 位设起来。

## 用 `PENDSTSET` 手工触发 SysTick 中断

在 Cortex-M 里，System Control Block 的 ICSR（Interrupt Control and State Register）中有一个非常好用的位：`PENDSTSET`。把它置 1，就能把 SysTick 中断直接置为 pending。

这意味着：

1. 你不需要等真实计时器自然溢出。
2. 你可以在任意你选定的机器指令边界前，安排一次即将到来的 SysTick。
3. 这样就能非常精细地观察中断是在哪里抢占了正常执行流。

课程还顺带强调了一点：Cortex-M 不只 SysTick 可以这么触发，很多中断源都有对应的 pending 机制。这是一种非常重要的调试技巧，因为它让中断分析从“碰运气”变成“可控实验”。

## 不要单步，要让程序“自由运行”一次

为了判断中断会不会在某个特定指令之后立刻抢占，课程做了一个非常讲究的实验布置：

1. 先在 `while(1)` 循环的某一条机器指令前停住。
2. 在 `SysTick_Handler()` 里再放一个断点。
3. 手动把 `PENDSTSET` 置 1。
4. 然后不是单步，而是点 `Resume` 让程序自由运行。

Samek 专门提醒，不能用单步继续，因为调试器单步机制会影响中断检查时机，破坏实验。只有让处理器真正跑起来，才能看到硬件在指令边界上的实际行为。

实验结果非常明确：程序会先进入 `SysTick_Handler()`，说明中断确实能在你安排的位置精确抢占当前线程代码。

这一步一旦建立起来，后面的整个 lesson 18 就有了实验基础：

1. 你能决定中断何时发生。
2. 你能看清入口前后寄存器和栈的变化。
3. 你能一步步分析 ISR 为什么可以像普通函数那样写。

## 先关掉 FPU：避免一开始就被扩展栈帧干扰

在正式看 exception entry 之前，课程先让你在工程选项里把 FPU 设成 `None`。原因很实际：FPU 会让中断栈帧更大、更复杂，而这一课一开始只想把 Cortex-M 最基本的 exception entry/return 讲清楚。

这一步也说明 Samek 一贯的讲法特点：先讲最小、最本质的情况，等主机制搞清楚后，再引入扩展情况。

## 中断一进来，SP 为什么一下掉了 8 个字

关闭 FPU 后，再次做同样的中断触发实验，并把 memory 窗口调到栈附近观察。你会看到一个特别醒目的现象：

1. 一进入 `SysTick_Handler()`，`SP` 明显向下跳了一大截。
2. 恰好是 8 个 32-bit words 的空间。

也就是说，中断入口时，硬件自动在栈上压入了一个 8-word 的 exception stack frame。

这一步是 lesson 18 的第一个核心发现：

Cortex-M 的中断入口并不是“先跳到你的 C 函数，让编译器自己想办法”，而是硬件已经先把一组关键寄存器压栈了。

## 这 8 个寄存器为什么如此关键

课程接着让你对照 datasheet 里的 exception stack frame 图，再结合调试器内存窗口，识别出到底保存了哪些寄存器。最重要的是：

1. 其中包含返回地址，也就是中断结束后要回去继续执行的 `PC`。
2. 还包含了一组非常“眼熟”的通用寄存器。

Samek 在这里埋了一个很漂亮的伏笔：这组寄存器如果你在 lesson 9 认真学过 AAPCS，会立刻觉得很熟。

因为 Cortex-M 自动压栈保存的，恰好是 AAPCS 中属于 caller-saved 的那组关键上下文。

## AAPCS 与 exception stack frame 的“互补关系”

这是整节课最关键的结论之一。

在 lesson 9 里，AAPCS 规定了函数调用时哪些寄存器由调用者负责保存，哪些由被调用者负责保存。到了 lesson 18，你会发现 Cortex-M 在异常入口自动保存的，正好是和普通 C 调用约定互补的那一组寄存器。

这意味着：

1. 硬件先帮 ISR 保存了调用者一侧最关键的现场。
2. 编译器再按普通 C 函数规则为 ISR 生成代码。
3. 如果 ISR 自己还需要用到更多 callee-saved 寄存器，仍按普通函数规则处理即可。

于是，“ISR 是普通 C 函数”这件事不再神秘，它的真正含义是：

Cortex-M 的异常入口机制被设计成正好补足 AAPCS，所以普通函数调用约定可以直接复用于中断处理。

Samek 甚至明确指出，这使得 AAPCS 不再只是函数调用规范，而是在 Cortex-M 上被提升成了更广义的上下文切换基础规则。

## lesson 18 回答的第一个问题：为什么 ISR 可以像普通函数那样写

到这里，第一个大问题已经有答案了。

在 MSP430 上，中断和普通函数差别很大，是因为硬件没有把异常入口专门设计成兼容通用 C 调用语义；而在 Cortex-M 上，硬件自动压栈的寄存器集合与 AAPCS 互补，因此：

1. ISR 可以按普通 C 函数的方式生成代码。
2. 不需要专门的 ISR 语法糖来补足现场管理。
3. 开发者因此获得了极高的一致性和可读性。

这就是 Cortex-M 在可编程性上非常突出的地方。

## 第二个问题：既然 ISR 是普通函数，它为什么还能“特殊返回”

不过到这时，问题还只解决了一半。即使你接受“中断入口由硬件自动压栈，因此 ISR 可以用普通函数体”，你仍然会问：

普通函数不是应该用普通返回方式吗？那它怎么知道自己是在“从中断返回”，而不是在普通地跳回调用者？

课程接下来就专门看中断返回过程。

## `BX LR` 看起来普通，但 `LR` 里装的根本不是普通地址

单步执行到 ISR 尾部时，你会发现返回指令仍然是很普通的：

```asm
BX LR
```

如果你只看这一条，会觉得它和普通函数返回没区别。但一看 `LR` 的内容，就发现完全不是那么回事。课程里典型看到的值是：

```c
0xFFFFFFF9
```

这根本不是正常代码空间里的可执行地址。Samek 直接说：这是一个 ARM-special。

也就是说，Cortex-M 并不是靠一条专用“中断返回指令”来区分语义，而是靠 `LR` 中的一个特殊数据值来告诉硬件：

“这不是普通返回，请按 exception return 的规则处理。”

## Cortex-M 用“特殊数据”替代“特殊指令”

这一步是 lesson 18 的第二个核心结论。

和 MSP430 这种需要单独 `RETI` 指令的设计不同，Cortex-M 让 ISR 仍然使用普通的分支返回指令，但在 `LR` 中预装一个特殊值。只要 `BX LR` 看到的是这种特殊值，硬件就不把它当普通分支，而是启动 exception return 过程：

1. 自动从栈上恢复之前保存的那组寄存器。
2. 恢复 `SP` 到中断前位置。
3. 把 `PC` 还原到被抢占位置的下一条指令。
4. 回到 thread mode 或其他相应执行状态。

这解释了为什么你会看到：

1. `SP` 瞬间恢复。
2. 通用寄存器内容回到中断前。
3. 程序继续从原先被打断的位置运行。

## 为什么 Samek 说 ARM 这种做法更灵活

课程还特别评价了这套设计：把“中断返回语义”编码在数据里，而不是硬编码成一条专用指令，会更灵活、更可扩展。

原因是：不同类型的异常返回，可以通过不同的 `LR` 特殊值表达。例如：

1. 是否返回到 thread mode
2. 是否使用 MSP 或 PSP
3. 是否伴随 FPU 扩展栈帧

这些差异都可以通过返回码的不同模式来编码，而不需要发明一堆不同的 return-from-interrupt 指令。

这也是 Cortex-M 架构里非常典型的设计哲学：

让硬件语义足够丰富，但尽量不把软件接口搞得碎片化。

## Main Stack Pointer 与 Process Stack Pointer：为 RTOS 埋下伏笔

在解释 exception return 类型时，课程顺带提到了 MSP 和 PSP，也就是 Cortex-M 的两套栈指针：

1. Main Stack Pointer
2. Process Stack Pointer

Samek 在这节课里没有展开到 RTOS 层面，但已经明确告诉你：这是 register banking 的一部分，后面谈 RTOS 上下文切换时会非常重要。

这一步的意义在于：lesson 18 看起来只是在讲中断细节，实际上已经开始为后面的线程/RTOS/上下文切换做铺垫。

## 为什么有时候栈帧会多出一个“aligner”字

课程接着解释了一个很容易在调试器里看懵的现象：有时 exception stack frame 不是 8 个字，而是看起来多空出一个字位置。

这和 8-byte stack alignment 有关。AAPCS 要求栈在关键边界上满足 8 字节对齐，以便硬件做高效块传输。所以如果中断到来时 `SP` 恰好没有 8 字节对齐，硬件会额外插入一个对齐填充字，也就是所谓的 aligner。

课程甚至在调试器里故意把 `SP` 人工减 4，制造未对齐状态，再触发中断。结果你就能清楚看到：

1. 8-word 的核心 exception frame 仍然存在。
2. 但前面会额外空出一个字，用来恢复对齐。
3. 中断返回时，这个 9-word frame 会整体被一起弹掉，`SP` 回到原先那个未对齐值。

Samek 也特别提醒：正常编译器生成的 AAPCS 兼容代码里，这种未对齐一般不应该发生。这个实验主要是为了让你理解硬件为什么需要那个 aligner，而不是说实际代码里经常会这样。

## 12 个时钟周期：为什么 Cortex-M 中断入口/出口这么快

在解释对齐时，课程还强调了 Cortex-M 异常入口和出口的效率。即便要压入或弹出这么多寄存器，中断入口和退出通常只需十几个时钟周期左右，这是因为硬件专门为这种块式栈传输做了优化。

这一步也让你更容易理解为什么 Cortex-M 如此适合做中断密集型嵌入式系统：

1. 中断入口成本低。
2. 异常返回路径短。
3. 与 C 调用规则结合得又很好。

## 重新打开 FPU：中断栈帧会瞬间膨胀

当基础 exception frame 讲清楚后，课程再把 FPU 打开，让你重新做同样的实验。结果非常震撼：

1. 中断入口后 `SP` 下移的距离大得多。
2. 栈帧从原来的 8-word 级别，一下膨胀到带浮点寄存器保存的扩展异常栈帧。
3. `LR` 中的 exception return 特殊值也不再是 `0xFFFFFFF9`，而会变成另一类 FPU 相关返回码，例如 `0xFFFFFFE9`。

这直接说明：

1. FPU 不只是“算浮点更快”。
2. 它还会显著扩大中断上下文保存成本。
3. 同时拉长中断入口与出口时间。

## FPU 对实时系统意味着什么

Samek 在这里给出的道理非常务实：如果你的工程开启 FPU，就必须把堆栈尺寸和中断开销重新估算，而不能沿用“无 FPU”时的经验。

因为：

1. 栈空间需求会大得多。
2. 中断嵌套时风险上升。
3. 入口/出口时间也更长。

这其实已经在把 lesson 18 和后面的实时系统设计联系起来了。硬件特性从来不只是“性能选项”，它会真实改变你的时序预算与资源预算。

## lesson 18 真正回答了什么

如果把整节课压缩成一句话，它真正回答的其实是：

Cortex-M 为什么能把“中断处理”这件在很多处理器上相当特殊的事情，尽可能做成和普通 C 调用一致。

答案是三层协同：

1. 硬件在 exception entry 自动压入关键寄存器。
2. 自动压栈的寄存器集合与 AAPCS 互补。
3. exception return 又通过 `LR` 中的特殊返回码，让普通 `BX LR` 指令触发特殊中断返回语义。

也就是说，ISR 并不是“真的和普通函数毫无区别”，而是 Cortex-M 把必要的特殊性尽可能藏在硬件机制和 ABI 配合里，从而让软件层看起来更统一。

## 总结

18 课的核心，是解释 ARM Cortex-M 为什么能够把 ISR 写成普通 C 函数的样子，以及这种能力背后依赖的硬件与 ABI 设计。

这一课最重要的结论可以概括为：

1. 相比 MSP430 这类更传统的 MCU，中断在 Cortex-M 上之所以更“像普通函数”，并不是因为中断不特殊，而是因为硬件替软件做了更多异常入口/退出工作。
2. Cortex-M 在异常入口时会自动压入一组关键寄存器，形成基本 exception stack frame。
3. 这组自动保存的寄存器与 AAPCS 调用约定形成精确互补，因此普通 C 函数调用规则可以直接复用于 ISR。
4. ISR 虽然用普通的 `BX LR` 返回，但 `LR` 中装的是特殊的 exception return 编码，例如 `0xFFFFFFF9`，硬件据此执行异常返回流程。
5. 8-byte 对齐要求会在必要时引入额外的 aligner 字，说明 Cortex-M 的中断栈帧还受到 ABI 对齐规则约束。
6. 一旦打开 FPU，中断栈帧会显著膨胀，返回码类型也会变化，这会直接增加栈需求和中断延迟。
7. lesson 18 不只是解释 ISR 为什么能写成普通函数，也是在为后续 RTOS、双栈指针、上下文切换和实时性能分析打基础。

如果说第 17 课通过 MSP430 的对照让你看到“中断本来应该有多特殊”，那么第 18 课就是在解释：Cortex-M 是怎样通过硬件和 ABI 的精心配合，把这种特殊性尽量吸收到体系结构内部的。也正因为这样，Cortex-M 才会成为现代嵌入式事件驱动与 RTOS 设计中如此友好的处理器平台。
