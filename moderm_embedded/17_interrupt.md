# 17 从 MSP430 反看 Cortex-M：为什么有些处理器的 ISR 不是普通 C 函数

lesson 16 已经把中断第一次真正放进程序结构里了。你已经看到，使用 SysTick 之后，原来依赖 `delay()` 忙等的 blinky 程序，变成了一个由周期性中断驱动的程序。看上去一切都很自然，尤其是 `SysTick_Handler()` 这个函数，几乎就像普通 C 函数一样简单。

但 Miro Samek 在这一课故意把视角一转，提出一个很多 Cortex-M 开发者平时根本不会追问的问题：为什么 ISR 可以写得这么“普通”？这是所有处理器的共同特性，还是 ARM Cortex-M 的特殊设计？

为了回答这个问题，lesson 17 不再只盯着 TivaC，而是把 MSP430 拉进来做对照。通过比较这两种处理器在中断入口和返回时的做法，Samek 想让你真正理解：Cortex-M 的方便不是理所当然，而是架构层面精心设计出来的结果。

## 先从一个反直觉现象开始：Cortex-M 的 ISR 居然可以直接调用

课程一开始，Samek 先在 TivaC 的代码里做了一个看似很随意、但其实意义非常重的修改：直接在 `main()` 里调用 `SysTick_Handler()`。

工程中的代码大致是这样：

```c
SysTick_Handler();

__enable_interrupt();
while (1) {
    GPIOF_AHB->DATA_Bits[LED_GREEN] = LED_GREEN;
    GPIOF_AHB->DATA_Bits[LED_GREEN] = 0U;
}
```

如果你只是把 `SysTick_Handler()` 当一个普通函数名来看，这一行似乎没什么奇怪。但如果你认真想，中断服务函数本来应该是“被硬件打断后异步进入”的代码入口。既然如此，它为什么还能像普通函数一样被 `main()` 显式调用？

Samek 用断点和反汇编证明了三件事：

1. 直接调用 `SysTick_Handler()` 时，编译器生成的是普通 `BL`（branch with link）调用。
2. 函数尾部返回用的是普通 `BX LR`。
3. 通过中断真正抢占进入时，逻辑效果一样，红灯依旧被翻转。

这说明在 Cortex-M 上，至少从函数体的角度看，ISR 的确可以长得像标准 C 函数。

## 这并不是常态，而是 Cortex-M 的特殊之处

Samek 很快指出，这种“ISR 像普通函数”的感觉，其实是非常少见的。在很多传统处理器上，中断处理函数根本不是普通 C 函数，而是需要额外语法、额外入口约定、额外返回指令的特殊实体。

所以这一课的任务，不是继续夸 Cortex-M 好用，而是要把对比对象找出来。只有看到另一种处理器怎么做，你才会明白 Cortex-M 到底特殊在哪里。

于是，课程切换到 MSP430 LaunchPad。

## 为什么用 MSP430 做对照

MSP430 是非常经典的小型 MCU 架构，和 Cortex-M 相比，它的中断机制更接近许多传统嵌入式处理器的思路。Samek 之前在课程里已经短暂用过这块板子，所以现在拿它来做对照非常合适。

他准备了一个和 TivaC 版本几乎等价的 blinky：

1. 初始化 LED GPIO。
2. 配置 `Timer0_A` 周期中断。
3. 使能全局中断。
4. 主循环里快速开关另一个 LED，用来制造一段容易被中断抢占的线性代码。

从应用功能上看，两边程序几乎一样；但正因为应用行为相似，底层实现差异才更容易被看清楚。

## MSP430 的 ISR 为什么不是普通函数

在 MSP430 工程里，课程给出的处理函数不是普通的：

```c
#pragma vector=TIMER0_A0_VECTOR
__interrupt void Timer0_Handler(void) {
    LED_toggle();
}
```

这里最关键的是两个非标准元素：

1. `__interrupt`
2. `#pragma vector=...`

它们都不是标准 C 语言的一部分，而是 IAR 为 MSP430 提供的扩展。

这就意味着：

1. 这个函数不是普通 ISO C 函数。
2. 编译器会为它生成专门的入口/出口代码。
3. 它的返回方式和寄存器保存策略都可能不同于普通函数。

Samek 在这里要传达的重点非常明确：

如果一个处理器不能把“异常上下文管理”设计成与常规调用约定兼容，那 ISR 就只能变成一种特殊函数。

## 为了研究中断，先得学会“按你想要的时机”触发它

Samek 的一大特点，是从不满足于“中断来了，说明程序没错”。他更关心的是：能不能把中断精确安排在某个选定指令后面触发？只有这样，才能真正研究中断入口和返回发生了什么。

在 MSP430 上，课程通过定时器寄存器来实现这种人工触发。`Timer0_A` 是一个 16-bit 向上计数器，计到 `TACCR0` 时触发中断。于是只要在调试器里把 `TA0R` 写成“距离 `TACCR0` 只差 1”的值，就等于人为安排了“下一拍就中断”。

这一步的价值非常大，因为它把中断从“不知道什么时候来”的偶发事件，变成了“我现在就让它来”的可控实验。

Samek 也顺势强调了一个以后会反复用到的调试原则：

1. 不要被动等待稀有问题发生。
2. 要尽量构造实验，让问题按你指定的时机发生。

这一点对后面分析 race condition、fault injection、RTOS 上下文切换都非常关键。

## 为什么不能单步，而必须让程序自由运行

在安排好中断时机后，课程设置了两个断点：

1. 一个在主循环里原本会继续执行的下一条指令。
2. 一个在 `Timer0_Handler()` 入口。

接下来最关键的一步，是不能单步。因为很多调试器在单步模式下会影响中断检查，使实验结果失真。必须让程序真正 free run 一次，才能看到硬件真实的抢占行为。

结果非常清楚：程序先停在 `Timer0_Handler()`，而不是主循环里的下一条普通指令。这就证明中断确实在你安排的位置发生了抢占。

## MSP430 的中断入口：SP 为什么下降了 4 个字节

接下来 Samek 开始观察寄存器和栈。MSP430 是 16-bit 架构，所以栈以 2-byte 为单位看最合适。实验中能看到，一旦中断发生，`SP` 从原来的位置往下掉了 4 个字节，也就是两个 16-bit 栈项。

根据 MSP430 文档，这两个被硬件压栈的值是：

1. `SR`，即状态寄存器。
2. `PC`，即返回地址。

这就回答了“中断结束后怎么知道回哪里”的第一个问题。答案很直接：硬件在异常入口时，已经把返回现场压进栈里了。

这也是中断机制与普通线性执行最大的不同之一。中断不是简单跳转，而是一次“保存现场 + 改变控制流”的组合动作。

## MSP430 的 RETI：它不是普通 return

当你在 `Timer0_Handler()` 末尾单步时，会看到一个非常关键的指令：`RETI`。这就是 MSP430 的 return-from-interrupt 指令。

它和普通函数返回不同，因为它不只是“跳回去”，还要：

1. 恢复之前保存的 `SR`。
2. 恢复被打断时的 `PC`。
3. 连同全局中断使能位 `GIE` 一起恢复原状态。

Samek 这里要你真正看到的是：在 MSP430 这类处理器上，“中断返回”是专门的体系结构概念，所以它需要专门的机器指令承载语义。也正因为如此，ISR 天然就不能完全等同于普通 C 函数。

## 为什么 ISR 往往还要保存比普通函数更多的寄存器

课程并没有停留在“有个 `RETI` 指令”这一级，而是继续追问一个更深的问题：如果 ISR 的函数体和普通函数一模一样，编译器为什么还要把它们区别对待？

Samek 做了一个很漂亮的对照实验：

1. 把 LED 翻转逻辑抽成普通函数 `LED_toggle()`。
2. 让 `Timer0_Handler()` 去调用这个函数。
3. 再造一个内容几乎一样的普通函数 `Timer0_Function()`。
4. 在 `main()` 中调用 `Timer0_Function()`，作为控制样本。

代码大意如下：

```c
void LED_toggle(void) {
    P1OUT ^= LED1;
}

#pragma vector=TIMER0_A0_VECTOR
__interrupt void Timer0_Handler(void) {
    LED_toggle();
}

void Timer0_Function(void) {
    LED_toggle();
}
```

然后在反汇编里比较两者。

## 对照实验的结论：普通函数和 ISR 的序言/尾声完全不同

实验结果非常有说服力。

普通函数 `Timer0_Function()` 的代码非常短，基本只是调用 `LED_toggle()`，然后普通返回。

而 `Timer0_Handler()` 就复杂得多。编译器会在进入 ISR 时额外压栈一组寄存器，在调用 `LED_toggle()` 之后再按相反顺序恢复它们，最后用 `RETI` 返回。

为什么会这样？因为普通函数调用是同步的，编译器知道调用点在哪里、知道调用约定是什么，也知道哪些寄存器按 ABI 允许被 clobber。

但 ISR 完全不同：

1. 它可能在任意普通代码位置异步打断程序。
2. 被打断代码并没有“同意”让某些寄存器被破坏。
3. 所以 ISR 必须更保守地保存上下文。

这就是为什么在很多传统架构里，中断函数需要比普通函数更重的保存/恢复逻辑。

## 这里真正要理解的是“异步”带来的约束

lesson 17 的关键，不是记住 MSP430 到底压了哪几个寄存器，而是理解“异步抢占”对编译器和 ABI 提出了什么要求。

普通函数调用的本质是双方都遵守过程调用标准（calling convention）：

1. 调用者知道哪些寄存器可能被改。
2. 被调用者知道自己该保存什么。
3. 双方在编译期就已经形成契约。

而中断不是由当前代码主动发起的，它随时可能闯进来。于是，ISR 的保存/恢复策略必须保证：无论它打断了谁，程序都还能像什么也没发生过一样恢复继续执行。

也正因为这个要求更强，大多数处理器都没办法让 ISR 彻底等同于普通函数。

## Cortex-M 的“普通 ISR”到底特殊在哪里

讲到这里，lesson 17 其实已经把 lesson 18 的问题推到台前了。

你已经看到：

1. 在 MSP430 上，ISR 要靠 `__interrupt` 这类扩展语法声明。
2. 它要通过专门 `#pragma vector` 方式绑定向量。
3. 它必须用 `RETI` 返回。
4. 编译器还会生成比普通函数更重的现场保护代码。

而在 Cortex-M 上，你却可以看到：

1. `SysTick_Handler()` 可以写成普通 C 函数原型。
2. 甚至可以直接从 `main()` 里调用。
3. 返回看上去也是普通 `BX LR`。

这就把真正的问题逼出来了：

如果 MSP430 才更像传统处理器，那 Cortex-M 为什么能这么特别？它把那些“ISR 本该特殊处理”的复杂性藏到哪里去了？

Samek 在这一课没有完全展开最终答案，但已经把问题边界划得非常清楚：答案一定在硬件异常入口/出口机制与 ARM 调用约定之间的配合上。下一课正是沿着这条线，去分析 Cortex-M 的 exception stack frame 和 exception return。

## 这一课的一个隐藏收获：你开始学会用实验研究中断，而不是只会“配出来就行”

如果只看表面，lesson 17 好像主要是在讲 MSP430 和 `RETI`。但它更深的一层意义，是 Samek 在训练你一种工程习惯：

1. 主动控制中断触发时刻。
2. 用断点、寄存器窗口和内存窗口观察入口/返回行为。
3. 用对照实验区分“普通函数”和“ISR”到底差在哪里。

这和很多教材最大的不同在于，它不满足于给一个“中断发生时自动保存现场”的泛泛说法，而是要求你真的在调试器里看见这个“自动保存现场”具体表现成什么。

这种实验方法本身，就是嵌入式调试能力的重要组成部分。

## 总结

lesson 17 的核心价值，是借助 MSP430 这个对照样本，让你意识到：ISR 并不天然等于普通 C 函数。恰恰相反，在很多传统 MCU 上，中断处理函数之所以“特殊”，是因为它们必须承担普通函数不需要承担的上下文保存和返回语义。

通过 MSP430 的实验，Samek 展示了几个关键事实：中断入口时硬件会把 `SR` 和 `PC` 压栈；ISR 返回必须使用专门的 `RETI` 指令；而当 ISR 内部再调用普通函数时，编译器还需要额外保存更多寄存器，以保证异步抢占不会破坏被打断代码的现场。

更重要的是，这一课并不是为了让你学 MSP430 本身，而是为了让你从反面看清 Cortex-M 的独特性。既然 MSP430 的 ISR 必须是特殊函数，那 Cortex-M 为什么能让 `SysTick_Handler()` 看起来像普通 C 函数？这个问题在 lesson 17 被真正提出，在 lesson 18 才会被彻底解释。也正因为有了这一课的对照，下一课关于 exception frame、AAPCS 和 `BX LR` 特殊返回语义的讨论，才会显得格外有说服力。
