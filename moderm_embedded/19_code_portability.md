# 19 从 IAR 切到 GNU-ARM：可移植性到底意味着什么

这一课，Miro Samek 做了一件很多观众一直在催的事情：把前面一直使用的 IAR 工具链，切换到免费的 GNU-ARM 编译器和 Eclipse 系的 Code Composer Studio（CCS）。

如果只从表面看，这节课像是在讲“怎么安装新 IDE、怎么新建工程”。但 Samek 真正想讲的，其实不是 IDE 使用教程，而是借这次迁移机会，把一个很容易被空谈的话题落到代码上：code portability 到底是什么意思。

前面十几课里你已经写了不少 Cortex-M 裸机代码，也理解了启动文件、链接脚本、向量表、CMSIS 和中断机制。第 19 课要做的，就是拿这些已经学过的东西做一次压力测试：

1. 哪些代码是真正可移植的？
2. 哪些代码只是“碰巧在 IAR 下能跑”？
3. 如果换了编译器和 IDE，系统会在哪些层面首先出问题？

## 为什么要切工具链：不是换编辑器，而是检验工程质量

Samek 一开始就把这节课的定位说得很明确：迁移工具链不是为了追求新鲜，而是一次很好的代码审计机会。

因为在嵌入式项目里，“可移植性”如果不经过真实迁移，往往只是口头自信。很多代码在原工具链里能工作，只是因为：

1. 编译器帮你容忍了某些扩展写法。
2. IDE 默认替你准备了某些头文件或库。
3. 启动代码和链接配置刚好与当前环境匹配。

一旦这些隐含条件改变，问题就会暴露出来。

所以第 19 课的真正主题是：工具链迁移是区分“标准化代码”和“环境绑定代码”的最好办法。

## 先换开发环境：CCS 与 GNU-ARM 的最小工作集

课程首先介绍了为什么选择 CCS：

1. 它支持 TivaC LaunchPad 上的 Stellaris-ICDI 调试接口。
2. 它可以集成 GNU GCC ARM 工具链。
3. 对这门课程所用板子来说，这是免费且现实可行的选择。

Samek 这里还顺带强调了一个很工程化的习惯：安装路径和工作目录尽量避免空格与特殊字符。这个建议看起来琐碎，但在跨工具链、脚本、Make、调试器配置混用时，非常实用。

另外，课程也引入了 Eclipse/CCS 的 `workspace` 概念，并建议为这门课程单独建一个 workspace，而不是把所有项目堆到一个默认工作区里。这其实和前面每课单独建目录的思路是一致的：

1. 保持工程边界清晰。
2. 让迁移和复制更可控。
3. 避免工具状态污染业务目录。

## 新建 GNU 工程后，第一眼看上去“能编译”，但并不等于可用

课程里先用 CCS 新建了一个基于 GNU 的 TM4C123 工程。它确实可以立刻 build 成功，IDE 也会帮你生成：

1. 一个空的 `main.c`
2. 一个厂商风格的 startup 文件
3. 一个 `.lds` GNU linker script

但 Samek 很快指出：这个“能编译”的起点，并不代表它已经是高质量、可维护、可移植的工程。

因为厂商模板代码往往有前面几课已经批评过的老问题，例如：

1. 启动代码使用 proprietary 的异常命名，而不是 CMSIS 标准名字。
2. 向量表维护方式不够优雅，使用中断时要手工改表、改声明。
3. 默认异常处理函数通常是死循环，等价于系统 freeze。
4. 链接脚本采用很常见但不一定最安全的 RAM/stack 布局。

也就是说，迁移并不是“把旧工程一导入就完事”，而是要重新审视整个工程底座。

## 第一层可移植性：应用逻辑其实大多没问题

课程接下来做了一件很能说明问题的事：把前一课中那些“真正有价值”的代码拷进 lesson 19 工程里，例如：

1. `main.c`
2. `bsp.c`
3. `bsp.h`
4. 改造过的 startup 文件
5. MCU 主头文件

这一步的结果非常有启发性：

应用层 C 代码本身，很多其实可以直接复用。

为什么？因为这些代码大多建立在：

1. 标准 C 语法
2. CMSIS 风格寄存器访问
3. 合理分层的 BSP 接口

之上。换句话说，真正写得“干净”的应用逻辑，并不会因为编译器从 IAR 变成 GNU 就立刻报废。

这就是第一层可移植性：

代码逻辑层，如果建立在标准和清晰抽象之上，迁移成本通常远低于想象。

## 真正先出问题的，是环境耦合层

把旧代码拷进新工程后，build 很快就报错了。而这些错误恰恰揭示了迁移里最脆弱的区域。

### 1. CMSIS 头文件不再由 IDE 隐式提供

在 IAR 环境下，像 `core_cm4.h` 这类 CMSIS 核心头文件对开发者来说常常像“系统自带的一部分”。但在 CCS/GNU 里，它并不会自动出现。

于是 lesson 19 第一个明显的问题就是：

1. 编译器找不到 `core_cm4.h`
2. 你必须显式把 CMSIS 放进工程可见路径中

这一步看起来只是“补 include path”，但 Samek 用它强调了一件非常实际的事：

不要把 IDE 提供的隐式环境误当成代码本身的一部分。

真正可移植的工程，应该把这类依赖显式化。

## include path 也分“可移植”和“不可移植”两种

课程这里用了一个特别好的例子来说明：就连 include path，也有写得好和写得差之分。

### 绝对路径虽然简单，但只在你机器上有效

如果你在工程设置里直接把 `C:\embedded_programming\CMSIS\Include` 这种路径塞进去，它当然能编译，但这条路径只有在你的机器、你的目录结构上才成立。

这意味着：

1. 代码交给别人就可能失效。
2. 换一台电脑就得重配。
3. 版本库里的工程配置实际上不可复用。

### 相对路径才更接近真正的工程可移植性

Samek 推荐的做法是借助 Eclipse 的 `PROJECT_LOC` 等变量，构造相对 include path，例如：

1. 当前工程目录
2. 往上一级
3. 再进入 `CMSIS/Include`

这样路径就不依赖某台机器的盘符和绝对目录结构。

这一步虽然只是工程配置细节，但它很好地说明：

可移植性不仅体现在 C 源码里，也体现在工程元数据里。

## 第二层可移植性：CMSIS 为什么能帮你保住大部分代码

在解决 include path 之后，课程非常自然地带出了 CMSIS 的价值。

`TM4C123GH6PM.h` 这样的设备头文件通过：

1. `#define __FPU_PRESENT 1`
2. `#include "core_cm4.h"`

把 Cortex-M4 的核心寄存器、SysTick、SCB、NVIC 等抽象统一到标准接口里。

这意味着，像下面这样的代码：

```c
SysTick->LOAD = SYS_CLOCK_HZ/2U - 1U;
SysTick->VAL  = 0U;
SysTick->CTRL = (1U << 2U) | (1U << 1U) | 1U;
```

以及：

```c
SCB->CPACR |= ((3UL << 20U) | (3UL << 22U));
```

本质上并不依赖 IAR 或 GNU，它们依赖的是 CMSIS 这套跨编译器、跨工具链的接口约定。

这也正是 Samek 想证明的：

只要你的代码建立在标准化抽象上，工具链迁移时真正需要动的大多不是业务逻辑，而是启动和构建边界层。

## 第三层可移植性：启动文件最容易暴露编译器依赖

在 include 问题解决后，下一批错误几乎都来自 startup code。课程明确指出，这一点并不意外，因为启动文件恰恰是最容易混入编译器私有扩展的地方。

在 IAR 里，启动代码常会依赖：

1. IAR 专用关键字
2. IAR 风格段名与符号
3. IAR 运行库入口

但在 GNU 环境里，对应机制往往变成另一套写法。

## GNU 启动代码的典型风格

lesson 19 的 GNU 启动文件里，可以很清楚地看到这种差异。

例如函数和向量表往往通过 GCC attribute 表达：

```c
__attribute__ ((section(".isr_vector")))
```

```c
void SysTick_Handler(void) __attribute__ ((weak, alias("Default_Handler")));
```

这种写法和 IAR 并不是一个语法体系，但表达的工程意图是一致的：

1. 把向量表放进指定 section
2. 给中断处理函数提供弱默认实现
3. 允许应用层同名函数覆盖默认 handler

所以 lesson 19 并不是在推翻前面学过的启动思想，而是在告诉你：

思想层是稳定的，语法层会随工具链变化。

## Samek 借迁移顺手“修正”了厂商模板的问题

这一课并没有机械沿用 CCS 自动生成的 startup 文件，而是继续使用自己前几课建立的那套改良思路。例如：

1. 使用 CMSIS 兼容的异常名，如 `SysTick_Handler`
2. 使用 weak alias 组织默认 handler
3. 不让 fault handler 直接陷入无穷循环，而是进入 `assert_failed()` 后重置系统

这点非常重要，因为它说明：

迁移不是被动适配新工具，而是借机把工程底层做得更一致、更可控。

## 链接脚本才是 GNU 世界里的“内存真相文件”

lesson 19 里另一个特别关键的文件是 `lesson.ld`。这份 linker script 清楚定义了：

1. `ROM` 和 `RAM` 的地址范围
2. `.isr_vector`、`.text`、`.data`、`.bss` 的放置位置
3. stack 和 heap 的边界符号

比如：

```ld
MEMORY {
    ROM (rx)  : ORIGIN = 0x00000000, LENGTH = 256K
    RAM (xrw) : ORIGIN = 0x20000000, LENGTH = 32K
}
```

以及：

```ld
.stack : {
    __stack_start__ = .;
    . = . + STACK_SIZE;
    . = ALIGN(4);
    __stack_end__ = .;
} >RAM
```

课程在这里再次强调：链接脚本不是“随便放着不动”的配角，而是系统运行布局的一部分。

## Samek 为什么批评默认 stack 放置方式

这一课里有一个很典型的 Samek 式观点：他指出很多“走老路”的 linker script 把 stack 作为 RAM 里最后一个 section，这看似自然，但其实在 ARM 这类栈向低地址增长的体系上未必安全。

他的担忧是：

1. 如果 stack 向下溢出，
2. 它可能直接破坏邻近 RAM 内容，
3. 而且这种破坏常常很隐蔽。

也就是说，lesson 19 虽然主题是工具链迁移，但 Samek 仍然在借这个机会持续强调一条主线：

底层布局不是模板问题，而是系统安全性问题。

## GNU 工程配置本身也在暴露“工具链语义”

搜索工程配置文件时，你还能看到 GNU/CCS 工程里显式配置了很多关键选项，例如：

1. `-mcpu=cortex-m4`
2. `-mfloat-abi=hard`
3. `-mfpu=fpv4-sp-d16`
4. `-ffunction-sections`
5. `-fdata-sections`
6. linker 使用 `--gc-sections`
7. `-nostdlib`

这些参数说明 GNU 工具链下很多默认行为都需要开发者自己更明确地掌控。好处是透明，可控；代价是你必须真正理解这些选项含义。

这与 IAR 那种“很多东西默认藏在 IDE 背后”的体验形成鲜明对比。

## 这一课还顺带触及了一个现实世界问题：FPU 和运行库兼容性

lesson 19 工程 README 里还特别记了一条后续更新：某些版本的 CCS/GNU 工具链里，`__libc_init_array()` 调用会因为 VFP register arguments 与运行库 ABI 不匹配而触发链接错误。

这件事本身不一定是课程主线，但它很好地说明：

1. 工具链迁移常不只是语法问题。
2. ABI、运行库、FPU 配置也会成为真实阻碍。
3. “能编译”背后涉及一整套 binary compatibility 条件。

这再次证明 Samek 的核心观点：可移植性是系统工程问题，不只是源码层问题。

## lesson 19 真正验证了什么叫“可移植”

把整节课压缩成一句话，其实就是：

真正可移植的，不是“某份工程文件”，而是你构建工程时所依赖的抽象层次。

更具体地说：

1. 如果代码依赖标准 C 和 CMSIS，这部分通常更可移植。
2. 如果代码依赖编译器专有关键字、专有启动文件、专有路径配置，就很容易在迁移时出问题。
3. 启动文件、链接脚本、工程属性，往往是迁移阻力最大的区域。

所以第 19 课不仅是在讲 GNU-ARM，也是在给前面所有课程内容做一次“压力回归测试”。

## 这节课和前后课程的关系

把 lesson 19 放回整体课程链路里看，它的位置其实非常关键：

1. 前面 lesson 13-15 讲了 startup、linker、vector table。
2. lesson 16-18 讲了 interrupt 和 Cortex-M 异常语义。
3. lesson 19 用工具链迁移把这些底层知识重新串起来，验证你是否真的掌握。

也正因为如此，lesson 19 不是一个“偏工具”的插曲，而是一次横向整合。

## 总结

19 课的核心，是借从 IAR 切换到 GNU-ARM/CCS 的过程，真正检验代码和工程配置的可移植性。

这一课最重要的结论可以概括为：

1. 工具链迁移是检验 code portability 最真实的方法之一。
2. 建立在标准 C 和 CMSIS 之上的应用层代码，通常比想象中更容易迁移。
3. 真正容易出问题的，往往是 startup code、linker script、工程路径和编译器专有扩展。
4. 绝对路径、IDE 隐式依赖和厂商模板代码，都会降低工程的可移植性。
5. GNU 环境下的 `__attribute__`、`.lds` 脚本和显式编译选项，让很多原本被 IDE 隐藏的细节暴露出来，也迫使你更彻底理解系统底座。
6. 可移植性不是一句“代码遵循标准”就能证明的口号，而是源码、构建配置、运行库和 ABI 共同作用的结果。
7. lesson 19 之所以重要，不只是因为它换了一个编译器，而是因为它把前面学过的 startup、CMSIS、linker 和 interrupt 知识，放进了一次真实迁移中重新验证。

如果说前面的课程是在教你“如何把一个嵌入式系统写对”，那么第 19 课进一步问的是：“当环境变化时，它还能不能继续被正确地构建、运行和维护？” 这正是工程实践里真正的 portability 问题。
