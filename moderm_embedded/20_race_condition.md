# 20 竞态条件：为什么“每段代码都对”系统却仍然会错

这一课，Miro Samek 终于系统展开了 race condition。前面几课已经把几个关键前提都铺好了：

1. 中断会异步打断当前执行流。
2. 主循环和 ISR 可能交替访问同一资源。
3. 一个看似简单的寄存器更新，在机器层面往往不是一步，而是一串 read-modify-write。

到了第 20 课，这些前提第一次汇合成一个真正危险的问题：两段代码单独看都没错，但只要它们能彼此抢占，并且又访问同一个共享资源，系统行为就可能依赖“谁刚好先执行”，从而出现极难复现的 bug。

这节课的重点，不只是定义什么叫 race condition，而是通过一个极小的 LED 例子，把竞态条件的形成过程、危险性，以及两种典型消除思路完整展示出来。

## 从一个看起来完全正常的程序开始

课程先延续上一课的 GNU/CCS 工程环境，在 `main.c` 和 `bsp.c` 里放了一个非常简单的程序：

1. 主循环负责闪烁绿灯。
2. `SysTick_Handler()` 每半秒触发一次，用来翻转蓝灯。

从表面上看，这个程序非常无害：

1. 主循环只改绿灯。
2. 中断只改蓝灯。
3. 绿灯和蓝灯互不相干。

所以直觉上，你很容易认为这两个执行上下文可以和平共处。

课程一开始故意利用这种“看起来没问题”的直觉，然后一步一步把问题挖出来。

## 问题的切入点：不用 `DATA_Bits`，改用整个 `DATA` 寄存器

这一课里，Samek 故意把前面 lesson 7 里介绍过的安全写法先注释掉，不再通过 `GPIOF_AHB->DATA_Bits[LED_GREEN]` 来点亮或熄灭绿灯，而改成对整个 `GPIOF_AHB->DATA` 寄存器做 read-modify-write。

也就是把“设置一个位”写成：

1. 先读取 `DATA` 寄存器。
2. 用按位 OR/AND 改掉目标位。
3. 再把结果写回 `DATA`。

这段代码在 C 里可以写得很短，例如：

```c
GPIOF_AHB->DATA |= LED_GREEN;
GPIOF_AHB->DATA &= ~LED_GREEN;
```

但课程特意强调：即使你写成 `|=` 或 `&=` 这种简洁形式，底层做的事情仍然没有变化，依然是 read-modify-write 序列，而不是单条“原子设置位”的神奇指令。

## 为什么这段代码单独看完全正确

如果你只看主循环本身，它当然是对的：

1. 读出当前 `DATA`。
2. 只改绿灯对应那一位。
3. 其他位保持原值不动。

这正是寄存器编程里最常见、也最自然的写法。

如果你只看 `SysTick_Handler()`，它也完全合理：

```c
void SysTick_Handler(void) {
    GPIOF_AHB->DATA_Bits[LED_BLUE] ^= LED_BLUE;
}
```

这段中断代码只翻转蓝灯，而且利用的是 `DATA_Bits` 别名寄存器，看起来也没有问题。

所以到这里为止，你看到的是两段分别都正确的代码。课程接下来要证明的是：并发系统里，“分别正确”远远不够。

## 程序现象：大部分时间都正常，偶尔蓝灯节奏不对

把程序下载到板子后，你会发现它大多数时候确实工作正常：

1. 绿灯在主循环里闪烁。
2. 蓝灯由 SysTick 中断定期翻转。

但如果你多观察一会儿，会注意到一个细节：蓝灯闪烁并不总是那么均匀，有时会突然停顿更久，看起来像节奏被打乱了。

这正是竞态条件最典型、也最讨厌的外观：

1. 不是每次都错。
2. 不是一上电就错。
3. 甚至单步看局部代码时完全看不出问题。

也就是说，它不是逻辑显而易见的 bug，而是时序碰撞下才出现的 bug。

## 用调试器把 read-modify-write 拆开来看

课程接下来做的事情非常关键：不是停留在源代码层面猜，而是进入反汇编和寄存器视图，把“关绿灯”这条 C 语句拆成真实机器步骤。

在汇编层面，你会看到类似这样的序列：

1. 先把 `GPIOF DATA` 寄存器地址装入寄存器。
2. 再把当前 `DATA` 寄存器值读入某个通用寄存器。
3. 用 `BIC` 等指令清掉绿灯位。
4. 最后把修改后的结果写回 `DATA` 寄存器。

Samek 特别指出：一条看似短小的 C 语句，实际上会展开成多条机器指令。也正因为它不是一步完成，所以中间就存在被中断打断的窗口。

## 人工制造最坏时刻：在“读完但还没写回”时触发中断

为了把问题清清楚楚地暴露出来，课程没有靠“等运气碰到”，而是主动在最敏感的位置手动触发 SysTick。

具体做法是：

1. 单步执行到主循环中那条即将清绿灯位的汇编序列。
2. 停在 `BIC` 之前，也就是“已经读到旧值，但还没写回新值”的时刻。
3. 手动在 NVIC 控制寄存器里置位 `PENDSTSET`，让 SysTick 变成 pending。
4. 设置两个断点：一个在 `SysTick_Handler()`，一个在主循环写回之后的位置。
5. 然后点 `Resume`，看谁先执行。

这里 Samek 还提醒：不能用单步继续，因为单步会影响中断行为；必须让程序自然恢复运行，这样才是真实的抢占时序。

## 真正的问题是：中断改了共享寄存器，但主循环不知道

恢复运行后，程序先进入 `SysTick_Handler()`。这说明中断确实抢在主循环写回之前执行了。

中断里发生了什么？

1. 蓝灯位被翻转。
2. `GPIOF_AHB->DATA_Bits[LED_BLUE]` 对应的硬件状态被更新。

然后 ISR 返回，主循环继续执行之前那条 read-modify-write 序列。但主循环手里拿的，仍然是“中断发生前”读出来的旧 `DATA` 值。

于是接下来：

1. 主循环把旧值清掉绿灯位。
2. 再把这个结果整个写回 `DATA` 寄存器。
3. 结果不仅绿灯被关掉，蓝灯刚刚被中断改出的新状态也一起被覆盖掉了。

课程在视频里用一句非常直接的话概括这个时刻：`Ooops!`

因为这最后一次存储，居然把绿灯和蓝灯都一起熄灭了。

## 这就是 race condition：结果取决于不可控的执行顺序

到这里，课程正式给出定义：

race condition 发生在两个或更多可以彼此抢占的执行片段访问共享资源时，而最终结果取决于它们实际发生的执行顺序。

这个定义里最关键的是“结果依赖顺序”，因为：

1. 代码本身未必写错。
2. 单独执行每段代码也可能完全正确。
3. 错误只出现在某些特定交叉时序下。

这就是为什么竞态条件特别违背直觉。它不是“逻辑写反了”，而是“时序恰好碰撞了”。

## 为什么竞态条件特别危险

Samek 在这里用了一个很强烈但很有效的比喻：如果中断代码不是切换蓝灯，而是打开核反应堆冷却系统，而主循环随后又用旧值把这个位写回成关闭状态，那么后果就不是“LED 看起来怪怪的”，而可能是灾难性的系统失效。

这个例子要强调的不是危言耸听，而是：

1. 竞态条件的破坏力不取决于它看起来多小。
2. 它常常发生在最底层、最不起眼的共享位操作里。
3. 一旦共享资源对应的是安全关键设备，后果可能非常严重。

## 更糟的是：这种 bug 往往是间歇性的

课程还强调了竞态 bug 最难处理的地方：它们通常只有在非常窄的时窗里才会出现。

这带来几个现实问题：

1. 你可能测试几小时、几天都看不见它。
2. 它却可能在现场运行一段时间后突然出现。
3. 加日志、单步或开断点都可能改变时序，反而让 bug 暂时消失。

这也是为什么竞态常被认为是并发系统里最糟糕的一类错误之一。

## 第一种解决思路：mutual exclusion

在把你充分“吓到”之后，课程开始讲第一种消除竞态的方法：mutual exclusion，也就是互斥。

核心思想很简单：

1. 当一段代码在访问共享资源时，禁止另一段并发代码插进来。
2. 这样共享资源的访问就被串行化。
3. 原本可被打断的复合操作，变成一个不可分割的原子片段。

在这节课的例子里，实现方式也很直接：在主循环访问 `GPIOF_AHB->DATA` 的那几行代码前后关中断和开中断。

也就是使用 CMSIS 提供的 intrinsic：

```c
__disable_irq();
GPIOF_AHB->DATA |= LED_GREEN;
__enable_irq();

__disable_irq();
GPIOF_AHB->DATA &= ~LED_GREEN;
__enable_irq();
```

## `__disable_irq()` 和 `__enable_irq()` 为什么适合做临界区

课程还专门说明，这两个函数并不是普通函数调用，它们会被编译成极短的机器指令：

1. `__disable_irq()` 对应 `CPSID i`
2. `__enable_irq()` 对应 `CPSIE i`

两者都只需一个时钟周期左右，因此从指令开销上看，非常适合围绕短小关键区使用。

这也引出了临界区（critical section）的定义：

处于关中断与开中断之间的那小段代码，就是 critical section。

## 临界区到底消除了什么

重新做同样的实验：

1. 先在主循环临界区内部停住。
2. 手动把 SysTick 置为 pending。
3. 设置断点继续运行。

这一次，中断不会在临界区中间打断主循环。主循环会先把 read-modify-write 整个序列完成；只有等 `__enable_irq()` 之后，挂起的 SysTick 才会真正进入执行。

也就是说，原来那三段可能交叉的代码：

1. 主循环的“开绿灯”关键区。
2. 主循环的“关绿灯”关键区。
3. `SysTick_Handler()`

现在都被串行化了。它们仍然可能先后执行，但不再可能相互插在彼此中间。这就是 mutual exclusion 的真正含义。

## 但课程马上指出：互斥不是最优答案

Samek 并没有停留在“关中断就好”。他接着提出一个更高层的观点：

比起用 mutual exclusion 修补共享资源问题，更好的办法，是尽量一开始就不要共享资源。

这也是整套课程方法论的一条主线：

1. 如果能避免共享，就优先避免共享。
2. 如果不能避免，再考虑互斥。

这个观点在后面的 active object 课程里会被进一步发展成“不共享、只发消息”的设计原则。

## 第二种解决思路：避免共享资源本身

为了说明这一点，课程把示例代码切回原始写法，也就是不用整个 `DATA` 寄存器，而是重新用 `DATA_Bits` 别名寄存器：

```c
GPIOF_AHB->DATA_Bits[LED_GREEN] = LED_GREEN;
GPIOF_AHB->DATA_Bits[LED_GREEN] = 0U;
```

而 `SysTick_Handler()` 仍然使用：

```c
GPIOF_AHB->DATA_Bits[LED_BLUE] ^= LED_BLUE;
```

这时候，主循环和中断虽然仍然都在“控制 GPIOF”，但它们不再共享同一个通用 `DATA` 寄存器访问路径，而是在用不同的别名寄存器操作不同的位。

## `DATA_Bits` 为什么从根上减少竞态

课程借这个例子，把 lesson 7 里讲过的硬件设计意义重新点明：`DATA_Bits` 数组其实为不同位组合提供了不同地址别名。

结果就是：

1. 绿灯位可以通过专用别名寄存器直接写。
2. 蓝灯位也有自己的专用别名寄存器。
3. 每次置位或清位都可以变成一次独立写操作。
4. 不需要先读整个共享 `DATA` 再改再写回。

于是 read-modify-write 被消掉了，共享窗口也就跟着消掉了。

这也是课程专门解释“为什么 TI 硬件工程师把 GPIO 设计得这么复杂”的原因：他们实际上是在硬件层帮软件设计者减轻并发问题。

## 这节课的真正主题，其实不只是“竞态条件”

如果把 lesson 20 放回整套课程里看，它真正传达的是一个更普遍的设计原则：

1. 共享资源 + 异步抢占 = 潜在竞态。
2. 互斥可以修补竞态，但会引入额外时序成本。
3. 最好的办法往往是重新设计访问方式，让共享本身消失。

这和后面事件驱动、active object、shared-nothing 的思想，其实是一条连续的线。

## 总结

20 课的核心，是通过一个极小但非常典型的 GPIO 例子，把 race condition 的形成和消除过程彻底讲清楚。

这一课最重要的结论可以概括为：

1. 一个看似简单的寄存器置位/清位，在底层往往是 read-modify-write 序列。
2. 如果这段序列在中间被中断打断，而 ISR 又访问同一共享资源，就可能出现 race condition。
3. 竞态条件最危险的地方在于：各段代码单独看都正确，错误只发生在特定交叉时序下，因此通常间歇、难复现、难定位。
4. 解决竞态的一种方法是 mutual exclusion，也就是用临界区把共享资源访问串行化。
5. 在 Cortex-M 上，`__disable_irq()` / `__enable_irq()` 可以非常低成本地形成短临界区。
6. 但比互斥更好的方法，是一开始就避免共享资源本身。
7. `GPIOF_AHB->DATA_Bits[]` 这类硬件别名寄存器设计，正是通过把位访问分离开来，减少甚至消除竞态窗口。

如果说第 16 到 18 课是在解释“中断是怎么工作的”，第 20 课则开始解释“有了中断之后，软件结构为什么会变难，以及该如何系统地处理这种难度”。这也是后面进入优先级、中断屏蔽、RTOS 同步，以及最终事件驱动并发设计之前，必须先建立起来的风险意识。
