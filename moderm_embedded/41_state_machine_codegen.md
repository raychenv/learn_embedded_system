# 41 状态机代码自动生成

这一课，Miro Samek 继续讲 state machines，但重点已经从“如何手工实现”转向“如何自动生成代码”。在 Samek 看来，graphical modeling 和 automatic code generation 虽然还不算主流，但它们确实是 modern embedded software development 的重要组成部分，而且很可能会越来越重要。

这节课的任务并不只是做一次工具演示，而是同时回答几个层次的问题：

1. 用 QM 这样的工具，状态机代码到底是怎么生成出来的。
2. 图形建模相比纯手写代码，真正带来的价值是什么。
3. 自动生成代码在工程实践中真正的难点到底是什么。

Samek 特别想澄清的一点是：很多人对 code generation 的第一反应，往往是担心“生成代码会不会低效、不可靠、不够灵活”。但在他看来，这些通常并不是最真实的问题。真正的问题，更多出在工具摩擦、物理设计支持、生成代码与现有工程如何融合这些实践层面。

## 从上一课的 TimeBomb 模型出发：这次 QM 不再只是“画图工具”

前面几课里，QM 已经出现过，但更多只是作为画状态图的工具使用。当时图怎么画并不重要，甚至用白板、纸笔都可以替代。

这一课则完全不同。QM 不再只是用来“展示状态图”，而是成为实际生成代码的核心工具。

因此，课程从 lesson 40 的 `TimeBomb` 开始，但不是先打开 Keil 工程，而是先打开 `TimeBomb.qm` 模型文件。这本身就体现了工作流的变化：

1. 之前是“先写代码，再用图辅助理解”。
2. 现在是“先维护模型，再由模型生成代码”。

这意味着模型已经不再只是文档，而开始成为真正的源头资产。

## 第一步：把模型从“图示”补全为“可生成代码的逻辑设计”

在之前的状态图里，很多 action 只是写成 pseudocode，用来减少图上的视觉杂乱。那样做对讲解和设计讨论已经足够，但对于 code generation 来说还不够。

### 状态机动作必须有真正的 C 代码

这节课中，Samek 的工作方式很务实：

1. 打开上一课已经手工写好的 `TimeBomb` 代码。
2. 反过来把那些已经验证可用的动作代码，复制回 QM 模型里。
3. 把所有 pseudocode 补成真实的 C 语句。

也就是说，这一课不是从空白模型凭空生成代码，而是通过“从手工代码回填模型”的方式，让你看清楚：模型里到底必须补到什么程度，生成出来的代码才会完整、可编译、可运行。

这种逆向补模的方式非常适合教学，因为你已经知道最终代码应该长什么样，所以可以更清楚地看出模型中哪些信息是必须的。

### 类的属性和操作也必须在模型中显式定义

除了状态机动作，模型还必须包含 `TimeBomb` 类本身的逻辑结构。因为在动作代码中，很多内容都是通过 `me` 指针访问的，比如：

1. `te`，也就是 `QTimeEvt`
2. `blink_ctr`

这些如果不在模型里声明，生成器当然不知道该生成哪些成员。

同样，如果类需要某些 operation，例如 constructor，也要在模型中定义出来，包括：

1. 返回类型。
2. 参数。
3. 实际操作代码。

QM 在这方面的一个优点是，它理解前面几课中 OOP in C 的命名约定，因此会自动补上类名前缀，也会自动处理 `me` 指针这样的成员函数约定。

## 这叫 logical design：模型描述类、属性、操作和状态机本身

Samek 用一个很明确的术语来描述这一层建模：logical design。

也就是说，模型在这一层面要完整表达：

1. 有哪些类。
2. 每个类有哪些属性。
3. 有哪些操作。
4. 状态机及其状态、转移、entry/exit action。

这部分关注的是系统“逻辑上是什么”，而不是“代码具体放到哪些文件、目录、包里”。

## 物理设计同样重要，而 QM 的一个独特优势就是它支持 physical design

接下来 Samek 把话题引向一个非常有工程味的问题：即使 logical design 完整了，生成代码最终落到哪里去，代码文件如何组织，这仍然是另一个完全不同的问题。

这就是 physical design。

Samek 之前其实已经在多个地方强调过 physical design 的重要性，而这一课进一步表明：很多 code generation 工具恰恰在这里做得很差。

### 多数工具只会把代码丢到固定结构里

大多数建模工具的做法是：

1. 给每个类固定生成一个 `.h`
2. 再固定生成一个 `.c`
3. 目录结构和文件布局高度刚性

这种方式对小 demo 可能没问题，但在真实工程里往往会很别扭，因为它会强行把你的工程结构重塑成工具喜欢的样子。

### QM 的不同点：目录和文件本身也是模型的一部分

QM 的一个明显特色是，它把目录和文件本身也作为模型中的一等元素。

这意味着：

1. 你可以在模型里显式指定生成目录。
2. 可以指定文件名。
3. 可以控制哪些模型元素生成到哪个文件里。
4. 可以让生成代码与手写代码混放在同一个文件中。

Samek 认为这点非常关键，因为真正好用的 code generation，不应该逼你迁就工具的文件组织，而应该允许你保留自己的 physical design。

## 用 TimeBomb 演示：只生成 `main.c` 的一部分，而不是整个工程全盘接管

为了说明这种灵活性，这节课的 physical design 很刻意地选择了一个非标准案例：

1. 不是给 `TimeBomb` 专门生成独立的 `.h/.c`。
2. 而是继续沿用现有工程结构。
3. 只在已有的 `main.c` 里生成 `TimeBomb` 类及其状态机相关部分。
4. `bsp.h`、`bsp.c` 等其他文件继续保持手写。

这种做法的意义很大，因为它说明：建模和代码生成并不要求“一次性吞掉整个项目”。你完全可以在一个现有工程中，逐步把某些部分交给生成器，而让其他部分继续保持手工维护。

## 具体做法：在文件模板中嵌入 `$declare` 和 `$define`

QM 的生成方式非常直接。你可以先把现有 `main.c` 的内容整体拷进文件模型项，然后把其中某些片段替换成生成指令。

在这个例子里，主要替换两块：

1. 原来手写的 `TimeBomb` 类声明，用 `$declare TimeBomb` 替换。
2. 原来手写的 `TimeBomb` 类定义和状态机实现，用 `$define TimeBomb` 替换。

剩余部分仍然保留原样，作为普通手写代码文本存在于文件模板里。

这样生成后的 `main.c` 就会是：

1. 一部分原样复制。
2. 一部分由模型自动展开生成。

这就是 QM 所谓“生成指定模型元素”，而不是简单粗暴地“接管整个文件”。

## 生成出的代码为什么能和手写版几乎一样

Samek 在这一课里反复强调，生成出的 `TimeBomb` 代码和前一课手写版本几乎一致。其原因并不神秘：

1. 你的模型已经把状态机逻辑完整表达出来。
2. QP/C 框架已经提供了确定的状态机编码规则。
3. QM 只是按照这些稳定规则把模型翻译成 C 代码。

因此，生成代码不会凭空“发明”一种新的实现风格，它只是系统化地复现你前几课已经学会的那套 state-handler、entry/exit、superstate、QP/C HSM 编码模式。

这点非常重要，因为它说明：code generation 的前提，不是“工具比你聪明”，而是“框架先定义好了稳定的映射规则”。

## 这也解释了：框架才是代码生成的真正支撑基础

Samek 明确指出，code generation 能成立，并不是因为建模工具本身有魔法，而是因为底层框架提供了可靠的 codification rules。

在这个例子里，这套规则来自 QP/C：

1. 状态处理函数怎么写。
2. `Q_TRAN`、`Q_SUPER` 如何表达转移和层次关系。
3. `Q_ENTRY_SIG`、`Q_EXIT_SIG`、`Q_INIT_SIG` 如何处理。
4. active object 和 hierarchical state machine 如何运行。

只要这些规则稳定，QM 就可以把模型自动映射成符合框架约定的代码。

换句话说：

1. framework 负责定义“状态机代码长什么样”。
2. modeling tool 负责按规则生成它。

## 为什么要建模：Samek 认为最大的理由不是“省代码”，而是激活视觉脑力

课程后半部分开始回答更根本的问题：既然手写也能实现，为什么还要建模和生成代码？

Samek 引用了 Bran Selic 关于 model-driven development 的观点，并给出一组更偏工程哲学的论据。

### 1. 其他成熟工程领域都离不开模型

无论造房子、造汽车，还是设计电路，都不可能跳过模型和图纸直接开工。软件工程之所以很多时候反其道而行，更多是历史习惯，而不是因为模型没价值。

### 2. 人脑对视觉信息处理能力远强于文本

Samek 强调，人的大脑在视觉处理上的“硬件投入”远超其他感觉通道。因此：

1. 我们处理图形信息更快。
2. 对复杂结构的整体把握更强。
3. 对图形的记忆保留也更好。

这就是为什么“a picture is worth a thousand words”在软件里依然成立。甚至可以说，一张好的状态图往往胜过几百上千行零散代码。

### 3. 但图必须既直观又精确

Samek 也提醒，不是任何图都能提供价值。很多软件项目中的“方框图”之所以作用有限，是因为它们并不精确：

1. 箭头到底表示继承、通信还是依赖？
2. 阴影、颜色、字体又分别表示什么？
3. 多种图形语义混杂在一起，反而制造歧义。

因此，像 UML state machine 这种“直观但语义明确”的图才真正适合做 constructive modeling，也就是可以直接支撑生成代码的模型。

## 状态图的另一个巨大价值：让非程序员也能参与行为设计

Samek 在这里引用了 David Harel 的一个故事：一位飞行员并不懂实现细节，但仅凭 statechart 图就看出了系统行为图中的一处错误。

这个例子说明，好的行为模型不只是给程序员自己看的，它还能把：

1. 领域专家
2. 测试人员
3. 系统工程师
4. 其他非实现人员

一起拉进设计评审过程。

而如果行为只存在于代码中，这种跨角色协作会变得困难得多。

## 只建模不生成代码，最终通常会失败

这节课中 Samek 对 code generation 的态度非常明确：建模如果不能进入生产代码，最终往往会失败。

原因很现实：

1. 不能生成代码的模型只是额外文档。
2. 一旦实现继续演进，模型很快会落后于代码。
3. 落后的模型会失去可信度。
4. 最终团队就会放弃维护它。

Samek 用了一个很形象的比喻：没有 code generation 的模型，就像没有轮子的汽车，跑不了多远。

所以在他看来，建模要真正成功，必须让模型成为生产代码链条的一部分，而不是停留在 PPT 或设计文档层面。

## 开发者最常担心的“代码效率和正确性”，反而通常不是真问题

Samek 接着回应了很多人对 code generation 的第一反应：

1. 生成代码会不会慢？
2. 会不会臃肿？
3. 会不会不正确？

他的观点是，这些担忧很像当年人们刚开始不信任 compiler 时的担忧。今天已经没有人会因为“手写汇编可能更快”就拒绝使用编译器。同样，成熟的 code generation 在正确性和效率上通常并不是瓶颈。

甚至很多时候，自动生成的代码反而是项目里最稳的一部分；真正更容易出错的，仍然是人工写的部分。

## 真正的问题是什么：不要让开发者“fight the tool”

在 Samek 看来，现实中 code generation 的最大问题并不是生成代码质量，而是工具摩擦，也就是所谓“fight the tool”。

所谓 fight the tool，指的是：

1. 原本简单的事，因为引入工具而变复杂了。
2. 工具强行规定很多不自然的表达方式。
3. 你为了配合工具，不得不写很多绕路代码。

Samek 用第 39 课提到的 SMC 作为反例。SMC 会对 action code 施加很多限制，例如：

1. 不允许你随意写简单赋值。
2. 不方便直接访问状态机上下文属性。
3. 逼得你只能通过额外包装 operation 来曲线表达动作。

这样一来，模型和代码都在为工具妥协，而不是工具在服务设计。

QM 在这方面的设计哲学刚好相反：

1. 你写的 action code 直接就是 C。
2. 工具不试图“聪明地改写你的代码”。
3. 你不需要为了生成器去重构整个行为逻辑。

Samek 认为，这是 QM 相比很多“高仪式感”建模工具的一大优势。

## 生成代码必须是完整的，而不是只吐出半成品 skeleton

Another practical point：Samek 非常反感那种只生成代码框架、再留一堆 `TODO` 让开发者去补的工具。

在他看来，真正可用的生成代码必须满足：

1. 生成完之后就能编译。
2. 不需要开发者再去手工补孔。
3. 生成文件不应该再被手改。

QM 的做法正是如此：

1. 生成文件默认视为只读产物。
2. 任何改动都应该回到模型中进行。
3. 再重新生成文件。

这建立了一条清晰的 single source of truth：模型才是源，生成文件只是产物。

## 易于与现有代码集成，是工具是否能落地的关键

Samek 还强调了另一条现实标准：自动生成的代码必须容易和 existing codebase 融合。

如果一个工具要求你：

1. 必须接受它固定的目录结构。
2. 必须把现有代码整体重写成它指定的模板。
3. 每加一个 include、宏、编译开关都要走复杂配置流程。

那么工具本身就会成为障碍。

QM 在这方面的优势是：

1. 文件体由你写。
2. 生成项只是嵌在文件中的 `$declare` / `$define` 指令。
3. 因此在生成段前后，你可以随意放 include、宏、手写辅助代码。
4. 甚至可以控制生成粒度，从单个 operation 到整个类、整个 package。

这种灵活性非常适合真实项目，而不只是 demo。

## 这节课真正传达的，不只是“会用 QM”，而是 model-driven 的判断标准

如果只把第 41 课理解成一节 QM 上手课，会低估它的重点。Samek 更想建立的是一套判断标准：什么样的 modeling/codegen 工作流才算真正可用。

从这节课来看，至少包括：

1. 模型必须足够精确，能生成生产代码。
2. 工具不能强迫你不断适应它。
3. 生成代码必须完整，而不是半成品。
4. 物理设计必须可控。
5. 模型和现有代码要能顺畅集成。
6. 底层必须有稳定框架来定义生成规则。

而 QM + QP/C 在 Samek 看来，正是较好满足这些条件的组合。

## 总结

41 课的核心，是把层次状态机从“可以手工编码”的工程技术，进一步推进到“可以自动生成”的现代开发流程中。

这一课最重要的结论可以概括为：

1. 在状态机建模中，QM 不只是画图工具，真正要让它发挥作用，必须把 pseudocode 补成完整 C 动作，并把类的属性、操作等 logical design 信息也完整建进模型。
2. modeling 只解决 logical design；而代码最终落在哪些目录和文件中，属于 equally important 的 physical design。
3. QM 的一个独特优势，是把目录和文件也视为模型元素，从而允许你保留自己的工程结构，只生成指定部分代码。
4. code generation 之所以能成立，不是因为工具“聪明”，而是因为 QP/C 这类底层框架先定义好了稳定的状态机编码规则。
5. 建模的真正价值不只是少写代码，更重要的是利用人类对视觉信息的强处理能力，并让更多角色参与行为设计评审。
6. 没有 code generation 的模型往往会迅速过时并被抛弃，因此建模若想长期存活，必须进入生产代码链条。
7. 自动生成代码的真正难点通常不在效率或正确性，而在于工具摩擦、完整性、与现有工程的集成，以及是否支持 physical design。

如果说第 40 课把层次状态机引入了专业框架，那么第 41 课则进一步说明：一旦框架和建模工具配合得当，状态机就不必长期停留在“图和代码手工同步”的阶段，而可以进入真正的 model-driven workflow。这也意味着，从这里开始，状态机不仅是行为建模方法，更开始成为工程自动化的一部分。
