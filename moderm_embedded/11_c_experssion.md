# 11 固定宽度整数、字节序与隐式转换：C 表达式为什么会悄悄背叛你

lesson 10 刚刚通过 stack overflow 和返回地址污染，展示了程序如何以非常戏剧化的方式失败。到了 lesson 11，Miro Samek 把主题切回了看似“基础”的数据类型，但你很快就会发现，这一课其实仍然在处理同一类问题：你以为 C 代码表达的是某个简单意图，可处理器和编译器最终执行的，却可能是另一套逻辑。

所以，这节课虽然标题里有 `stdint.h`、endianness 和 floating point，但真正的核心是可移植性和语义陷阱。尤其是当你跨不同位宽处理器工作时，表达式里的隐式类型提升和 signed/unsigned 转换，完全可能让“在 ARM 上看着没问题”的代码，到了 MSP430 上就悄悄给出另一个答案。

## 为什么内建整数类型对嵌入式并不够好

Samek 一开始就指出，C 的 built-in integer types 从语言设计上就是模糊的。比如：

1. `int` 多大，标准没写死。
2. `short <= int <= long` 是唯一相对关系，但具体位宽取决于平台。
3. `char` 通常是 1 byte，但它究竟默认有符号还是无符号，也可能由编译器或选项决定。

这套灵活性对早期 C 的可移植性是优点，但对嵌入式程序员来说却常常是麻烦。因为寄存器位宽、协议字段、存储布局、校验计算这些东西，很多时候都要求你明确知道变量到底是 8 位、16 位还是 32 位。

也正因为如此，课程先回顾了很多项目历史上自己手搓 typedef 的做法，再顺势引出 `stdint.h`。

## `stdint.h` 的价值：不是新语法，而是把“精确位宽”标准化

Samek 很明确地说，`stdint.h` 是 C99 对嵌入式程序员最有价值的贡献之一。因为它终于把这些最常用的固定宽度类型标准化了，例如：

1. `uint8_t`
2. `int8_t`
3. `uint16_t`
4. `int16_t`
5. `uint32_t`
6. `int32_t`

这件事的重要性不只是“少写几个 typedef”，而是：

1. 名字统一了。
2. 责任转交给编译器供应商。
3. 你的源代码不必再把“`unsigned int` 在本平台上刚好是 32 位”这种假设写死。

Samek 甚至强调，即使某个编译器不完全支持 C99，你也应该提供等价的 `stdint.h` 风格接口，而不是重新 invent 一套项目私有命名。因为在今天，继续发明自己的 `CPU_INT32U` 这一类名字，收益已经远小于成本。

## lesson 11 第一层实验：先验证这些类型在 ARM 上到底多大

课程里的代码先定义了一组变量：

```c
uint8_t  u8a, u8b;
uint16_t u16c, u16d;
uint32_t u32e, u32f;

int8_t  s8;
int16_t s16;
int32_t s32;
```

然后用 `sizeof()` 去验证这些类型在当前编译环境中的实际字节数。这个实验虽然简单，但目的非常明确：不要把“看起来像 16 位”“名字里写着 32”当成理所当然，而要在当前工具链中真正确认它们的定义。

这一步也再次体现 Samek 的风格：任何语言层面的假设，只要可能影响底层行为，就值得用实验验证。

## 编译器会重排全局变量：定义顺序不等于最终内存顺序

Samek 接着在 watch/memory 窗口里观察这些变量的地址，顺手指出一个很容易被忽视的事实：变量在内存中的实际排列顺序，并不一定和你在源码里书写的顺序一致。

这件事很重要，因为很多初学者会下意识假设：

1. 我先写 `u8a`，它就应该在最前面。
2. 再写 `u16c`，它就排在后面。

但编译器完全可能为了对齐或布局优化，调整这些对象的放置顺序。所以 lesson 11 先把这个误解提前打掉，避免你后面把“源码顺序”和“物理内存顺序”混为一谈。

## LDRB / LDRH / LDR：CPU 并不是一视同仁地读写所有整数

接下来，Samek 让这些变量装入一些方便区分字节顺序的十六进制值，例如：

```c
u8a  = 0xa1U;
u16c = 0xc1c2U;
u32e = 0xe1e2e3e4U;
```

然后再做简单赋值，观察反汇编。你会第一次比较系统地看到 Cortex-M 对不同宽度对象使用不同指令：

1. `LDRB/STRB` 处理 byte。
2. `LDRH/STRH` 处理 half-word。
3. `LDR/STR` 处理 full word。

这一步很有价值，因为它把“不同整数宽度”从纯语言概念落到了具体机器指令层面。固定宽度类型的选择，最终真的会影响 CPU 的取数和存数方式。

## little-endian：为什么寄存器里看到的字节顺序和内存里相反

lesson 11 很自然地接着引出 endianness。Samek 在 memory window 中让你直接看 `0xe1e2e3e4` 被存入内存后的字节顺序，然后指出：ARM Cortex-M 在实际产品里几乎总是 little-endian。

所谓 little-endian，就是数值的低有效字节放在低地址。例如：

1. 寄存器中的值是 `e1 e2 e3 e4` 这种人类习惯的高位在前表示。
2. 存到内存后，低地址开始看到的却是 `e4 e3 e2 e1`。

这件事看似基础，但在嵌入式中非常实用，因为一旦你在做：

1. 协议解析
2. 寄存器拼装
3. 校验计算
4. 原始存储 dump 分析

你迟早都要面对“寄存器值”和“内存字节序列”不是同一视觉顺序的问题。

Samek 甚至还手工演示了 big-endian 会长什么样，目的是让你意识到：字节序不是语言层面的抽象，而是处理器组织多字节对象的真实约定。

## 这节课最重要的部分，其实是隐式类型转换

如果说前半节在做铺垫，那么后半节才是 lesson 11 的真正核心。Samek 明说，不可能在一课里讲完所有 C 隐式转换规则，但有几个原则你必须刻在脑子里，否则非常容易写出“在一台板子上对、换个平台就错”的代码。

### 规则一：小于 `int` 的整数，参与运算前会先提升到 `int` 或 `unsigned int`

这是第一条最容易制造错觉的规则。课程用这个表达式做示范：

```c
u16c = 40000U;
u16d = 30000U;
//u32e = u16c + u16d; // NOT portable!
u32e = (uint32_t)u16c + u16d;
```

在 32-bit ARM 上，`int` 本身就是 32 位，所以 `u16c + u16d` 被提升后在 32 位精度里计算，结果自然是 70000，看起来完全没问题。

但在 MSP430 上，`int` 只有 16 位，于是这两个 `uint16_t` 在参与运算时并没有提升到你以为的 32 位，而只是提升到 16 位 `int/unsigned int` 级别，结果在计算阶段就已经溢出了。最终再把这个结果赋给 `u32e` 时，左边虽然是 32 位，但接收的只是一个已经算坏了的 16 位结果。

Samek 借这个例子强调了一个非常重要的原则：

**表达式的计算精度，由参与运算的操作数类型决定，而不是由赋值左侧目标类型决定。**

这是很多人第一次真正踩坑后才会记住的规则。

## 为什么显式 cast 必须打在操作数上，而不是寄希望于左值够宽

为了解决上面的问题，课程给出的正确写法是：

```c
u32e = (uint32_t)u16c + u16d;
```

这样一来，由于一个操作数已经是 32 位，另一个也会被提升到 32 位，整个表达式在 32 位里完成，结果才是可移植的 70000。

这也是 Samek 一再强调“portable code”时真正指的东西：不是“这份代码在两台机器上都能编译”，而是“在两台机器上表达式求值语义相同”。

## 规则二：signed 和 unsigned 混合时，结果往往朝 unsigned 倾斜

第二个例子更微妙：

```c
u16c = 100U;
//s32  = 10 - u16c;  // NOT portable!
s32  = 10 - (int32_t)u16c;
```

很多人直觉会觉得，这不过是 `10 - 100 = -90`，左值 `s32` 也足够大，能出什么问题？

问题恰恰在于：表达式里混入了 unsigned 对象。按 C 的 usual arithmetic conversions，signed 和 unsigned 混合时，常常会把运算拉向 unsigned 域。

在 ARM 上，由于 `unsigned int` 是 32 位，结果会表现成一个 32 位无符号值，再在赋给 `s32` 时按二补码解释成负数，看起来“碰巧”还是对的。

但在 MSP430 上，这个中间 unsigned 结果只有 16 位，随后再扩展到 32 位时不会发生你期待的 sign extension，于是就会变成一个很大的正数，例如 65446，而不是 -90。

这正是 Samek 想让你警惕的地方：有些代码在 32 位 ARM 上“表现正常”，只是因为平台特性帮你掩盖了不良写法。

## 这个例子里的正确修复方式

Samek 给出的修复原则很明确：不要让 unsigned 悄悄支配整个表达式，而是显式把 unsigned 操作数转换到你真正希望的 signed 精度上。这里他最终使用的是：

```c
s32 = 10 - (int32_t)u16c;
```

这也说明一个更一般的经验：

1. 尽量不要随意混用 signed 和 unsigned。
2. 如果必须混用，明确写出你想让编译器采用的类型。
3. cast 的目标类型应该服务于表达式语义，而不是只是“看起来差不多”。

## 规则三：signed/unsigned 比较常常会产生“违反直觉”的结果

课程里的第三个陷阱是：

```c
//if (u32e > -1) {  // ALWAYS false!
if ((int32_t)u32e > -1) {
    u8a = 1U;
}
else {
    u8a = 0U;
}
```

初看时很多人会想：无符号数最小也有 0，所以它当然总是大于 -1，条件应该总为真才对。

但编译器却认为这个比较恒为假，甚至把分支直接优化掉。原因就在于 mixed signed/unsigned comparison 时，`-1` 会先被转换成无符号值。对于 32 位平台来说，它会变成 `0xFFFFFFFF`，也就是最大的 `uint32_t`。于是任何正常的 `u32e` 都不可能大于它，条件当然恒假。

Samek 通过这个例子想传达的是：

1. 你的数学直觉并不总等于 C 的类型转换规则。
2. 一旦 signed/unsigned 参与比较，必须先问“双方比较前会被转换成什么”。
3. 编译器的警告往往不是在挑刺，而是在提醒你表达式语义已经变味了。

## 规则四：小整数做位运算时，也会先提升成 `int`

最后一个非常经典的坑是：

```c
u8a = 0xffU;
//if (~u8a == 0x00U) { // ALWAYS false!
if ((uint8_t)(~u8a) == 0x00U) {
    u8b = 1U;
}
```

很多人直觉会觉得：`0xFF` 的按位取反当然应该是 `0x00`，所以比较应该为真。但课程里编译器甚至直接把这段 if 优化没了，因为它知道结果永远不可能为真。

原因还是 integer promotion。`u8a` 在做 `~` 之前先被提升成 `int`，于是高 24 位全是 0；取反之后，高位全变成 1，得到的并不是一个 8 位 `0x00`，而是类似 `0xFFFFFF00` 这样的整型结果，自然不可能等于 0。

解决办法就是显式把结果再裁回 byte：

```c
if ((uint8_t)(~u8a) == 0x00U)
```

这个例子特别重要，因为它说明“小整数做位运算”时，很多人想象中的 8 位封闭世界在 C 里并不存在。编译器会先把它们提升成 `int` 再算。

## 为什么 Samek 要拉出 MSP430 做对照

lesson 11 里最精妙的教学设计之一，就是把同一段代码搬到 MSP430 上跑一遍。因为如果你只在 ARM 上观察，很多表达式虽然写法有问题，但结果“恰好”仍是你期望的值，根本不容易看出不可移植性。

MSP430 作为 16-bit 平台，正好把这些潜伏问题全都放大出来。于是你会真正明白：

1. 可移植性不是语法层面的问题。
2. 真正的风险在于表达式求值精度和 signedness 规则。
3. 只在单一 32-bit 平台上测试，很容易误把偶然正确当成逻辑正确。

这也解释了为什么 Samek 说，这一课对面试准备也很有用。因为很多所谓“刁钻 C 题”，本质上都是在考你是否真正理解了这些隐式转换规则。

## 这一课和前后课程的关系

表面上 lesson 11 好像从 stack overflow 突然跳回了基础语法，但其实主题是一致的。lesson 10 告诉你：程序可以因为你误判栈行为而灾难性失败；lesson 11 则告诉你：程序也可以因为你误判表达式求值规则而静悄悄地算错。

前者更像 crash/debugging 问题，后者更像 correctness/portability 问题，但本质上都在强调一件事：嵌入式程序员不能只看源码表面语义，而必须知道编译器和目标处理器最终会怎样解释这些代码。

## 总结

lesson 11 的核心，是把几个经常被低估的 C 语言基础问题，重新放回嵌入式上下文中审视。Samek 先通过 `stdint.h` 说明固定宽度整数为什么对嵌入式至关重要，再通过反汇编和内存窗口展示 ARM 如何用 `LDRB/LDRH/LDR` 对应不同宽度对象，以及 little-endian 为什么会让内存中的字节顺序与寄存器显示顺序相反。

但整节课最重要的部分，仍然是隐式类型转换。`uint16_t + uint16_t` 可能在 16 位平台上先溢出；signed/unsigned 混合减法在不同平台上可能给出完全不同结果；`u32e > -1` 这类比较会因为 `-1` 被转换成无符号最大值而恒假；而 `~u8a` 这样的位运算也会因为 integer promotion 而产生违反直觉的结果。所有这些例子都在提醒你：表达式的计算方式由操作数类型决定，而不是由你主观想象或左值类型决定。

从课程节奏上看，lesson 11 是一堂非常典型的“把基础讲到会伤人程度”的课。它看起来不像后面的 startup code、interrupts 那样宏大，但实际上直接决定了你写出的底层代码在不同 MCU 上是否还能保持同样语义。接下来的 lesson 12，Samek 会把视线从标量类型转到结构体，并开始引入 CMSIS，让你看到 C 结构体如何不只是组织数据，还能直接成为访问硬件寄存器的语言接口。
