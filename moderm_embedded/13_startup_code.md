# 13 startup code：进入 main 之前，系统到底做了什么

这一课，Miro Samek 把视线第一次从 `main()` 往前推。前面 1 到 12 课，你几乎总是默认程序从 `main()` 开始，仿佛只要下载到板子、点运行，CPU 就自然会落到你的 C 代码入口。但 lesson 13 要解决的正是这个默认前提：真正的嵌入式程序，在 `main()` 之前其实已经发生了不少关键动作，而这些动作直接决定了你的变量、栈、向量表和运行时环境是否正确。

如果说前几课主要是在学“C 代码如何映射到机器行为”，那么 lesson 13 更进一步，开始追问：“C 代码开始之前，谁先把舞台搭好？”这也是很多嵌入式问题的真正源头。因为程序一旦在 `main()` 前就已经跑偏，后面再怎么调应用逻辑，方向都不对。

## 为什么这一课必须暂时离开 `main()`

Samek 一开始就让你关闭调试器里的 `Run to main`。这个动作看似只是 IDE 选项，实际上是在强迫你承认一个事实：

1. `main()` 不是程序的第一个执行点。
2. C 运行时环境不是天然存在的。
3. 如果不观察启动阶段，你就无法真正理解后面变量为什么会是那个值。

重新下载程序后，调试器停下来的第一个标签不是 `main`，而是 `__iar_program_start`。这一步非常关键，因为它第一次把“启动代码”从抽象名词变成了真实可见的控制流入口。

同时你还能观察到一个很有意思的现象：大多数寄存器都接近初始状态，但 `SP` 已经是一个看起来合理的 RAM 地址。也就是说，在你自己的 C 代码还没开始前，系统已经有了一套最基本的执行上下文。

## `__iar_program_start`：你并不是直接跳进 `main()`

课程里最先看到的调用，是启动代码中的：

```c
__iar_init_vfp();
```

然后还有一个更奇怪的入口 `?main`，接着才会进一步走向真正的 `main()`。这些符号提醒你：你现在所处的阶段还不是标准 C 语言的世界，而是工具链和运行时库为你准备执行环境的阶段。

Samek 特别指出，FPU 初始化之所以这么早发生，是因为后续任何代码如果要使用浮点硬件，都必须先让它处于可用状态。这也是 startup code 的典型职责之一：某些初始化必须足够早，否则后面的普通 C 代码连运行前提都不成立。

## `__low_level_init()`：留给用户最早的切入点

在 IAR 的启动流程里，`?main` 又会进一步调用 `__low_level_init()`。这是一个非常有代表性的“早期钩子”：

1. 如果用户没有自己定义，就使用库里的默认空实现。
2. 如果用户定义了它，就能在数据初始化前做一些非常早期的配置。
3. 它的返回值还能决定后续是否执行标准数据初始化。

这个设计背后的思路很实用。比如你如果想尽早切换系统时钟、提高 CPU 频率，就可以在这个阶段做，这样连后面的启动代码本身都能跑得更快。

Samek 借这一点想说明：startup code 不是一块完全封闭、神秘不可碰的黑盒，而是标准运行时与用户定制之间的一个接口层。

## 为什么要先理解“段（section）”

课程接着转去讲 map file，但在那之前先澄清“data section”到底是什么。对链接器来说，section 的定义其实很朴素：

1. 它是一个连续的内存块。
2. 这个内存块有名字。
3. 链接器负责把不同 section 放到 ROM/RAM 的合适位置。

在 lesson 13 之前，你已经写了各种变量、数组、结构体初始化，但如果不从 section 角度重新理解这些东西，就看不清启动代码到底在初始化什么。

## linker map file：嵌入式程序最有价值的“体检报告”之一

Samek 很强调 map file 的价值。他直说，这东西乍看像乱码，但对嵌入式程序员来说是宝藏。因为它至少回答了三个很重要的问题：

1. 你的代码、只读数据、可读写数据一共占了多少空间。
2. 每一类 section 被放到了哪里。
3. 某个符号、某个对象文件、某个库模块最终是怎么进入镜像的。

这和桌面程序开发的习惯差别很大。在嵌入式里，程序是否能跑、是否还放得下、某段 RAM 为什么被吃掉，经常都要回到 map file 才能讲清楚。

## `.text`、`.rodata`、`.bss`、`CSTACK`：启动前后的内存角色分工

在 map file 的 `Placement Summary` 里，课程让你第一次比较系统地看到几个核心 section：

1. `.intvec`：位于地址 0 附近的中断向量表。
2. `.text`：代码段。
3. `.rodata`：只读常量。
4. `.bss`：未显式初始化的数据。
5. `CSTACK`：栈空间。

这里最容易误解的是 `.bss`。很多初学者会觉得“没初始化就是没值”，但 C 标准要求静态存储期、未初始化对象在程序开始运行时必须为 0。所以 `.bss` 虽然在镜像里不需要存放初始化内容，但 startup code 必须在 RAM 中把它清零。

而 `CSTACK` 又不同，它通常只是预留一块栈空间，并不要求 startup 去按某个固定模式填值。

这一步是 lesson 13 的核心基础：不是所有 RAM 内容都来自 ROM 复制，也不是所有变量都靠同一种办法初始化。

## 为了观察 `.data`，课程故意往代码里加入显式初始化变量

最开始的工程里，真正带初始化值的数据还不够丰富，所以 Samek 在 `main.c` 中专门加入了一批示例变量：

```c
int16_t x = -1;
uint32_t y = LED_RED | LED_GREEN;

int16_t sqr[] = {
    1*1,
    2*2,
    3*3,
    4*4
};
```

还进一步加了结构体、嵌套结构体、数组成员等初始化对象，例如 `Point p1`、`Window w`、`Triangle t`。这些示例并不是为了教 C 语法本身，而是为了让链接器生成足够典型的 `.data` 和 `Initializer bytes`，从而能在 map file 里清楚地看到“已初始化数据”到底是如何被组织的。

也就是说，这一课虽然表面在讲 startup，但依然沿用了 Samek 一贯的做法：先准备一个方便观察底层行为的实验样本。

## 为什么会出现 `Initializer bytes`

当程序里加入带初值的全局/静态对象后，map file 会出现一个新的现象：

1. RAM 里新增 `.data` section。
2. ROM 里新增与之对应的 `Initializer bytes`。
3. 两者大小一致。

这个现象非常关键，因为它揭示了启动代码真正做的一件大事：

1. 已初始化数据的最终驻留位置在 RAM。
2. 但它们的初始值必须先放在 ROM 镜像里。
3. 启动时再由 startup code 把 ROM 中那一整块初始化镜像复制到 RAM 对应位置。

所以 `.data` 并不是“编译器直接放到 RAM 里就结束”的概念，而是“ROM 中有模板，启动时复制到 RAM”的概念。

## 链接器不是按变量逐个初始化，而是重排成块拷贝

Samek 在这里强调了一个特别值得记住的点：startup code 并不会“找到变量 `x`、写入 `-1`，再找到变量 `y`、写入某个常量”。它不会按源码级别一项一项初始化。

真正发生的是：

1. 链接器把所有需要带初值的对象重新布局进一块或几块连续 section。
2. 同时在 ROM 生成对应布局的初始化模板。
3. 启动代码只需要执行整块复制。

这就是为什么 startup code 看起来那么“低级”和“机械”，却又完全足够。它不理解你的 C 变量语义，它只理解 section copy 和 zero-fill。

## 单步观察 `__iar_data_init3()`：`.bss` 清零和 `.data` 复制是怎么发生的

lesson 13 后半段最有价值的部分，是 Samek 没有停留在原理图解，而是让你真的单步进 `__iar_data_init3()`。

为了让现象更明显，他甚至先把 RAM 区域填成 `0xFF`。这样，当 startup code 开始工作时，你就能非常直观地看到哪些字节被改成 0，哪些字节被拷进特定初值。

### `.bss` 清零

进入数据初始化函数后，先做的是 `.bss` 清零。课程里反复看到类似这种核心模式：

1. 一个寄存器装着 0。
2. 另一个寄存器指向 `.bss` 开始地址。
3. `STR` 循环不断把 0 写入 RAM。

Samek 还顺便借这个机会解释了带 post-increment 的寻址方式，这种写法非常适合 startup 这种高密度线性内存处理代码。

### `.data` 复制

清完 `.bss` 之后，startup code 才去处理 `.data`。这次会看到成对出现的 `LDR/STR`：

1. 源地址寄存器指向 ROM 中的 `Initializer bytes`。
2. 目标地址寄存器指向 RAM 中的 `.data`。
3. 循环执行，直到整块复制完毕。

这一步的工程意义特别大，因为它把“C 标准说变量会有初值”这件事，真正落到了机器层面。

## 向量表第一次真正出现在视野里

虽然这一课的主线是 startup code 和数据初始化，但在最后 Samek 还是把你一路带回到了最前面，也就是地址 0 处的 `.intvec`。这意味着 lesson 13 真正完成了一个从 `main()` 倒推回 reset 起点的闭环：

1. `main()` 之前有 startup code。
2. startup code 之前有 reset 后的第一入口。
3. 第一入口来自向量表。

这里向量表还只是“被发现”，不是“被你接管”。但正因为 lesson 13 先让你看见系统默认的 `__iar_program_start` 和 `.intvec`，lesson 14 才能顺理成章地继续问：既然这是默认库给我们的，那能不能换成我们自己的？

## 这一课真正建立起来的能力

lesson 13 的重点，表面上是“看 startup code”，但实际上它建立的是一种以后非常关键的调试能力：当程序在一开始就不对劲时，你不能只盯着 `main()`，而要能一路往前检查：

1. 栈是不是已经被正确设置。
2. `.bss` 是否被清零。
3. `.data` 是否从 ROM 正确搬到了 RAM。
4. 向量表是不是位于正确地址。
5. reset 之后真正跑进了哪个入口。

这类问题在裸机开发里非常常见，尤其是一旦更换工具链、修改链接脚本、手写 startup 文件，或者移植到别的芯片后，最先出错的往往就是这些“应用逻辑还没开始之前”的部分。

## 总结

lesson 13 的核心，不只是“知道有个 startup code”，而是第一次把 `main()` 之前的世界拆开来看。Samek 通过关闭 `Run to main`、单步进入 `__iar_program_start` 和 `__iar_data_init3()`，让你真正看到：C 程序的执行环境是被一步步搭出来的，而不是天然存在。

这一课最重要的技术结论有三个。第一，`.bss` 和 `.data` 的初始化机制不同：`.bss` 需要在 startup 阶段清零，而 `.data` 需要从 ROM 的 `Initializer bytes` 拷贝到 RAM。第二，map file 是理解程序内存布局、段大小和启动行为的关键工具。第三，startup code 最终会把你带到地址 0 附近的 `.intvec`，也就是向量表这个更底层的入口。

从课程节奏上看，lesson 13 像是“进入 reset 世界的第一课”。它先让你承认 `main()` 不是起点，然后把变量初始化、RAM 布局、ROM 模板和向量表这些概念串起来。接下来的 lesson 14 和 15，就会顺着这条线继续深入：既然默认 startup 和向量表都是库提供的，那么编译器、链接器到底怎样把这些东西拼起来？我们又怎样把向量表真正接管到自己手里？
