# 4 从内存映射寄存器开始：第一次点亮板载 LED

## 引言

从这一课开始，课程第一次真正进入“嵌入式”的核心：**程序不只是算数和控制流，而是要直接驱动硬件**。

你在上一课已经验证了工具链、下载和调试流程；这一课的目标非常明确：在开发板上点亮并闪烁 LED。但它背后真正要建立的能力是：

1. 理解 MCU 地址空间中“外设寄存器”也是内存地址的一部分。
2. 学会用 C 指针把某个固定地址当成可读写对象。
3. 看懂“写某个值到某个地址”如何变成一条汇编 `STR` 指令。

这套思路会贯穿后续所有章节。

## 地址空间全景：代码、RAM、外设

以 Cortex-M 微控制器为例，程序看到的是统一的 32 位地址空间（$2^{32}$ 字节）。

典型分区（概念级）如下：

- `0x00000000` 附近：Flash/ROM（代码和常量）
- `0x20000000` 附近：SRAM（变量、栈）
- `0x40000000` 附近：外设寄存器（GPIO/UART/Timer 等）

关键思想：

- 访问 RAM：是“普通数据读写”。
- 访问外设寄存器：从 CPU 看也是“普通数据读写”。
- 差别在于，外设地址背后不是 SRAM 单元，而是硬件电路逻辑。

所以“控制 LED”在机器层面就是：

$$\text{store value} \rightarrow \text{specific peripheral address}$$

## Lesson-04 代码主线（TM4C）

课程源码（lesson-04）里最核心的写法是这种“裸地址写寄存器”：

```c
*((unsigned int *)0x400FE608U) = 0x20U;
*((unsigned int *)0x40025400U) = 0x0EU;
*((unsigned int *)0x4002551CU) = 0x0EU;

while (1) {
    *((unsigned int *)0x400253FCU) = 0x02U;
    ...delay...
    *((unsigned int *)0x400253FCU) = 0x00U;
    ...delay...
}
```

这四个地址分别做了什么：

1. `0x400FE608`：使能 GPIOF 时钟（不给时钟，外设不工作）。
2. `0x40025400`：GPIOF 方向寄存器，设置 LED 对应引脚为输出。
3. `0x4002551C`：数字功能使能寄存器。
4. `0x400253FC`：GPIO 数据寄存器（写 0x02 点亮红灯，写 0 熄灭）。

你会看到这里还没有“优雅命名”，全是硬编码常量，这是故意的：先把硬件访问本质讲清楚。

## STM32 版本为何看起来不同

在 lesson-04 的 STM32C031 工程中，写法仍是裸地址，但寄存器地址和配置步骤不同，例如：

```c
*((unsigned int *)0x40021034U) |= 0x01U; // RCC IOPENR
*((unsigned int *)0x50000000U) &= ~0xC00U; // MODER
...
*((unsigned int *)0x50000018U) = 0x20U;      // BSRR 置位
*((unsigned int *)0x50000018U) = 0x200000U;  // BSRR 复位
```

同样的“点灯任务”，两家芯片的寄存器组织完全不同，这正是嵌入式开发的现实：

- 逻辑共通：开时钟、设方向、使能数字输出、写输出。
- 细节平台相关：寄存器地址、位定义、寄存器语义不同。

这也解释了后面为什么要引入设备头文件和抽象层。

## 指针强制类型转换的真正含义

表达式：

```c
*((unsigned int *)0x400253FCU) = 0x02U;
```

分解成三步：

1. `0x400253FCU` 是整数常量地址。
2. `(unsigned int *)0x400253FCU` 把它“解释”为指向 `unsigned int` 的指针。
3. `*pointer = value` 对该地址执行写操作。

编译器不会替你“验证这地址是不是合法寄存器”。合法性完全由你和数据手册保证。

## 这条 C 语句对应的汇编

在 Cortex-M 上，这类语句通常会被编译成：

```asm
LDR   R0, =0x400253FC
MOVS  R1, #2
STR   R1, [R0]
```

核心是最后一条 `STR`：把寄存器值存到外设地址。也就是说，CPU不区分“这是 RAM 还是 GPIO”，总线和外设逻辑去解释这次访问。

## 延时循环：为什么要 `volatile`

lesson-04 里有一个忙等待循环：

```c
int counter = 0;
while (counter < 1000000) {
    ++counter;
}
```

在高优化等级下，编译器可能把它优化掉（因为它对“纯软件可见状态”没有贡献）。

因此后续课程会把它改成：

```c
int volatile counter = 0;
```

这告诉编译器：每次访问都必须保留，不可随意删减或合并。

## 为什么先用“丑代码”

新手常问：为什么不用 HAL、库函数、驱动框架，非要写裸地址？

因为课程目标是“先掌握最底层因果关系”。你现在学到的是：

- LED 闪烁不是魔法。
- 不是“调用某 API 就亮了”。
- 本质是向某个寄存器地址写某个比特模式。

当你之后使用 CMSIS/HAL 时，才能清楚它们只是“更好读的封装”。

## 常见错误与排查思路

### 1. LED 完全不亮

优先检查：

1. 外设时钟是否使能。
2. 引脚是否被配置为输出模式。
3. 数字功能是否开启（平台相关）。
4. 写入的数据寄存器地址是否正确。

### 2. 代码跑飞或 HardFault

常见原因：

- 地址写错（打错一个 hex 位）。
- 使用了未对齐访问（某些架构/配置下触发 fault）。
- 访问了该芯片并不存在的地址窗口。

### 3. 延时快到看不见

常见原因：

- 编译优化删除了延时循环。
- 主频与预期不同，导致循环时间常数变化。

## Lesson-04 的教学价值

这一课真正建立的不是“闪灯技能”，而是三个长期收益：

1. **内存映射 I/O 心智模型**：所有外设都可以被看作“有副作用的内存单元”。
2. **数据手册驱动开发**：寄存器地址、位定义、访问语义都要回到官方文档确认。
3. **机器级可解释性**：从 C 到汇编再到总线行为，可以形成闭环理解。

后续第 5 课会把这些硬编码地址改造成更可维护的命名形式；第 6 课会进一步引入位运算，解决“只改某一位，不破坏其余位”的真实工程问题。

## 小结

本课完成了从“运行程序”到“控制硬件”的第一步：

- 你知道了 MCU 地址空间中外设寄存器的位置意义。
- 你可以用指针解引用完成寄存器读写。
- 你看到了点灯背后的最小机器动作：`STR` 写入特定地址。

这为下一课的主题自然铺垫：**如何把可运行但难维护的裸地址代码，升级为可读、可维护、可扩展的工程代码。**
