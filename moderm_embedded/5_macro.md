# 5 让裸地址代码可维护：宏、设备头文件与 volatile

## 引言

第 4 课我们已经完成了“裸地址点灯”：代码能跑，但充满神秘数字，不利于维护和协作。

第 5 课的核心目标是把“能跑”升级为“可维护”：

1. 用宏和命名常量替代硬编码地址。
2. 认识预处理器只做文本替换，不做类型检查。
3. 理解为什么在嵌入式里 `volatile` 是必须掌握的关键字。
4. 引入设备厂商头文件，让寄存器访问从“记地址”转为“读名字”。

## Lesson-05 源码变化（TM4C）

lesson-05 的 TM4C 工程已经从裸地址切换到符号名：

```c
#include "tm4c.h"

int main(void) {
    SYSCTL_RCGCGPIO_R = 0x20U;
    GPIO_PORTF_DIR_R = 0x0EU;
    GPIO_PORTF_DEN_R = 0x0EU;

    while (1) {
        GPIO_PORTF_DATA_R = 0x02U;

        int volatile counter = 0;
        while (counter < 1000000) {
            ++counter;
        }

        GPIO_PORTF_DATA_R = 0x00U;
        counter = 0;
        while (counter < 1000000) {
            ++counter;
        }
    }
}
```

代码逻辑和第 4 课几乎一样，但可读性提升非常明显。

## 宏的本质：预处理阶段文本替换

在 C 的编译流程里，预处理是独立阶段。`#define` 的作用是**替换字符序列**，不是“创建类型安全变量”。

例如（概念示例）：

```c
#define GPIO_PORTF_BASE   0x40025000U
#define GPIO_PORTF_DIR_R  (*((unsigned int *)(GPIO_PORTF_BASE + 0x400U)))
```

当编译器真正看到代码时，宏名会先被替换成对应文本。意味着：

1. 宏没有作用域语义（与变量不同）。
2. 宏替换可能受上下文影响。
3. 写错宏定义，错误会在替换后才暴露。

所以宏定义通常要加括号保护优先级。

## 为什么宏要加括号

危险写法：

```c
#define MASK 1U << 3
```

如果你写 `x & MASK + 1U`，替换后可能变成：

```c
x & 1U << 3 + 1U
```

优先级就乱了。更稳妥写法：

```c
#define MASK (1U << 3)
```

对于寄存器宏，常见风格是把整个表达式包裹好：

```c
#define GPIO_PORTF_DATA_R (*((volatile unsigned int *)(0x400253FCU)))
```

## 设备头文件带来的工程价值

lesson-05 使用 `tm4c.h` / `stm32c031.h` 这类头文件，本质是把：

- 地址常量
- 位定义
- 寄存器访问宏

集中到一个可复用、可审计的地方。

好处：

1. **一致性**：全团队使用同一寄存器命名。
2. **可维护性**：芯片迁移时集中修改。
3. **可读性**：`GPIO_PORTF_DEN_R` 比 `0x4002551C` 更有语义。
4. **减少手误**：少写一堆十六进制字面量。

这也是为什么工业项目通常不会在业务代码里到处直接写裸地址。

## volatile 的真正语义

`volatile` 的语义不是“线程安全”，也不是“原子”。它告诉编译器：

- 对这个对象的每次读写都必须保留。
- 不要把它缓存到寄存器后长期复用。
- 不要合并或消除你认为“多余”的访问。

### 在嵌入式里的两类典型场景

1. **外设寄存器**：寄存器值可能被硬件在后台改变。
2. **延时计数器/中断共享变量**：值可能被不可见路径改变。

因此寄存器宏一般都应基于 `volatile` 指针。

## 为什么延时计数器也要 volatile

若你写：

```c
int counter = 0;
while (counter < 1000000) {
    ++counter;
}
```

在优化级别较高时，编译器可能把循环删掉或重排，因为它认为循环结果“不影响可观察行为”。

改为：

```c
int volatile counter = 0;
```

就会强制每次读写都落地，从而保留延时循环。虽然这种延时不精确，但用于教学示例和简单演示足够直观。

## Lesson-05 的 STM32 对照

lesson-05 的 STM32 版本同样体现“命名替代裸地址”：

```c
#include "stm32c031.h"

RCC_IOPENR_R |= 0x01U;
GPIOA_MODER_R   &= ~0xC00U;
GPIOA_MODER_R   |=  0x400U;
...
GPIOA_BSRR_R = 0x20U;
...
GPIOA_BSRR_R = 0x200000U;
```

与 TM4C 一样，核心动作仍是配置 GPIO 并在循环里置位/复位，但通过命名寄存器表达语义。

## 预处理器常见陷阱

### 1. 宏参数副作用

```c
#define SQUARE(x) ((x) * (x))
```

如果调用 `SQUARE(i++)`，`i` 可能自增两次。

### 2. 调试困难

编译错误行号可能指向“替换后代码”，而不是你直观看到的宏定义位置。

### 3. 类型不安全

宏本身不携带类型信息。若需要更强类型约束，可用 `static inline` 函数替代部分宏。

## 宏与 inline 的取舍

对于寄存器地址常量，宏非常自然；对于可执行逻辑，`static inline` 往往更安全：

```c
static inline void led_on(void) {
    GPIO_PORTF_DATA_R = 0x02U;
}
```

优点：

1. 有类型检查。
2. 可调试性更好。
3. 在优化打开时通常同样无调用开销。

## 第 5 课真正建立的能力

你现在不只是“会点灯”，而是开始具备工程化习惯：

1. 避免魔法数字（magic numbers）。
2. 把硬件信息收敛到头文件。
3. 正确使用 `volatile` 保证硬件访问语义。
4. 认识到可读性和可维护性也是嵌入式质量的一部分。

这些能力会在第 6 课继续扩展：当你要同时控制 RGB 三个通道时，简单赋值会破坏其他位，需要位运算做“精确改位”。

## 小结

本课把第 4 课“能跑但粗糙”的裸地址代码升级到可维护版本：

- 通过寄存器命名提升可读性。
- 通过预处理器理解宏的力量和边界。
- 通过 `volatile` 建立与优化器正确协作的思维。
