## Lesson-02 源码主线

lesson-02 的核心代码非常经典：

```c
int main(void) {
    int counter = 0;
    while (counter < 21) {
        ++counter;
        if ((counter & 1) != 0) {
            /* odd path */
        }
        else {
            /* even path */
        }
    }
    return 0;
}
```

它同时演示了三件事：

1. 有界循环（直到条件不满足退出）。
2. 条件分支（奇偶路径分离）。
3. 用按位与判断奇偶（`counter & 1`）。

## while 循环在机器层的样子

`while (counter < 21)` 并不是“高级概念”，编译后通常是：

1. 比较（`CMP`）
2. 条件跳转（`BGE`/`BLT` 等）
3. 回跳到循环头（`B`）

概念汇编可写成：

```asm
LOOP_HEAD:
    LDR   R0, [counter]
    CMP   R0, #21
    BGE   LOOP_END

    ADDS  R0, R0, #1
    STR   R0, [counter]

    ... if/else ...
    B     LOOP_HEAD

LOOP_END:
```

这说明循环的本质就是“比较 + 跳转 + 回边”。

## if/else 的底层结构

`if ((counter & 1) != 0)` 先计算表达式，再分支：

```asm
LDR   R0, [counter]
ANDS  R1, R0, #1
BEQ   EVEN_PATH   ; 若结果为0走偶数分支

ODD_PATH:
    ...
    B   AFTER_IF

EVEN_PATH:
    ...

AFTER_IF:
```

关键点：高级语言中的“真假判断”，在机器层通常是设置条件标志位（flags）后执行条件分支。

## 为什么用 `counter & 1` 判断奇偶

数学上：一个整数最低位为 1 表示奇数，为 0 表示偶数。

因此：

- `counter & 1 == 1` → odd
- `counter & 1 == 0` → even

相比 `counter % 2`，按位与通常更直接，且更贴近底层思维。在许多编译器上 `%2` 也会被优化成位运算，但显式写位运算有助于教学和可解释性。

## 控制流图（CFG）直觉

把这段程序画成控制流图，你会得到：

1. 入口块（初始化 `counter`）
2. 循环判断块（`counter < 21`）
3. 循环体块（自增）
4. if 判断块（奇偶）
5. odd 块 / even 块
6. 回边到循环判断
7. 退出块（`return 0`）

这张图的价值在于：

- 你可以直观看到“路径数量”。
- 后面做代码覆盖率和测试时，会用到“每条边是否执行”。

## 调试时应重点观察什么

第 2 课推荐你在调试器里观察：

1. `counter` 每轮变化是否符合预期（1 到 21）。
2. odd/even 路径是否交替命中。
3. 退出条件触发时 PC 是否跳到循环后。

如果你看到 `counter` 没变化或者跳转异常，通常是：

- 条件写错（如 `<` 与 `<=` 混淆）。
- 自增位置放错。
- 变量被优化器处理后观察窗口不直观（可调低优化级别）。

## 常见控制流错误

### 1. 无限循环

例如把更新语句漏掉：

```c
while (counter < 21) {
    // 忘记 ++counter
}
```

结果条件永远为真。

### 2. 边界 off-by-one

`< 21`、`<= 21`、`< 20` 的行为不同，循环次数分别会差 1。

### 3. 条件表达式副作用

若把复杂副作用塞进 `if` 条件，调试难度会明显上升。初学阶段建议条件表达式保持纯粹。

## 从 lesson-02 到后续章节的桥梁

你在这课获得的不是语法，而是“分支模型”：

1. 所有高级控制流最终都落到比较与跳转。
2. 分支会产生多条执行路径。
3. 程序正确性不仅看“结果”，还要看“路径是否正确命中”。

这些思想会在后续持续出现：

- 第 6 课：位运算条件判断。
- 第 9 课：递归导致更复杂控制流。
- 第 35 课后状态机章节：显式管理事件驱动路径。

## 代码风格建议（从第二课就开始）

虽然示例很短，依然建议保持工程习惯：

1. 条件表达式写成可读形式（必要时加括号）。
2. `if`/`else` 块都用花括号，避免未来改动引入歧义。
3. 对位运算意图写简短注释（如 odd/even）。

这能显著降低后续维护成本。
