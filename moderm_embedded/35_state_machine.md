# 35 状态机入门

这一课，Miro Samek 开始一个新的主题：state machines。它并不是和前面的 event-driven programming、active object 平行无关的内容，恰恰相反，Samek 把它视为上一课的自然延续。因为一旦你采用 event-driven 和 active object 的写法，就会立刻遇到一个核心问题：程序在处理完一个事件并返回 event loop 之后，如何记住“自己当前处在什么上下文里”，从而在下一个事件到来时做出正确反应。

上一课的 `blinky button` 例子其实已经暴露了这个问题。为了知道 timeout 到来后应该把绿灯打开还是关闭，代码里不得不引入一个布尔变量 `is_led_on`。这个变量在小例子里看上去没什么问题，但 Samek 指出，这正是很多事件驱动代码开始滑向杂乱 flag 管理的起点。于是这一课的任务，就是解释：为什么 state 是保存“相关历史”的正确抽象，为什么 state machine 是事件驱动系统里组织行为的合适机制，以及如何把一个简单 active object 改写成 state machine。

## 从上一课的 `is_led_on` 开始：事件驱动代码为什么会需要显式保存上下文

课程一开始先回顾上一课的 `blinky button` active object。为了实现 LED 在 timeout 事件之间交替亮灭，代码里引入了一个 `is_led_on` 标志，用来记住当前绿灯是不是亮着。

Samek 强调，这个标志在原来的 RTOS 线程版本中其实并不需要。原因在于顺序式代码使用 blocking delay，于是“程序执行到了哪一步”本身就隐含保存了上下文：

1. 线程阻塞前在某个位置。
2. 等待结束后从该位置继续执行。
3. 所以下一步该处理什么，是由程序计数器和栈上下文自然决定的。

这是一种“自动的上下文管理”。它很方便，但代价很高，因为线程必须持有完整的栈上下文，消耗宝贵 RAM。

事件驱动代码则恰恰相反：

1. 处理完一个事件后必须返回 event loop。
2. 栈上的执行位置不会跨事件保留下来。
3. 因此程序必须用别的方式保存跨事件所需的上下文。

如果没有更好的方法，程序员往往会自己发明各种 flag 和变量，这就是 manual context management。

## 手工 flag 管理上下文，最终很容易退化成 spaghetti code

Samek 用一个 Visual Basic 写的四则运算计算器作为反例，说明“手工维护上下文变量”为什么会迅速失控。

这个计算器并不复杂，但当用户输入一些不那么常规的按键序列时，程序会表现异常，甚至直接崩溃。比如某些按键顺序会导致 runtime error。

深入代码后会发现，问题的根源不是某一行简单 bug，而是整个程序把大量精力都用在“猜测和维护当前上下文”上了。代码里充斥着各种变量，例如：

1. 是否已经输入过小数点。
2. 当前输入的是第几个操作数。
3. 上一个操作符是什么。
4. 当前是否在等待下一输入。

这些变量彼此组合、互相制约，又散落在不同事件处理函数里。于是：

1. 上下文信息变得冗余。
2. 这些冗余信息可能彼此不一致。
3. 修复一个 corner case 时，很容易破坏另一个路径。

Samek 的意思并不是“不要保存上下文”，而是说：上下文必须被更严谨地组织，而不是随手堆成一堆 flags。

## 真正需要记住的不是全部历史，而是 relevant history

课程接着提出一个关键洞见：为了正确响应未来事件，系统并不需要记住完整的过去历史，而只需要记住其中对未来行为有影响的那一部分，也就是 relevant history。

Samek 用键盘的 Shift 键举例。一个键盘控制器输出大写还是小写字符，真正相关的历史只有一件事：Shift 当前是否被按下。

至于：

1. 之前按过哪些其他键。
2. 一共按过多少次键。
3. 输入序列有多长。

这些信息对“下一次按键应该输出大写还是小写”并不重要。

因此，这个系统的相关历史其实只需要压缩成两个类别：

1. 正常状态。
2. Shift 按下状态。

这个例子说明，保存上下文最有效的方法，不是记住所有细节，而是提炼出那些真正决定未来行为的类别。

## 状态的定义：对过去历史进行等价类压缩

这就引出了 Samek 给出的核心定义：

state of a system is an equivalence class of past histories。

也就是说，所谓“状态”，本质上是对过去历史进行分类：如果两段过去历史会导致系统未来行为完全相同，那么它们就属于同一个状态。

从工程角度看，这个定义非常重要，因为它说明 state 不是随便拍脑袋起的名字，而是：

1. 对 relevant history 的最小、最有效表示。
2. 只保留决定未来行为所需的信息。
3. 把无关细节全部抽象掉。

因此，state 可以被看作是“跨事件保存上下文”的正规方法，而不是权宜之计。

## 状态机的定义：状态集合加上状态转移规则

有了状态的概念，state machine 也就自然出现了。Samek 给出的理解很直接：

1. state machine 包含一组状态。
2. 它还包含在不同事件到来时，状态如何变化的规则。

这些规则就是 transitions。

从行为上看：

1. 有些事件会改变系统未来的行为，因此触发状态转移。
2. 有些事件虽然要执行动作，但不会改变当前上下文，因此不改变状态。

这就解释了为什么 state machine 能把“上下文管理”和“事件处理”统一起来：状态负责表示 relevant history，transition 负责说明哪些事件会改变 relevant history。

## UML 状态图：把行为结构清晰画出来

课程随后引入了 UML state diagram 的基本记法。Samek 用它来说明状态机之所以强大，不只是因为概念正确，还因为它天然具有很强的可视化表达能力。

在 UML 中：

1. 状态画成圆角矩形。
2. 普通状态转移画成带箭头的连线，并标注触发事件。
3. 动作写在斜杠 `/` 后面。
4. 不改变状态、只执行动作的处理称为 internal transition，写在状态内部。
5. 状态机必须有一个 initial transition，用来进入默认初始状态。

Samek 也提到，状态机有很多变体，后面还会讲 hierarchical state machines，也就是 UML statecharts。但这一课先聚焦最基础、最直接的平面状态机。

## 状态机与事件驱动系统天然匹配，因为二者都假设 RTC

这一课中最重要的连接点之一，是 Samek 明确指出：所有 state machine formalism 都默认 Run-to-Completion 语义。

也就是说：

1. 状态机一次只处理一个事件。
2. 当前事件的处理必须完整结束。
3. 下一个事件必须等当前处理完成之后才能开始。

这与上一课 event-driven system 和 active object 的 RTC 假设完全一致。

因此，state machine 非常适合用来描述 active object 的行为。换句话说：

1. active object 提供并发与事件队列的运行框架。
2. state machine 提供该对象内部行为的规范表达方式。

这两者不是竞争关系，而是天然互补关系。

## 用状态机重写 `blinky button`：不再依赖 `is_led_on`

接下来，课程把上一课的 `blinky button` active object 重画成状态机。

设计过程很有代表性。首先从 initial transition 开始，因为它决定系统一创建出来时的默认条件。

在这个例子里，initial transition 的动作包括：

1. 关闭绿灯。
2. 让 time event 在“灭灯时长”后超时。

这说明默认状态应该是 `Off`。

然后在 `Off` 状态下，收到 `TIMEOUT` 事件时：

1. 打开绿灯。
2. 重新设置下一个 timeout。
3. 状态转移到 `On`。

在 `On` 状态下，收到 `TIMEOUT` 事件时：

1. 关闭绿灯。
2. 设置另一个 timeout。
3. 状态转回 `Off`。

这样，原来需要靠 `is_led_on` 手工判断的逻辑，现在被两个显式状态 `Off` 和 `On` 自然表达出来。当前到底该亮还是灭，不再由某个临时布尔变量隐含表达，而是由 state machine 的当前状态直接表达。

## 按钮事件在这里属于 internal transition

除了 timeout，`blinky button` 还要处理按钮按下和释放。

在这个例子里，这两个事件并不会改变绿灯处于 `On` 还是 `Off` 的状态，它们只是：

1. 在按下时点亮蓝灯，并缩短 `blink_time`。
2. 在释放时熄灭蓝灯。

因此它们属于 internal transition，而不是普通的状态转移。

这点很重要，因为 state machine 并不是说“每个事件都必须切状态”，而是要区分：

1. 哪些事件改变 relevant history，因此要变状态。
2. 哪些事件只触发动作，但不改变当前行为语境，因此只是 internal transition。

## 平面状态机里的重复，是层次状态机将来要解决的问题

课程还特意指出一个现象：按钮按下和释放这两个 internal transition，在 `Off` 和 `On` 两个状态里都要重复写一遍。

在这个小例子里问题不大，但在复杂系统中，这种重复会迅速积累。这正是 hierarchical state machine 要解决的问题之一。

Samek 在这里没有展开，而是先留到后续课程处理。不过这个提醒很有价值，因为它说明：

1. 平面状态机已经比 flags 清晰很多。
2. 但如果系统再复杂一些，层次化会变得非常有必要。

## 最直接的 C 实现：枚举状态 + 双层 switch

设计完状态图后，课程开始说明如何把它手工实现成 C 代码。Samek 先选了一种最直接、最容易理解的方法，而不是最优雅或最紧凑的方法。

### 用枚举定义 state variable

状态机最核心的数据，就是 current state。因此首先定义一个枚举：

1. `OFF_STATE`
2. `ON_STATE`

然后在 active object 的私有数据里保存一个 `state` 变量。

这就是对前面各种 improvised flags 的替代。Samek 特别强调：虽然 `blinky button` 里只是用一个 state variable 替换了一个 `is_led_on`，看起来改进不算巨大，但在更复杂的系统里，一个设计良好的 state variable 可以替代大量互相纠缠的 flags。

### 把状态机代码放进 AO 的 `dispatch()`

Samek 认为，状态机代码最适合写在 active object 的 `dispatch()` 中。原因很简单：

1. active object 没有事件时，本来就处于 dormant 状态。
2. 只有事件到来时，`dispatch()` 才会被调用。
3. 这与状态机“平时停留在当前状态，等事件触发后才运行转移”的行为完全一致。

这再次体现了 state machine 与 active object 的天然匹配。

## 把状态图机械映射成代码：先处理初始转移，再按状态分发事件

课程给出了一套很直接的编码规则。

### 1. 特判 `INIT_SIG`

在 `dispatch()` 一开始，先检查当前事件是不是保留的 `INIT_SIG`。如果是，就执行 initial transition 上的动作，然后把状态变量设置为初始目标状态，最后直接返回。

### 2. 第一层 switch 按当前状态分发

如果不是初始化事件，就对 `state` 变量做 `switch`，分别进入 `OFF_STATE`、`ON_STATE` 等分支。

### 3. 第二层 switch 按事件 signal 分发

在每个状态分支内部，再对事件 signal 做 `switch`。

例如在 `OFF_STATE` 中：

1. `TIMEOUT_SIG` 执行动作并把状态改成 `ON_STATE`。
2. `BUTTON_PRESSED_SIG` 执行动作但不改状态。
3. `BUTTON_RELEASED_SIG` 执行动作但不改状态。

在 `ON_STATE` 中则对应相反的 timeout 转移和相同的按钮内部转移。

这就是最基础的“状态 + 事件 -> 动作 + 下一个状态”编码方式。

## 状态机代码的一个关键优点：编码过程变成机械映射

Samek 特别强调了一个体验上的变化：一旦状态图设计清楚，写代码这件事本身就不需要太多“重新思考业务逻辑”。

因为：

1. 设计阶段已经决定了状态、事件、转移和动作。
2. 编码阶段只是在按一组稳定规则做映射。
3. 这使代码实现更机械、更系统，也更不容易漏掉情况。

这也是为什么建模工具可以自动生成状态机代码。不是因为它能“理解业务”，而是因为状态图本身已经把核心设计表达清楚了。

## traceability：代码和设计之间可以一一对应

课程把这一点称为 traceability。也就是说，状态图中的每个状态、每条转移、每个动作，都能在代码里找到清楚对应的位置。

这在 safety-critical software 中尤其重要，因为：

1. 你必须能说明某段行为是从哪里设计出来的。
2. 也必须能从设计反查到代码实现。

相比之下，杂乱 flags 和一堆分散的 if/else 往往很难做到这种双向追踪。

## extensibility：新增状态或事件时，修改范围更清晰

Samek 还强调了另一个很实际的收益：extensibility。

当系统需要扩展行为时，状态机给出了非常明确的修改入口：

1. 如果新增一个状态，就知道要在哪些状态分支中补上处理。
2. 如果新增一个事件，就知道要检查哪些状态需要响应该事件。
3. 如果改变一条转移，就知道该改哪条规则，而不是到处猜测哪些 flags 会受影响。

这使系统演进时的影响范围更可控。

## 运行结果不变，但结构质量显著提高

课程最后验证，改成状态机实现后，`blinky button` 的运行效果与原先保持一致，编译也没有额外错误或警告。

但 Samek 真正想强调的并不是“代码行数变少了”或者“运行更快了”，而是：

1. 上下文管理从临时 flags 升级成正式的 state 表达。
2. 行为从零散事件处理升级成有结构的状态转移系统。
3. 设计与代码之间建立了清晰映射。

这是一种结构质量上的提升，而不是表面语法变化。

## 总结

35 课的核心，是回答一个前面 event-driven 设计中已经浮现的问题：当程序不能阻塞等待时，应该如何正确保存上下文。

这一课最重要的结论可以概括为：

1. 顺序式线程通过栈和阻塞调用自动保存上下文，但代价是更高的 RAM 消耗和更差的可组合性。
2. 事件驱动代码在每个事件后都会返回 event loop，因此必须显式保存跨事件的 relevant history。
3. 随意堆积 flags 和变量的 manual context management 很容易退化成 spaghetti code。
4. 状态是对 past history 的等价类压缩，只保留影响未来行为的最小信息。
5. 状态机由状态集合和状态转移规则组成，是组织事件驱动行为的自然方法。
6. state machine 与 active object 天然兼容，因为二者都依赖 Run-to-Completion 事件处理语义。
7. 在 C 中，最直接的实现方式是 state variable 加双层 switch，而这一实现过程具有很强的机械性、traceability 和 extensibility。

如果说第 34 课解释了 active object 如何组织并发，那么第 35 课则开始解释 active object 内部的行为应该如何被规范地表达。它真正建立起来的是一个关键认识：event-driven 代码不是不需要上下文，而是需要把上下文从“临时变量拼凑”提升为“明确建模的状态”。这也是后续层次状态机和更高级行为建模方法的起点。
