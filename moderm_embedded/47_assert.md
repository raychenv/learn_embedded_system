## 什么是软件断言

断言本质上是一个 Boolean expression，用来在程序运行时检查某个条件是否成立。

1. 条件为真，说明程序当前仍处在设计者预期的范围内。
2. 条件为假，说明已经出现 programming error，也就是 bug。
3. 一旦断言失败，继续执行通常已经没有意义。

课程里先用标准 C 的 `assert()` 说明最基本形式，例如：

1. 整数除法函数里断言除数不能为 0。
2. `log2()` 函数里断言输入不能越界。
3. 当故意传入非法参数时，程序会立即报出失败的断言表达式和位置。

Samek 想借这个最简单的例子说明，断言不是在“补救”错误，而是在尽可能接近错误源头的位置把问题直接暴露出来。

## 标准 `assert()` 的几个关键特征

标准 C 断言机制虽然简单，但它有几个很值得注意的行为特征：

1. 失败信息通常输出到 `stderr`，普通输出走 `stdout`。
2. 断言失败会中止程序，因此后续代码不会继续执行。
3. 定义 `NDEBUG` 后，断言可以被完全编译掉。

这几个特征放在桌面程序上很自然，但也带出了 embedded context 下的问题：

1. 很多 MCU 系统根本没有标准输出错误流。
2. 也没有桌面系统那种“abort 后交还给 OS”的环境。
3. 因此标准 `assert()` 思路不能直接照搬到嵌入式产品里。

不过，在进入 embedded 版本之前，这一课更重要的是先把断言的使用边界讲清楚。

## 错误和异常条件不是一回事

要正确使用断言，必须先区分两类完全不同的情况：error 和 exceptional condition。

error 指的是程序设计或实现错误，例如：

1. 除以 0。
2. 数组越界。
3. 解引用空指针。
4. 外设未初始化就直接使用。

这些情况不是系统运行中“合理但不常见”的输入，而是软件本身出了错。面对这种错误，目标不应该是优雅处理，而应该是：

1. 尽快检测。
2. 清晰报告。
3. 阻止系统继续在错误状态下乱跑。

而 exceptional condition 则是系统生命周期中可能合法出现、但不在主路径上的情况，例如：

1. 用户输入错误。
2. 无线链路传输错误。
3. 降级模式或异常工况。

这些不是 bug，而是系统设计本来就应该考虑并通过 regular code 处理的场景。Samek 明确强调：这种情况不该用断言。

## 为什么 defensive programming 往往适得其反

课程专门批评了一种很常见但有害的习惯：defensive programming。

所谓 defensive programming，就是把真正的 bug 也当作普通异常来“兼容”或“兜底”，例如：

1. 除数为 0 时不报错，而是返回某个伪造值。
2. `log2()` 越界时随便返回 `-1` 或 `999`。
3. LED 驱动函数发现 GPIO 没初始化，就偷偷自己补初始化。

表面看这似乎更“robust”，但实际问题很严重：

1. 它隐藏了真正的 bug。
2. 它让错误变成更隐蔽、更晚才暴露的问题。
3. 它增加额外分支和复杂度，反而会制造新 bug。

Samek 的立场非常明确：真正的系统级容错行为必须是明确设计出来的，而不是靠“到处多写点 if”神奇地产生。

## Design by Contract 的核心思想

在断言之上，Samek 引入了 Design by Contract。

DBC 把软件组件之间的关系看成“契约”：

1. 调用者必须满足某些前提。
2. 被调用者承诺某些结果。
3. 模块内部还维持某些 invariant。

这些契约最好直接写进代码，并在运行时自动检查。这样做有两个直接收益：

1. 它能帮助快速发现 bug。
2. 它本身就是一种非常强的文档，清楚表达模块之间的责任边界。

也就是说，断言不仅是 runtime check，也是设计说明的一部分。

## 断言不等于错误处理

这一课最重要的澄清之一是：断言不是 error-handling mechanism。

1. 它不能阻止 bug 发生。
2. 它也不会修复 bug。
3. 它做的是在 bug 已经发生时，尽早、明确、强制地暴露它。

Samek 特别强调，看似“把每个断言失败都变成致命错误”似乎很激进，但这恰恰是它的价值所在。因为一个在源头附近 loudly fail 的 bug，远比一个继续运行、最终在几百万条指令之后才表现成奇怪现象的 bug 更容易定位和修复。

所以，断言并不是为了让系统更“宽容”，而是为了让系统对 bug 更“不宽容”。

## 把断言看作软件里的保险丝

为了帮助建立正确直觉，Samek 提出一个很好的类比：断言像电路里的 fuse。

1. 保险丝不能防止问题出现。
2. 它也不能修好问题。
3. 但它能在系统异常时用一种受控方式“烧断”，避免事态无边界扩散。

这个类比非常适合 embedded software。因为嵌入式系统出错后最危险的不是“程序停了”，而是程序在错误状态下继续控制硬件、继续输出错误行为、继续破坏更多状态。

断言的作用，就是像保险丝一样，在关键边界上把损害控制住。

## 本课小结

第 47 课的重点不是某个具体宏，而是建立一套关于 assertions 的工程观念：

1. 断言用于检测 programming error，而不是处理正常运行中的异常条件。
2. defensive programming 常常只是把 bug 藏起来，并带来更多复杂度。
3. Design by Contract 让前置条件、后置条件和不变量变成可执行的设计文档。
4. 断言失败应该被视为 loud failure，它比静默产生错误结果更有价值。
5. 在 embedded systems 里，断言更像 software fuse，是 damage control 的重要边界。

下一课会继续把这个观念落到 embedded C/C++ 实践层面，讨论如何实现更适合嵌入式系统的断言、如何设计 assertion handler，以及为什么真正成熟的产品通常不应该把断言在发布版里全部关掉。
