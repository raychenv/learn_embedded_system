## 为什么 GUI 迫使软件转向新的编程范式

课程先从命令行系统和 GUI 的差异讲起。

在传统 command-line 程序中，软件模型非常简单：

1. 输入通常只有键盘。
2. 输出通常只去屏幕底部。
3. 程序可以按“等待输入 -> 处理输入 -> 输出结果”的顺序循环执行。

这种程序天然适合 sequential programming，因为输入源单一，输出位置也固定。

但 GUI 完全不同。它会立即引入几个根本性问题：

1. 输入源不再只有键盘，还有 mouse。
2. 程序必须同时对多个输入源保持响应。
3. 输出不再总是写到屏幕底部，而是要写到当前活动窗口或控件。
4. 鼠标输入还带有二维坐标，需要进一步判断屏幕上哪个对象收到了这次输入。

这意味着 GUI 程序已经不能继续沿用“先等这个输入，再处理那个输入”的简单顺序结构。因为你一旦阻塞地等键盘，就无法响应鼠标；反过来也是一样。于是，软件必须换一种思路：不再以程序步骤为中心，而要以外部发生的输入事件为中心。

## 事件驱动的核心思想：由事件驱动程序，而不是程序驱动流程

Samek 用一句话概括这个转变：在 event-driven programming 中，events drive the software。

也就是说：

1. 外部输入会先被记录成 event 或 message。
2. 程序不是主动去“按顺序推进”所有逻辑。
3. 程序的任务是等待事件到来，并在事件到来时做出响应。

这就是事件驱动和顺序式编程最本质的区别。

从这个角度看，GUI 编程的关键已经不再是“下一步执行什么语句”，而是“来了什么事件”和“当前该由谁处理这个事件”。

## Win32 示例：一个最经典的事件驱动程序骨架

为了说明事件驱动程序的结构，课程选用了一个非常经典的 Win32 C 程序 `HelloWin`。虽然 Win32 API 很老，但它足够底层、足够直接，能够把事件驱动机制的核心特征完整暴露出来。

### WinMain 与窗口类初始化

在 Win32 GUI 程序中，入口点不再是普通 C 程序熟悉的 `main()`，而是 `WinMain()`。这相当于 GUI 应用的主入口。

在 `WinMain()` 里，程序首先做了一些初始化工作：

1. 准备窗口类 `WNDCLASS` 的属性。
2. 设置窗口样式、光标、窗口类名等信息。
3. 指定与该窗口类关联的 `WndProc`。
4. 调用 `RegisterClass()` 向 Windows 注册这个窗口类。
5. 调用 `CreateWindow()` 创建窗口对象。

Samek 特别指出，这一段其实很像上一课讲过的 OOP in C：

1. `WNDCLASS` 像一个“类描述结构”。
2. `WndProc` 很像这个类的 virtual function。
3. `CreateWindow()` 很像构造函数，用来创建对象实例。

换句话说，Win32 API 虽然不是以 OOP 术语设计的，但其结构上已经带有明显的对象化和事件分发特征。

## 事件循环是事件驱动程序的核心

初始化完成后，真正的工作并不是在 `WinMain()` 中顺序展开，而是进入 event loop，也叫 message loop 或 message pump。

这个循环的结构非常固定，核心只有两步：

1. `GetMessage()` 阻塞等待事件。
2. `DispatchMessage()` 把取出的消息分派给对应窗口的 `WndProc`。

### GetMessage 解决了“同时等待多个输入”的问题

`GetMessage()` 的作用非常关键。它并不是只等键盘或只等鼠标，而是统一等待来自 Windows 系统的各种输入与系统事件。当这些事件发生时，Windows 会先把它们记录成一个 `MSG` 对象，并放入应用程序的 message queue。

于是 GUI 程序就不再需要自己同时轮询多个输入源，而是统一从消息队列中取消息。

这一步解决了 GUI 面临的第一个大问题：如何同时等待多个来源的输入。

### 事件通过消息对象异步投递

课程强调，Windows 在产生事件时，并不会等待你的应用程序把事件处理完。它只是把事件封装成 message，放入队列中。

这意味着：

1. 事件生产者是 Windows 系统。
2. 事件消费者是你的应用程序。
3. 二者彼此独立运行，并不同步等待对方。

这种投递方式叫 asynchronous event delivery，也就是异步事件投递。

消息队列的存在很重要，因为它让系统既能在应用空闲时投递事件，也能在应用正忙于处理上一个事件时继续缓存后续事件。

### DispatchMessage 体现了 inversion of control

`DispatchMessage()` 的作用，是把当前消息派发给当前窗口对应的 `WndProc`。

这带来一个对很多嵌入式程序员来说非常关键的观念转变：

1. 在 RTOS 线程式编程里，通常是应用代码主动调用系统服务。
2. 在事件驱动系统里，反过来是系统调用你的应用代码。

这就叫 inversion of control，也就是控制反转。

这正是“事件驱动”真正的含义：控制流不再由应用自己线性推进，而是由外部事件和事件循环来驱动。

## WndProc：事件处理函数本身就是程序的行为中心

在 Win32 中，真正处理事件的是 `WndProc`。它的签名看起来像一个普通回调函数，但 Samek 用 OOP 视角解释它，会更容易理解：

1. 它可以看作窗口类的成员函数。
2. 它实际上还是一个“按消息类型分发”的 virtual function。
3. 第一个参数可以理解为“当前窗口对象”，Samek 甚至把它类比为 `me` 指针。

消息结构 `MSG` 的前几个字段，会被拆成 `WndProc` 的参数传入，其中最重要的是消息类型，也就是 message signal。

现代事件驱动术语里，这个“消息种类”通常叫 signal。Samek 因此把 Windows 的 `message` 参数类比理解为 `sig`，而另外两个参数 `wParam` 和 `lParam` 则是该事件附带的参数。

## 在 WndProc 中用 switch(sig) 按事件类型处理

`WndProc` 的核心工作，是根据消息类型决定如何处理当前事件。Win32 的经典写法，就是对消息 signal 做一个 `switch`。

例如：

1. `WM_CREATE` 表示窗口创建。
2. `WM_DESTROY` 表示窗口销毁。
3. `WM_PAINT` 表示窗口需要重绘。
4. `WM_KEYDOWN` 表示键盘按下。
5. `WM_MOUSEMOVE` 表示鼠标移动。
6. `WM_TIMER` 表示定时器到期。

每一个 `case` 都对应一类事件的处理逻辑。

这里可以看出，event-driven 程序虽然不再按“顺序步骤”来组织逻辑，但它并不是毫无结构，而是按“事件种类”来划分行为。

## WM_PAINT 展示了 GUI 中输出也必须遵循事件机制

在这个 `HelloWin` 示例里，窗口会显示键盘按键次数和鼠标移动次数。真正的绘制工作并不是在按键或鼠标移动时直接把内容写到屏幕上，而是在 `WM_PAINT` 中统一完成。

这说明 GUI 程序的输出也必须遵循事件驱动模型：

1. 状态变化先发生在应用内部。
2. 然后通知系统“这个窗口需要重绘”。
3. 最终由系统触发 `WM_PAINT`，再进入重绘逻辑。

这和传统命令行程序“想输出时就立刻打印”是很不一样的。

课程里用于显示计数值的变量被定义为 `static`，也是为了说明：`WndProc` 每次处理完消息都会返回，因此事件之间要共享的状态不能放在自动局部变量中，否则每次返回后都会丢失。

## 事件处理不是单向的，而是应用和系统的双向通信

课程中有两个很典型的例子说明这一点。

第一个是 `PostQuitMessage()`。在处理 `WM_DESTROY` 时，应用调用这个 API 往自己的消息队列里再投递一个 `WM_QUIT`，从而让事件循环退出。这说明应用也可以异步地给自己“发事件”。

第二个是 `InvalidateRect()`。当键盘计数或鼠标计数变化后，应用并不是直接重画窗口，而是调用 `InvalidateRect()` 通知 Windows“这个区域脏了，需要重绘”。随后 Windows 再通过 `WM_PAINT` 驱动实际的绘制。

这两个例子说明，事件驱动系统中应用与框架之间是双向协作关系，而不是简单的单向调用。

## default case 很重要：它决定了 GUI 的默认行为和整体风格

`WndProc` 里最容易被忽略，但其实非常重要的一部分，是 `default` 分支调用 Windows 提供的默认 `window proc`。

这意味着：

1. 所有消息都会先经过你的 `WndProc`。
2. 你可以选择处理某些消息。
3. 其余没有显式处理的消息，会交给系统默认处理。

这就是为什么即使你的程序只显式处理了少数几种消息，它仍然天然具备一个“标准 Windows 窗口”该有的大量行为，比如：

1. 可以移动。
2. 可以缩放。
3. 可以最小化、最大化、恢复。
4. 有标题栏。
5. 具备标准的 look and feel。

Samek 把这种设计称为：

1. Ultimate Hook
2. Programming by Difference

含义很清楚：你不需要重写整个系统行为，只需要编写与默认行为不同的那一小部分逻辑。

从 OOP 角度看，这也很像一个层次化的基类/子类设计：Windows 系统提供大规模默认行为，你的应用只 override 少数感兴趣的部分。

## 事件驱动程序的几个核心特征

通过这个 Win32 示例，Samek 提炼出 event-driven programming 的几个关键属性。

### 1. 用事件对象或消息对象来记录外部刺激

事件不会以“立即执行某段应用代码”的形式出现，而是先被记录成 message object。这样它们可以被排队、传递、缓存和延后处理。

### 2. 事件投递通常是异步的

事件生产者不会等待消费者处理完成，而只是把事件放进队列。

### 3. 事件处理以 Run-to-Completion 方式进行

这是本课最重要的性质之一。事件循环一次只派发一个事件，而某次 `DispatchMessage()` 调用必须完整返回，事件循环才会继续取下一个事件。

这意味着：

1. 当前事件的处理必须完整结束。
2. 在它结束之前，不会开始处理下一个事件。
3. 每个事件处理步骤都应当短小、完整、不可中途被另一个事件打断。

这就叫 Run-to-Completion，简称 RTC。

### 4. 控制流发生反转

应用代码不是主动推进主流程，而是在系统调用你时做出响应。这就是 inversion of control，也是所有事件驱动系统的本质特征之一。

## 事件驱动和顺序式编程不能混着用，尤其不能在事件处理里阻塞

课程最后一部分是整课最重要的工程结论。Samek 没有只停留在理论说明，而是故意把“顺序式思路”硬塞进事件驱动程序里，展示它会如何出问题。

### 错误示范：在 WM_KEYDOWN 中用 Sleep 模拟 LED 闪烁

为了模拟“按下键盘后 LED 亮 200ms 再灭”，课程在 `WM_KEYDOWN` 里尝试写出顺序式逻辑：

1. 先把 LED 状态改成 ON。
2. 调用 `InvalidateRect()` 请求刷新。
3. 调用 `Sleep(200)` 阻塞等待。
4. 再把 LED 状态改成 OFF。
5. 再次请求刷新。

从传统顺序式编程的角度，这段逻辑非常自然。但把它放进 `WndProc` 里之后，程序行为马上变坏：

1. 界面变得卡顿、冻结。
2. 连续按键时，计数器不会及时更新。
3. 鼠标事件和键盘事件会在积压后一起“跳变”显示。
4. LED 状态也没有按预期正确显示。

### 为什么界面会冻结：阻塞把 event loop 堵死了

原因并不复杂。`Sleep()` 阻塞的是当前 `WndProc`，而 `WndProc` 不返回，事件循环就无法继续取下一个事件。

结果就是：

1. 新的键盘和鼠标事件还在不断到来。
2. Windows 继续把它们异步塞进消息队列。
3. 但应用暂时没有机会处理这些事件。
4. 队列逐渐积压。
5. 等阻塞终于结束后，积压事件会被快速清空，于是计数值突然跳变。

这就是 Windows 程序员常说的“应用像 pig 一样卡住了”的由来。

### 为什么 LED 显示没有按预期更新：阻塞破坏了 RTC 语义

更有意思的是，LED 状态在 `Sleep()` 前请求了一次重绘，但界面上并没有及时显示出 ON 状态。

原因在于：

1. `InvalidateRect()` 只是告诉 Windows 这个窗口需要重绘。
2. 真正的重绘要等 `WndProc` 返回后，Windows 才能再投递 `WM_PAINT`。
3. 但此时代码立刻进入 `Sleep()` 阻塞，没有返回事件循环。
4. 因此系统没有机会派发 `WM_PAINT`。
5. 所以 LED 的 ON 状态根本没来得及显示出来。

从事件驱动视角看，任何 blocking call 本质上都等价于“在处理一个事件的中途，又偷偷等待另一个事件发生”。这直接违反了 Run-to-Completion 假设，因此会把事件模型搞乱。

## 正确做法：把“等待”改写成新的事件

这节课最关键的结论之一就是：在 event-driven 系统里，不要用阻塞等待去表达时间流逝，而要把未来时刻要发生的事情转化成另一个事件。

在 Win32 中，这正是 timer 的用途。

### 用 WM_TIMER 实现真正的事件驱动式 LED 闪烁

正确方案是：

1. 在 `WM_KEYDOWN` 中把 LED 状态设为 ON。
2. 请求窗口重绘。
3. 调用 `SetTimer()`，让系统在 200ms 之后投递一个 `WM_TIMER`。
4. 立即返回，不阻塞当前事件处理。

随后：

1. 系统继续正常处理其他键盘、鼠标和绘图事件。
2. 200ms 到达后，Windows 投递 `WM_TIMER`。
3. 在 `WM_TIMER` 中把 LED 状态设为 OFF。
4. 请求重绘。
5. 调用 `KillTimer()` 停止该定时器。

这样一来，LED 闪烁逻辑就不再靠顺序式等待实现，而是被拆成两个事件处理步骤，中间的时间间隔交给框架负责。

这才是真正“事件驱动”的写法。

## 这节课对嵌入式系统的真正启发

虽然本课示例运行在 Windows GUI 上，但 Samek 的真正目的并不是教 Win32 编程，而是借 GUI 这个历史上最典型的场景，建立对事件驱动编程本质的理解。

这对后续嵌入式系统非常重要，因为很多 reactive embedded system 和 GUI 一样，都具有这些特征：

1. 输入来源多且异步。
2. 程序必须持续保持响应。
3. 系统行为更多由外部事件触发，而不是按固定步骤顺序推进。
4. 应用层代码中不能随意阻塞。

也就是说，后面在嵌入式中看到的事件队列、active object、state machine 等做法，并不是凭空发明出来的，它们和 GUI 事件循环在本质上是一脉相承的。

## 总结

33 课的核心，是用 Win32 GUI 这个经典例子，把 event-driven programming 的基本思想讲清楚。

这一课最重要的结论可以概括为：

1. GUI 的出现使传统顺序式编程模型不再够用，因为程序必须同时处理多个异步输入源。
2. 事件驱动程序以 event 或 message 为中心，程序通过 event loop 从消息队列中取事件并分派处理。
3. 事件投递通常是 asynchronous 的，而应用对事件的处理应当遵循 Run-to-Completion 语义。
4. `DispatchMessage()` 调用应用自己的 `WndProc`，体现了 inversion of control，这是事件驱动系统的本质特征。
5. `WndProc` 中的 `switch(sig)` 是典型的事件分派方式，而默认 `window proc` 提供了系统级的默认行为。
6. 事件驱动程序和顺序式程序不能混写，尤其不能在应用层事件处理代码里使用 blocking call。
7. 正确的做法不是“阻塞等待某件事发生”，而是“安排未来产生一个新事件”，例如使用 `WM_TIMER`。

如果说前几课通过 OOP 解释了如何组织对象，那么这一课则开始解释现代 reactive software 是如何组织控制流的。它把一个非常关键的原则立住了：在事件驱动系统里，响应性比顺序直觉更重要，而保持响应性的前提，就是应用层代码必须 non-blocking。下一课进入嵌入式场景后，这条原则会变得更加关键。
